# Karpathy 知识库系统

## 核心观点

**你可能根本不需要 RAG**。

只需要一个**聪明的文件夹结构 + LLM**。

## Karpathy 是谁

- OpenAI 创始成员之一
- 前特斯拉 AI 总监
- 深度学习领域的顶级专家

## Raw vs Wiki 双层结构

| | raw/ | wiki/ |
|--|------|-------|
| 本质 | 冰箱里的生食材 | 做好的菜 |
| 谁写的 | 作者原文，你搬运 | LLM 重新组织 |
| 格式 | 混乱无结构 | 统一格式+摘要+索引 |
| 关联 | 文件孤立 | [[链接]] 形成知识网络 |
| 能否提问 | 不能 | 能，直接问 LLM |
| 谁维护 | 你（往里丢就行） | LLM（编译、更新） |

## 三核心动作

```
Ingest（摄入）→ Query（查询）→ Lint（体检）
```

1. **Ingest**：把好文章丢进 raw/
2. **Query**：问 Claude Code，它检索 wiki 回答
3. **Lint**：定期检查，找矛盾、过时内容

## 什么时候用 RAG

| Obsidian 方案 | 需要 RAG |
|--------------|----------|
| <10 人小团队 | 数万~百万级文档 |
| <1000 文档 | 毫秒级响应需求 |
| 个人研究者 | 多用户并发查询 |
| 需要可视化、可编辑 | 企业级部署 |

> **为什么不先从 Obsidian 开始？如果发现规模超出了它能处理的范围，再迁移到 RAG。**

## 文章拆解工作流

1. 看到好文章 → Web Clipper 存到 raw/
2. Claude Code 逆向拆解 → 生成 7 个角度 + 21 个开头钩子
3. 编译成 wiki → 自动打标签
4. 下次想写文章 → 按标签找灵感和手法

## 核心原则

| 原则 | 说明 |
|------|------|
| LLM 写，你读 | wiki 是 LLM 的领域 |
| 累积式增长 | 每次查询都增强 wiki |
| 索引即检索 | 文件结构替代检索算法 |
| 透明可解释 | 所有数据人眼可读 |

## 传统 RAG vs Obsidian 方案对比

| 维度 | 传统 RAG | Obsidian 方案 |
|------|----------|---------------|
| 技术复杂度 | 向量数据库 + 嵌入模型 + 检索流程 | 只有 Markdown |
| 成本 | 服务器 + API 调用 | 几乎免费 |
| 可解释性 | 黑盒 | 完全透明 |
| 维护成本 | 需要技术背景 | 任何人都能用 |
| 设置时间 | 数小时~数天 | 10 分钟 |
