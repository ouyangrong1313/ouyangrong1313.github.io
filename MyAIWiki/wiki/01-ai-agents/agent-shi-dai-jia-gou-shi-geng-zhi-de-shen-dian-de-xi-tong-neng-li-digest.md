## 核心摘要

Agent 时代，框架半衰期极短，但**系统能力不会**。本文提出架构师应聚焦那些跨模型、跨框架依然有用的工程能力：上下文分层、工具契约、评估体系、状态管理、Harness 演化、权限收口。

**核心观点**：
- Filter 比 Feed 更重要——先练筛选，再谈学习
- 上下文不是聊天记录，是运行时工作集
- 工具是给模型用的接口，不是给人类用的 API 列表
- 评估前置，否则团队永远靠体感讨论"有没有变好"
- Harness 是模型外面的运行底座，不是薄薄一层壳
- 起点：窄目标 + 小闭环 + 可回滚

## 文章骨架（结构拆解）

### 开头钩子（已积累）
- 以"朋友丢过来一篇 Rohit 长文"引入，建立真实感
- 用"Agent领域变化太快"的痛点引发共鸣
- 转换视角："不追框架名单，先看进入真实系统后会遇到哪些问题"

### 正文展开（8大模块）

**模块1：筛选比路线图更耐用**
- 核心类比：过去学技术栈路径稳定；Agent 表层抖得厉害
- 关键词："半衰期"——模型封装层半衰期短；协议/状态/沙箱/评估半衰期长
- 实用筛选问题表：半年后还重要吗？是包装层还是底层能力？能接进现有系统吗？
- 结论：帮团队更快试一下的用就行；会改变构建系统方式的才值得沉淀

**模块2：上下文分层（最值得补的能力）**
- 核心洞察：很多失败表面是"模型没想明白"，实际上是上下文坏了
- 类比：上下文窗口更像运行时工作集，不像聊天记录
- 实操问题清单：当前最需要哪几块信息？哪些只要摘要？哪些留文件/数据库？
- 上下文分层表：
  - 模型窗口：当前目标、关键约束、最近观察、下一步决策
  - 会话状态：任务计划、已完成动作、待办、错误修复
  - 文件/数据库：大对象、日志、代码、历史文档
  - 项目规范：AGENTS.md、README、Runbook、团队约定
  - 工具层：检索、分页、验证、权限受控动作
- 引用 Claude Code 92% 缓存命中率的案例——背后靠的是上下文结构干净

**模块3：工具设计**
- 核心洞察：工具不是"给模型接 API"，而是给模型用的业务接口
- 实用经验：5-10个边界清楚的工具 > 20个模糊重叠的工具
- 好工具说明文档六要素：什么时候用/不该用、参数填什么、返回关键结果、失败怎么修、危险动作权限确认
- MCP 值得关注的理由：把工具/资源/能力边界拆开，不必靠自定义胶水代码
- 沉淀资产的标准：命名干净、参数边界明确、错误可执行、返回有上限、大结果可分页

**模块4：评估前置**
- 核心问题：Agent 经常"不崩但给错结果"，不上线永远不知道
- 类比：Eval = Agent 的单元测试
- 评估分层（分业务场景）：
  - 编程 Agent：代码保留率、测试通过率、返工次数、回滚率
  - 客服 Agent：是否继续追问、是否转人工、投诉率
  - 写作 Agent：段落保留率、改稿轮数、标题采用率
  - 数据分析 Agent：SQL是否执行、图表是否进入报告、结论是否被修正
  - 运维 Agent：告警恢复率、误操作率、回滚次数、MTTR
- 核心观点：最值得沉淀的是样本/标签/失败分类/验收标准，而非某个评估平台

**模块5：Harness 运行底座**
- 核心洞察：Harness 不是 Agent 外面的壳，是模型和真实工作之间的运行底座
- 短任务 vs 长任务：短任务 Harness 像执行壳；长任务撞上后端同类问题（状态/队列/日志/权限/恢复/审计/成本）
- 扎实的分工：模型负责选下一步；Harness 负责验证/执行/捕获/反馈/设置检查点/调度子任务
- 模型升级后 Harness 要做清理：旧 prompt/工具包装/规则可能是新模型的负担
- 案例：Cursor Harness 复盘——模型决定能力上限，Harness 决定生产下限

**模块6：状态管理**
- 核心问题：模型不保存状态，长任务靠聊天历史硬撑会被冲散
- 状态分层：
  - 模型窗口：当前推理状态
  - Session/Checkpoint：任务进度，可回放
  - 文件/数据库：事实、trace
  - AGENTS.md/Skill/Runbook：团队经验
  - 用户偏好：小型记忆或画像
- 类比：Claude Code/Cursor/Devin/Aider 绕不开文件系统——因为文件系统可 diff/可回滚/可审查/可复现
- 不能简化成"加一个记忆库"，要先拆清楚边界

**模块7：多 Agent 与任务边界**
- 核心原则：先看上下文边界，再谈角色分工
- Subagent 适用场景：查证/对比/验证，探索过程留在独立窗口，主 Agent 只拿结论和证据
- 实操判断问题：是否独立调查？是否污染主上下文？只读还是写共享状态？结果由谁合并？
- 默认建议：单 Agent 起步更稳，等上下文压力/串行延迟/任务异质性出现再上 orchestrator-subagent
- 警告：为了架构看着先进而提前拆，只会把错误并行化

**模块8：权限和沙箱**
- 核心原则：安全不能后补，越早进设计越好
- 基本功：文件读写范围、命令白名单、网络出口控制、密钥作用域、权限分级、危险操作二次确认、沙箱隔离、trace审计、回滚路径
- 架构师价值点：把权限/沙箱/审计/回滚提前到设计阶段，而非等产品快上线

**模块9：落地路径（小闭环）**
- 反面案例：先搭"万能平台"——容易被抽象拖住
- 正面建议：从窄目标开始（已有需求、重复度高、结果能验证）
- 验收写法：从"提升效率"改为具体口径（自动解决率/人工复核时间降多少/返工率/失败回滚方式）
- 最小闭环8要素：
  1. 明确目标
  2. 单 Agent 主循环
  3. 3-7个边界清楚的工具
  4. 窗口外状态层
  5. 沙箱
  6. Trace
  7. 50条初始评估样本
  8. 能回滚的发布方式

**可以晚一点的东西**
- 包装层很厚的新框架
- 说不清解决哪类真实失败的复杂记忆系统
- 按角色搭的多 Agent（先画边界）
- 公开 benchmark（替不了内部 eval）
- 每周追榜的模型更新（季度重评估更稳）

## 写作手法分析

| 手法 | 出现位置 | 具体用法 |
|------|---------|---------|
| 对比冲突 | 全文 | "今天讲 LangGraph，明天讲 Mastra" vs "底层能力半衰期长" |
| 权威背书 | 各模块 | Anthropic/Claude/Cursor/Karpathy 多方引用 |
| 类比说理 | 上下文/评估/Harness | "Eval = Agent的单元测试" / "上下文 = 运行时内存" |
| 清单式总结 | 模块2/4/9 | 表格 + 分层列表，信息密度高 |
| 反面教材 | 模块9 | "万能平台"+"每周追模型"做对比 |
| 克制收尾 | 结尾 | "不是否定，只是先排序"——不偏激，容易被接受 |

## 新角度（7个写作方向）

1. **《Agent 时代的 CDK：为什么上下文分层比模型选择更重要》**
   - 钩子：Claude Code 的 92% 缓存命中率不是靠缓存，靠的是上下文结构设计
   
2. **《"工具即接口"：为什么五到十个工具比二十个更好》**
   - 钩子：模型不是人类工程师——它看到 400 Bad Request 不会自己翻文档

3. **《Harness 是新后端：一个模型外面那层运行底座的工程化思考》**
   - 钩子：Cursor 团队复盘 Harness 时说"模型决定上限，Harness 决定下限"

4. **《评估即单元测试：如何让团队的 Agent 讨论不再靠体感》**
   - 钩子："换了模型，感觉更聪明"——但哪类任务变好，哪类退化，没人说得清

5. **《单 Agent 起步：多 Agent 的前提是边界，不是角色》**
   - 钩子：为了架构看着先进而提前拆，只会把错误并行化

6. **《模型升级后 Harness 要做减法：哪些旧补丁成了新负担》**
   - 钩子：大部分团队低估了自研 Agent 的工作——大家觉得核心在模型公司

7. **《协议先于生态：为什么 MCP 值得认真看》**
   - 钩子：不是因为它热，而是因为它把工具/资源/能力边界拆开了
