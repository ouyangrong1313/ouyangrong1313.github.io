## 为什么值得收进知识库

这篇文章的价值不在于单纯介绍几个项目，而在于它把一个常被忽略但又决定 Agent 上限的问题讲透了：**知识如何被组织、维护、调用，并在持续使用中形成复利。**

对 Agent 系统来说，Prompt、Context、Harness 很重要，但这些更偏“运行时控制”；真正决定 Agent 能否随着时间不断变强的，是 **Skill + 知识库 + 维护机制**。

## 核心观点

### 1. 知识不是“多存点资料”，而是要能被编译和治理

文章反复强调一个现实：
- 人很擅长收藏
- 但不擅长持续整理
- 企业级知识库的维护成本更高，尤其是时效性和组织结构复杂性

所以问题不是“有没有知识”，而是：
- 能不能把非结构化资料转成 Agent 真能用的结构
- 能不能持续更新、清洗、去重、淘汰
- 能不能让知识随着使用而越来越顺手，而不是越来越乱

这和我们做 Agent 的经验高度一致：**Memory 堆积不是长期记忆，结构化可调用的知识才是。**

### 2. Knowledge Engineering 是独立于 Prompt 的关键层

文章把知识分成两类：

#### 经验性知识
例如：
- 编码规范
- 工具使用路径
- 做事步骤
- 任务中隐含的判断原则

这类知识更接近 Skill。

#### 事实性知识
例如：
- 定义、原理、FAQ
- 技术资料、博客、文档
- 业务背景、方案、案例

这类知识更适合作为 wiki / second brain 内容。

文章的核心判断是：
> 如果 Prompt Engineering 决定“怎么说”，Knowledge Engineering 决定“知道什么”和“凭什么判断”。

这也是 Agent 质量上限的关键。

### 3. LLM Wiki 的意义：把知识编译成“活体”

LLM Wiki 表面上看只是 Markdown 文件，但真正重要的是它代表了一种范式：
- 用 Schema/结构约束来指导 LLM 维护知识
- 把原始材料编译成有结构的 Wiki
- 建立交叉引用、矛盾标记和上下文链接
- 让知识成为一个可演进的系统，而不是静态文档堆

这种方式的好处是：
- 避免每次都从头检索和重新推导
- 让历史理解被沉淀下来
- 降低重复劳动和算力浪费

### 4. Skillify：把知识变成“像 Skill 一样能被调度”

文章很有启发的一点，是把 Skill 从“固定格式文件”扩展为一种知识组织方式。

也就是说：
- Skill 不一定非得是 `SKILL.md`
- 一篇 Markdown、一段说明、一份案例，都可以成为可调度知识单元
- 关键不是文件名，而是元数据、Schema 和调用机制

这其实就是一种“渐进式披露”的知识设计：
- 先只暴露入口和元信息
- 命中场景时再读取详细内容
- 避免上下文被无脑塞满

对 OpenClaw / Obsidian / 各类 Agent 平台都很有参考价值。

### 5. GBrain：从轻量知识库走向知识运营系统

文章把 GBrain 视作比 LLM Wiki 更工程化的方向，重点包括：
- 多模态知识接入（文档、网页、语音、图片、视频等）
- OCR、转录、实体抽取、关系抽取
- 采集 → 摘要 → 归档 → 向量化 → 检索 → 使用 → 修复 的全流程闭环

这说明当知识规模增大后，重点已经不是“有没有知识库”，而是：
- 有没有 ingestion pipeline
- 有没有索引和检索策略
- 有没有更新、修复、淘汰机制

也就是知识运营，而不只是知识存储。

### 6. 不能只指望 Agent 自动沉淀 Skill

文章也指出了一个非常现实的问题：
- 自动沉淀 Skill 很诱人
- 但它依赖模型触发和判断
- 所以实际效果往往不稳定

相比之下，建设一个**可持续维护的知识库系统**，通常是更可靠的“让 Agent 变聪明”路径。

这个判断很重要，因为它提醒我们：
- Skill 自动化有价值，但不是银弹
- 更稳定的复利来自：知识结构 + 调度方式 + 维护流程

## 对当前知识库体系的直接启发

### 1. raw / wiki 分层是对的

这篇文章再次说明，把原始资料和编译知识分层是必要的：
- `raw/` 存原始文章、资料、摘要初稿
- `wiki/` 存整理后的长期知识

这样既保留来源，也利于后续再编译。

### 2. Skill 与 wiki 应该双向流动

可以继续沿用这条判断：
- **高频、可执行、步骤化经验** → Skill
- **背景理解、案例、方法论、架构认知** → wiki

如果某个 wiki 内容反复被调用，就值得进一步 skillify。

### 3. 混合架构比单一路线更现实

文章明确提到：
- RAG 快，但未必准
- 渐进式披露和深读更准，但更慢

所以真正可落地的方案，通常是混合式：
- 先用检索快速缩小范围
- 再用 LLM 深读命中文档
- 最后把高价值结果沉淀回知识库

### 4. 长期记忆的重点是维护成本

知识系统最怕“越来越多，越来越没人敢动”。

所以建设重点应该放在：
- 更新机制
- 去重机制
- 过时知识清理
- 引用来源保留
- 可调用性验证

## 和 OpenClaw / Agent 实践的关联

这篇文章和 OpenClaw、Hermes、Obsidian 一类系统是强相关的，因为它讨论的是同一个本质问题：

> 如何让 Agent 不只是“这次回答对”，而是“下次更容易答对”。

从这个角度看：
- Prompt 是短期控制
- Skill 是经验封装
- Wiki / GBrain 是长期知识底座
- 真正的复利来自它们之间的协同演化

## 推荐沉淀的几个长期原则

1. **不要把 Memory 当知识库**
   - Memory 更像日志或短期上下文
   - 知识库需要结构、索引和维护策略

2. **把知识当“可编译资产”而不是“静态收藏”**
   - 原始文章只是素材
   - 编译后的 wiki 才是 Agent 更易调用的形式

3. **优先沉淀高复用经验和高价值背景知识**
   - 不是所有内容都值得长期保存
   - 重点是复用率和决策价值

4. **Skillify 是连接知识与执行的关键桥梁**
   - 让知识不只可读，还能在场景中被触发

## 一句话总结

**真正让 Agent 具备长期竞争力的，不只是更强的模型，而是一套能持续整理知识、提炼经验、反复调用并不断更新的“知识—Skill—执行”闭环。**
