# AI 团队协作案例：Loop + SDD（Digest）

> **一句话总结**：叶小钗核心命题"个人提效 ≠ 团队提效"——Loop Engineering 隐藏了 3 个前提（Skill 维护/最终验证/Token 成本），组织级补丁是 SDD，本质构建规则和信息流；最终落到 AI 进入组织 4 层爬坡视角。

## 速查表

### 核心观点（5 条）
1. **个人提效 ≠ 团队提效** — 技术负责人的真正课题是"团队要怎么和 AI 合作"
2. **Loop 隐藏 3 个前提** — Skill 从哪来/谁做最终验证/Token 成本谁买单（总 token 消耗 3-8x 手动 Prompt）
3. **SDD 是组织级补丁** — 共享上下文/规则/边界/验收标准的方法论，Spec 是研发流程事实真相
4. **Spec 三角色 = 上下文 + 协作契约 + 质量基准** — 避免"产品/研发/AI/测试各理解一套"
5. **AI 进入组织 4 层爬坡** — 个人工具→流程标准→组织协作→评价体系

### 关键金句
- "个人提效 ≠ 整体提效"（核心命题）
- "Loop Engineering 最大的问题：只说结果，不说前提"
- "方向正确和实现简单之间，隔着大量的工程细节"
- "AI 当前最大的问题不是输出速度，他只解决了干得快的问题，不解决干得对的问题"

### 关键数字
- Loop 模式 token 消耗 = 手动 Prompt 的 3-8 倍 / 单任务成本 2-4 倍人工
- Boris Cherny 单人 6 个月 259 PR / 497 提交 / 4 万行代码
- 某学员 3 个月 + 十几次设计推倒 才把团队级 AI 跑通

## 6 段式 Spec 骨架（实操）

**目标** — 用户/系统/业务流程能获得什么能力（错误："新增商品订单详情页" / 正确："为运营人员提供订单详情查看，便于定位履约问题"）

**范围** — 做什么/不做什么（明确"本期只支持查看，不支持编辑"）

**约束** — AI 必须遵守的边界（"敏感信息落库必须加密；外部 AI 接口超时 15s 触发降级"）

**决策** — 不许 AI 重新设计（"用哪个表/复用哪个组件/哪个状态枚举/接口命名"）

**任务** — 拆小任务（数据模型→接口→前端→测试，逐项推进）

**验收** — 写清"什么情况下算修复完成"

## 任务复杂度 3 分级 + AI 进入组织 4 层

**3 分级**：小改动 Prompt 即可 / 中等需求 轻量规格 / 大型任务 完整规格

**4 层爬坡**：①个人工具→暴露团队协作 ②流程标准→暴露组织边界 ③组织协作→暴露专业评价 ④评价权专业→暴露商业模式

**关键判断**：这些问题不是 AI 带来的，组织原本就存在——只是过去被低效率掩盖。

## 5 个对 Seetong 团队可借鉴动作 + 关联

1. **SDD 做 Seetong 团队需求交付新规范** — TAPD PRD 模板加 6 段式 Spec 骨架 + 任务复杂度分级；嵌入 `seetong-requirement-clarifier` skill
2. **"客服反馈 → Spec 链路"入 Seetong Bug Triage** — 对照 `seetong-bug-triage` skill，反馈先整理成轻量 Spec
3. **项目级上下文建 Seetong 项目知识库** — 架构约束/目录规范/接口规范/权限模型/状态机/错误码/数据字典
4. **Spec 6 段式入 Seetong Bug 描述模板** — TAPD BUG 描述加 6 段，重点约束"AI 重构时不许换组件库/状态枚举/接口命名"
5. **AI 进入组织的 4 层爬坡做 Seetong 团队 AI 落地 roadmap** — 当前卡 1-2 层，6 个月推 SDD 落地，12-18 月推 Spec 锚定
