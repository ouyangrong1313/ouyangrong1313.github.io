# 我用 Claude Code 搭了个四个 AI 的团队，居然真的能协作开发

> 来源：微信公众号  
> 探索用 Claude Code 搭建多 AI 协作框架，每个 AI 专职一个角色，通过文件共享通信。

## 背景：单 Agent 的致命问题

### 上下文膨胀
- 单个 AI 做复杂项目时，每次对话都要处理越来越长的历史信息
- Token 大量浪费在重复分析已知信息上
- 一个 Agent 既做产品设计，又写代码，又测试——角色混乱

### 核心痛点
- 🔥 **上下文膨胀**：每次对话都要处理越来越长的历史信息
- 💰 **Token 浪费**：大量 Token 消耗在重复分析已知信息上
- 🧠 **角色混乱**：一个 Agent 既做产品设计，又要写代码，还要测试
- ⏱️ **效率递减**：项目越复杂，单次响应越慢

## 解决方案：用 Claude Code 搭建专职 AI 团队

### 四个专职角色
```
agentGroup/
├── max/     麦克斯 - 项目经理 (PM)
├── ella/    艾拉 - 设计师 (UI/UX)
├── jarvis/  贾维斯 - 程序员 (全栈开发)
├── kyle/    凯尔 - 测试工程师 (QA)
└── shared/  共享工作区
    ├── status.json       # 项目状态看板
    ├── notifications.json # 内部消息系统
    ├── tasks/             # 待办事项
    ├── docs/              # 需求文档
    ├── designs/           # 设计稿
    └── reviews/           # 测试报告
```

### 关键发现
- Claude Code 支持独立项目配置：`claude --project xxx`
- 每个 AI 运行在独立实例中
- **文件系统是最简单的通信方式**：Claude 天然擅长读写 JSON 文件

### 文件通信方案
```json
// shared/status.json - 团队状态看板
{
  "current_task": "开发个人网站",

// shared/notifications.json - 内部消息系统
```

**优点：**
- ✅ 零配置：不用装数据库、不用启动服务器
- ✅ Claude 原生支持：读写 JSON 文件是 Claude 的强项
- ✅ 直观易懂：出了问题直接打开文件看
- ✅ 版本可控：所有状态都在 Git 里，可以回滚

## 核心工程：AI 约束机制

### 问题：AI 每次重启都"失忆"
- 不遵守应该遵循的工作流程
- 跳过检查步骤

### 解决方案：强制检查点流程（刻入 CLAUDE.md）

**每个 AI 收到消息后，必须按以下检查点顺序执行：**

1. **第0检查点 - 任务范围确认**：`📋 任务范围确认: [明确/需澄清]`
2. **第1检查点 - 策略读取**：必须用 Read 工具读取 token-optimization.md
3. **第2检查点 - 通知检查**：运行 check_notifications_simple.sh 脚本
4. **第3检查点 - 任务分解**：判断是否需要拆分子任务
5. **第4检查点 - Skill 检查**：评估是否有专业技能可用
6. **第5检查点 - 执行选择**：选择合适的模型和执行方式
7. **第6检查点 - Git 安全**：检测是否需要 Git 操作授权

**关键技术：必须物理执行，不能口头承诺**
- ❌ 不允许：直接说"已了解优化策略"
- ✅ 必须做：实际调用 Read 工具读取文件

### Skill 优先机制
- 强制检查是否有专业技能可用
- 如果有匹配 skill，优先使用 Skill 工具执行
- 避免重复造轮子

### 自省机制
```markdown
在每次工具调用前，必须自问:
❓ 我是否已完成6个强制检查点？
❓ 如果任务可分解，我是否使用了Task工具？
❓ 如果直接执行，我是否说明了模型选择原因？

IF (发现任何跳过) THEN {
  🛑 立即停止当前操作
  🔴 输出: "⚠️ 检测到流程违规，正在强制纠正..."
  ✅ 重新完整执行6个检查点
}
```

## Token 优化的两个层次

### 第一层：时间戳检查避免重复读取

**核心突破：mtime 时间戳预检**

```bash
check_notifications_simple.sh 的核心逻辑
current_mtime=$(stat -f %m "$NOTIFICATIONS_FILE")
last_mtime=$(cat "$CACHE_FILE")

if [ "$current_mtime" = "$last_mtime" ]; then
  echo "文件未变化，跳过读取"
  exit 0  # 0 Token 消耗
else
  echo "文件已更新，需要读取"
  echo "$current_mtime" > "$CACHE_FILE"
  exit 1  # 触发文件读取
fi
```

**优化效果：**
- 文件未变化：0 Token 消耗（97% 的情况）
- 文件有更新：正常读取处理（3% 的情况）
- 平均节省：**97% 的文件读取 Token**

### 第二层：不同难度用不同 AI

**模型选择经验：**

| 任务类型 | 推荐模型 | 适用场景 |
|---------|---------|---------|
| 提取信息 | Haiku | 从日志找错误、统计代码行数 |
| 格式转换 | Haiku | JSON 转 CSV、生成表格 |
| 逻辑分析 | Sonnet | 诊断 Bug 原因、设计方案 |
| 创新设计 | Opus | 全新架构设计、商业策略 |

**Task 工具指定模型示例：**
```markdown
# 错误方式：全用 Sonnet (花费 $0.24)
Task(prompt="分析这个系统架构，找出问题，生成报告")

# 优化方式：分工协作 (花费 $0.13，节省 46%)
Task(model="haiku", prompt="从代码中提取所有API端点和数据库表")  # 信息收集
Task(model="sonnet", prompt="分析架构设计问题和性能瓶颈")        # 核心分析
Task(model="haiku", prompt="把分析结果格式化成规范报告")         # 格式化输出
```

**月度模型分布目标：Haiku 30-40%、Sonnet 50-60%、Opus 5-10%**

## 踩坑经验

### 1. Git 操作必须严格控制
```markdown
🚫 绝对禁止的操作：
- git commit (必须用户确认)
- git push (必须用户确认)
- git reset --hard (绝对禁止)
- rm -rf (绝对禁止)

✅ 安全操作流程：
1. 修改文件 ✓
2. 告诉用户"准备提交，请授权"
3. 等用户说"可以提交"才执行
```

### 2. 模型选择教训
- **昂贵的错误**：用 Opus 做简单格式转换（$0.20 vs $0.02）
- **便宜但无效的错误**：用 Haiku 做复杂架构分析（质量不行，需要重做）

### 3. 防止 AI"装听话"
检查点必须**物理不可跳过**：
- ❌ 错误：让 AI 口头保证"已读取优化策略"
- ✅ 正确：强制用 Read 工具读取文件前 20 行
