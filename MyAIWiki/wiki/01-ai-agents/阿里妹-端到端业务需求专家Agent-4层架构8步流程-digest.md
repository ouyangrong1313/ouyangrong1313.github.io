# 阿里妹·端到端业务需求专家 Agent(速读)

> 森叶(阿里妹) | 2026-06-15 | 原文:https://mp.weixin.qq.com/s/9o_z-Poj9r4dbwe3NlC1pw

## 一句话

**这篇文章最值钱的不是"AI 会写代码",而是把业务需求从进入到结项的整条链路组织成闭环:上下文、工具、人工确认、验收证据、反馈沉淀,都被放进了一套 4 层架构 × 8 步流程里。**

## 速查表

| 维度 | 结论 |
|---|---|
| 真瓶颈 | **代码之外的串联成本**——上下文切换、工具切换、人工补位 |
| 横向架构 | 4 层:上下文输入 / 业务专家编排 / 工具执行 / 反馈学习 |
| 纵向流程 | 8 步:需求进入 / 澄清 / 方案 / 实现 / CR 协同 / 验收 / 发布 / 结项 |

| 最关键协作动作 | **反馈留痕到平台**——评论/工单/里程碑,而不是聊天框 |
| 3 大问题 | 接入成本高 / 缺效果度量体系 / 单 Agent → Agent Team |

## 3 个反直觉点

1. **评论最密的阶段不是编码,而是澄清和方案。** 需求理解对了、方案确认了,到真正写代码时,以现在 Agent 的能力反而不需要那么多微操。
2. **最重要的不是让 Agent 记住一切,而是别把过程噪声直接升级进长期知识。** 这就是"不直接升级"原则。
3. **真正的质量门禁不能靠提示词。** 提示词只会提醒,硬规则才会阻止——不通过就推不上去,这才叫工程化卡口。

## 3 个对 Seetong 可借鉴动作

1. **继续坚持 MyAIWiki 的 raw/wiki 分层**——过程材料留 raw,结项后才进 wiki。
2. **给 Seetong 技能补阶段编排**——在现有 `seetong-*` 之外,补 `clarify / plan / execute / finish` 4 阶段链路。
3. **给版本复盘补结项审计表**——稳定知识 / 流程改进 / 归档 / 待补证分开处理。
