# Agent时代，架构师更值得沉淀的系统能力

## 核心结论

Agent 时代，框架半衰期极短，但**系统能力不会**。架构师应聚焦跨模型、跨框架依然有用的工程能力：上下文分层、工具契约、评估体系、状态管理、Harness 演化、权限收口。

## 分类提炼

- **场景**：AI Agent / 架构方法论
- **标签**：#主题/AI-Agent #概念/Harness #概念/评估体系
- **类型**：方法论

## 要点列表

### 1. 筛选比路线图更耐用

关键词：**半衰期**
- 模型封装层、CLI 参数半衰期短
- 协议、状态、沙箱、评估、工具契约半衰期长

新东西过一遍：
- 半年后它还重要吗？
- 是包装层，还是底层能力？
- 能接进现有系统吗？
- 解决什么失败模式？是真实痛点还是 demo 新鲜感？

### 2. 上下文是运行时工作集

上下文分层：

| 层级 | 适合承载 |
|------|----------|
| 模型窗口 | 当前目标、关键约束、最近观察、下一步决策 |
| Session/Checkpint | 任务进度、已完成动作、待办、错误修复 |
| 文件/数据库 | 大对象、日志、代码、历史文档 |
| 项目规范 | AGENTS.md、README、Runbook、团队约定 |
| 工具层 | 检索、分页、验证、权限受控动作 |

参考 Claude Code 92% 缓存命中率——稳定内容和动态内容分干净。

### 3. 工具是给模型用的接口

五到十个边界清楚的工具 > 二十个模糊重叠的工具

好工具六要素：
- 什么时候该用 / 不该用
- 参数填什么
- 返回关键结果
- 失败怎么修
- 危险动作权限确认

MCP 值得关注：把工具/资源/能力边界拆开，不必靠自定义胶水代码。

### 4. 评估前置

| Agent 类型 | 关键指标 |
|------------|----------|
| 编程 Agent | 代码保留率、测试通过率、返工次数、回滚率 |
| 客服 Agent | 是否继续追问、是否转人工、投诉率 |
| 写作 Agent | 段落保留率、改稿轮数、标题采用率 |
| 数据分析 Agent | SQL是否执行、图表是否进入报告、结论是否被修正 |
| 运维 Agent | 告警恢复率、误操作率、回滚次数、MTTR |

**Eval = Agent 的单元测试**。最值得沉淀：样本、标签、失败分类、验收标准。

### 5. Harness 是运行底座

核心分工：
- **模型**：负责选下一步
- **Harness**：验证、执行、捕获输出、反馈、设置检查点、调度子任务

模型升级后要做清理：旧 prompt/工具包装/规则可能是新模型的负担。

### 6. 状态管理

状态分层：

| 类型 | 放哪 |
|------|------|
| 当前推理状态 | 模型窗口 |
| 任务进度 | Session、Checkpint |
| 可回放事实 | 文件、数据库、Trace |
| 团队经验 | AGENTS.md、Skill、Runbook |
| 用户偏好 | 小型记忆或画像 |

文件系统：可 diff、可回滚、可审查、可复现。

### 7. 多 Agent：先看边界

判断是否拆：
- 是否独立调查？
- 是否污染主上下文？
- 只读还是写共享状态？
- 结果由谁合并？

默认：单 Agent 起步更稳。等上下文压力、串行延迟、任务异质性出现再上 orchestrator-subagent。

### 8. 权限和沙箱

安全不能后补。基本功：
- 文件读写范围、命令白名单
- 网络出口控制、密钥作用域
- 权限分级、危险操作二次确认
- 沙箱隔离、trace 审计、回滚路径

### 起点原则

> 窄目标 + 小闭环 + 可回滚

不要一上来搭"万能平台"。从业务已经在意、重复度高、结果能验证的任务开始。
