# 从Prompt、Context到Harness，工程的三次进化与终局之战

## 核心结论（一句话）

AI 工程的真正升级路径不是“更会写 Prompt”，而是从表达任务、组织上下文，走向通过 Harness 设计让整个 Agent 系统稳定、可验证、可持续地产出结果。

## 要点列表

1. **三层结构不是替代关系，而是嵌套关系**
   - Prompt Engineering 负责“我该怎么跟模型说”
   - Context Engineering 负责“模型回答时应该知道什么”
   - Harness Engineering 负责“整个 AI 系统如何可靠运转”

2. **Prompt 的边际价值正在下降，但不会消失**
   - 模型更强后，写 Prompt 不再是主要护城河
   - Prompt 仍然是任务表达层，没有它，后续注入的上下文也无法被准确理解

3. **Context Engineering 是当前最具工程价值的能力之一**
   - 关键问题是上下文窗口有限、信息竞争激烈、长上下文会发生中间遗忘
   - 实际抓手包括 RAG、滚动摘要、重要性评分、层次记忆、单一事实来源
   - 把文档全部塞进系统提示词，往往比不塞更糟

4. **Harness Engineering 解决的是“流程级可靠性”**
   - Agent 跑偏、越权修改、自称测试通过、风格漂移、重复造轮子，本质都不是 Prompt 或 Context 单独能解决的问题
   - Harness 需要覆盖规则、工具、权限、验证、反馈、回滚、监控和协作机制

5. **OpenAI 的案例说明，真正决定产出质量的是系统闭环**
   - 用上下文治理把海量规则压缩成索引，再按需动态加载
   - 用 Chrome DevTools、日志、指标、lint、自动化测试，把“声称完成”改造成“验证完成”
   - 用后台清理任务持续修正规范漂移和技术债积累

6. **Anthropic 的案例说明，多 Agent 是复杂任务的补稳机制**
   - Planner 负责拆解任务
   - Generator 负责逐项实现
   - Evaluator 负责独立审查
   - 当单 Agent 的可靠性边界被复杂任务击穿时，角色分工是必要工程手段

7. **Harness 有一个重要规律：模型越强，Harness 越薄**
   - 不要过度设计那些模型将来自然能解决的问题
   - 重点保留两类约束：业务边界 / 合规要求，以及外部工具接口 / 权限控制
   - 好的 Harness 不是最厚，而是最贴着当前模型失效边界

8. **工程师价值正在从“实现”转向“掌舵”**
   - Human steer, agents execute
   - 人类更像方向制定者、支架设计者、关键决策者
   - 衡量标准从个人代码产出转向系统杠杆与闭环设计能力
