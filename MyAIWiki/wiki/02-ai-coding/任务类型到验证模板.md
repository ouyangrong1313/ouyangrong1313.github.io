# 任务类型到验证模板

## 核心结论（一句话）

定义任务时，必须同时定义验证方式；没有 verifier 的任务，默认视为未完成。

## 分类提炼

- 场景：Codex 任务定义 / AI Coding 收尾 / 知识库编译 / 配置修改
- 标签：#主题/AI-Coding #主题/效率 #手法/工程实践 #场景/技术博客
- 类型：短规则页 / 验证模板 / 收尾清单

## 使用原则

1. 验证优先用最小、最便宜、最能拦错的动作  
2. 不默认全量 build / lint / test  
3. 模型自称“完成”不算完成，必须有外部证据  
4. 如果验证被跳过，要明确写出为什么跳过  

## 模板

```text
Task type:
Goal:
Stop condition:
Verifier:
Evidence:
Not verified:
```

## 按任务类型分类

### 1. 知识库编译类

**适用场景**
- 用户发来文章链接，要求“编译知识库”
- 新增 raw / digest / wiki 页面

**最小 verifier**
- `raw` 文件存在
- `digest` 文件存在
- `wiki` 文件存在
- `wiki/master-index.md` 有条目
- 对应分类 `index.md` 有条目
- `log.md` 有记录
- 读回 wiki 页面前几段，确认结构正常

**证据模板**

```text
Verifier:
- raw/digest/wiki files exist
- master-index/category index/log updated
- wiki page read-back OK
```

### 2. 文档改写 / 规则页新增

**适用场景**
- 新增知识页
- 更新 `AGENTS.md` 相关规则说明
- 补短规则页 / SOP 页

**最小 verifier**
- 目标文件存在
- 关键标题和结构可读
- 若有索引，应补索引
- 若有日志，应补日志

### 3. 脚本类

**适用场景**
- 新增 CLI 脚本
- 修改自动化脚本
- 工具封装

**最小 verifier**
- `--help` 能运行，或最小输入样例能跑通
- 返回码正常
- 输出与预期主路径一致

**避免事项**
- 不默认跑全量数据
- 不把“脚本能执行”误当成“业务逻辑正确”

### 4. 窄范围代码修改

**适用场景**
- 单文件 bugfix
- 小型逻辑修补
- 小范围 UI 修正

**最小 verifier**
- 命中的最小 smoke test
- 若无测试，则至少给出稳定复现路径
- 读回 diff，确认没有误改周边行为

### 5. 共享模块 / 高风险代码修改

**适用场景**
- 公共组件
- 跨模块接口
- 工具链 / 配置核心路径

**最小 verifier**
- 命中相关测试或最小子集测试
- 确认受影响调用点无明显断裂
- 必要时补一条人工验证说明

**判断标准**
- 改动触达范围越广，verifier 就要越靠近真实调用链

### 6. 配置类修改

**适用场景**
- `AGENTS.md`
- skill 配置
- Codex / OMX / OpenClaw 配置

**最小 verifier**
- 配置文件可读
- 关键字段存在
- 若配置直接影响运行，则跑最小命令验证

**典型例子**
- 浏览器/MCP 通道配置后，应至少验证一次实际调用链

### 7. 浏览器 / 外部抓取类

**适用场景**
- 微信文章正文抓取
- 网页抽取
- 浏览器登录态任务

**最小 verifier**
- 页面标题抓到
- 正文字段抓到
- 关键元信息抓到（作者 / 时间 / 来源其一）
- 与目标页面主题一致，不是验证码页或错误页

**当前经验**
- 微信文章优先用 `Playwright + Chrome`
- `curl` 直抓仅作为低优先级尝试

## 一个简单判断法

如果任务结果不能回答下面三个问题，就还不算定义完整：

1. 什么时候停？
2. 用什么证据证明停得对？
3. 哪些东西没有验证到？
