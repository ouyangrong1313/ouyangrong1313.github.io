# Codex配置下一步改造：从规则层走向线程、工具、目标与共享记忆

## 核心结论（一句话）

当前 Codex 配置的下一阶段重点，不该继续堆规则，而应把改造重心转向四个执行层：长期线程、稳定工具面、目标与验证模板、线程外共享记忆。

## 分类提炼

- 场景：Codex CLI / oh-my-codex / 长任务协作 / 个人工作台优化
- 标签：#主题/AI-Coding #主题/AI-Agent #主题/效率 #手法/工程实践 #场景/技术博客
- 类型：改造路线 / 配置清单 / 执行优先级

## 为什么现在该做这一步

你现有的配置已经把很多基础打好了：

- `AGENTS.md` 已经承担路由和边界职责，而不是纯知识仓库
- `MyAIWiki` 已经被明确为正式知识库
- 验证闭环、skills、记忆、状态工具这些部件也已经开始成型

这意味着当前瓶颈不再是“规则不够多”，而是：

1. 长任务缺少稳定承载容器  
2. 浏览器 / MCP / 外部系统接入还不够稳  
3. Goal 和 verifier 没有沉淀成标准模板  
4. 共享记忆还偏“文章归档”，没有完全升级成“线程外上下文层”

## 四个最高优先级方向

### 1. 长期线程（Durable Threads）

**要解决的问题**  
高频任务虽然反复发生，但每次仍然像新任务一样启动，很多上下文要重新说明。

**建议做法**

- 为高频、长期、可持续推进的工作建立固定线程类型：
  - 幕僚长线程
  - 知识库编译线程
  - 发布跟踪线程
  - 代码审查线程
  - 外部信息监控线程
- 每种线程都明确：
  - 目标
  - 默认输入源
  - 输出位置
  - stop condition
  - 最小 verifier

**判断标准**  
如果一个任务在 7 天内重复出现 3 次以上，并且每次都要重新解释背景，就值得升格为长期线程。

### 2. 稳定工具面（Tools and Reach）

**要解决的问题**  
能力差异往往不在模型，而在浏览器、MCP、登录态、桌面操控和外部连接器是否稳定可用。

**建议做法**

- 把浏览器执行面视为一等基础设施，而不是临时手段
- 为以下几类工具建立稳定入口：
  - 浏览器正文抓取
  - MCP 查询
  - 桌面 GUI 补位
  - 外部服务连接器
- 把“能否访问真实工作入口”当作配置检查项，而不是只检查模型和技能

**当前直接启发**

- 微信文章抓取之所以卡住，不是文章难，而是浏览器执行面不稳定
- Claude Code 能完成的动作，很多不是因为模型更强，而是因为 Playwright/浏览器面已经打通

### 3. 目标与验证模板（Goals and Verifiers）

**要解决的问题**  
很多长任务有目标，但没有明确终点和验收标准，最后只能靠“模型自称完成”。

**建议做法**

- 给高频任务建立统一模板：

```text
Goal:
Stop condition:
Verifier:
Failure signal:
Escalation rule:
```

- 典型 verifier 分类：
  - 文档类：索引 / 日志 / 链接 / 可读性
  - 知识库类：目标路径、索引条目、日志条目、正文回读
  - 脚本类：帮助信息、最小输入样例、退出码
  - 代码类：最小 smoke test、指定测试、复现路径

**判断标准**  
如果任务结果不能被一句“验证器是什么”回答清楚，那这个任务定义还不够完整。

### 4. 线程外共享记忆（Shared Memory）

**要解决的问题**  
很多上下文虽然被记录了，但只停留在单次对话里，没有成为未来线程可直接继承的结构化记忆。

**建议做法**

- 把 `MyAIWiki` 从“文章知识库”继续升级为“正式共享记忆层”
- 增加几类短页，而不是只存长文：
  - 任务类型 -> 验证模板
  - 高频工作流 -> 标准步骤
  - 工具入口索引
  - 多 agent 使用边界
  - 长任务线程说明页
- 保持原则：
  - 长文保留背景
  - digest 保留提炼
  - wiki 页保留执行结论
  - 短页保留可立即调用的规则和模板

## 推荐实施顺序

1. **先补 Goal/Verifier 模板页**  
   这是最低成本、最高收益的改造，能立即减少“看起来完成了，实际没完成”的情况。

2. **再定义 3 到 5 个固定线程类型**  
   不求多，先把最常出现的工作模式固定下来。

3. **再稳定浏览器和 MCP 工具面**  
   优先补那些已经反复证明重要、但每次都临时打通的入口。

4. **最后把共享记忆页补齐**  
   让未来任务不再只依赖当前会话，而是能直接继承结构化上下文。

## 一个简单判断法

如果一个配置项同时满足下面三点，就应该从“对话里临时说”升级成正式配置资产：

1. 会重复发生  
2. 失败成本高  
3. 结果需要验证

例如：

- 知识库编译流程
- 微信文章抓取路径
- 完成前最小验证动作
- 哪些任务适合长线程
- 哪些任务值得启用多 agent
