## 一句话总结

作者用一个"在线系统重构迁移"项目真实跑过 **Spec-Kit** 和 **BMAD** 两套 SDD 工具,核心结论是:**AI 原生 ≠ 人人都用 AI 工具;AI 原生 = 让 AI 真正参与团队协作流程,把视角从"AI 能不能帮我写代码"切换为"AI 依据什么写代码"。Spec-Kit 拉上限、依赖团队基建;BMAD 拉下限、依赖人审阅;两者没有优劣,匹配的是不同团队规模与基建。**

## 关键观点

1. **AI 原生 ≠ 人人都用 AI 工具**:"个人要求把 AI 能力发挥到极致,组织要求为 AI 准备好稳定的环境"——这是个人提效和组织提效的根本分水岭
2. **AI 原生组织的"割裂感"** = 一线使用者觉得"酷",组织建设者觉得"到处是规则、人工兜底、半自动流水线",真正要做好必须处理十几项暗面清单(权限/监听/PR 改动范围/单测兜底/模型冲突/上下文压缩/合并事故责任等)
3. **SDD 回答的是底层问题**:"当 AI 开始进入研发流程之后,团队应该如何组织需求、方案、任务和代码?"
4. **SDD 三剑客分工**:Spec-Kit(秩序/标准)/ OpenSpec(变更/演进)/ BMAD(编排/角色化)——三者可组合,不是单选题
5. **Spec-Kit 的 4 步流程**:`/specify` → `/plan` → `/tasks` → `/implement`,核心是"维护 AI 稳定运行需要的上下文环境"
6. **Spec-Kit 在多仓/重构迁移里的 3 大硬骨头**:多仓适配(主仓 spec 怎么被前/后端子仓各自消费)/ 顺序 vs 并行(流程是 1234 顺序,现实前后端是并行)/ 来回拉扯(实际是 123 → 234 → 12341234 反复)
7. **作者为了补足 Spec-Kit 链路之外的事,做了三类扩展**:多仓适配(按端自动切片的实现命令)/ 多视角评审(架构/后端/QA 各看各的)/ 门禁卡点(启动前/写完后/发布前三次卡点)
8. **BMAD 是什么** = AI-driven development framework = 角色化 Agent + 引导式工作流,4 大阶段(分析→规划→方案→实现),特色是 Party 模式 + 圆桌评审
9. **BMAD 两个被点醒的瞬间**:(1)从标准 SaaS 错位成 BPO 系统,建设方向从源头改;(2)圆桌挑战"行为基线",从模糊"对方无感"到硬约束"没有基线,不切换"
10. **砍需求要分阶段持续砍**:PRD 阶段砍 13 项、架构砍一轮、epics 砍一轮,逻辑都是"代价 > 收益即可砍"
11. **BMAD 最大问题:心智负担**——工作量翻倍、流程长、审阅累、不擅长领域(前端审后端)要先吃透、AI 是断断续续协同、整体时间不见得比人类协同快
12. **一句话的差异**:Spec-Kit 拉上限(依赖团队基建)/ BMAD 拉下限(依赖 Agent 内部封装)/ **最终拍板都要靠人,差别在于 AI 帮你补的是上限还是下限**
13. **AI 原生团队的核心 = 重新组织协作流程**,不是"多买几套 AI 工具",AI 是新的协作成员,不是新的 IDE

### 2. 这篇文章给出的 3 个可立刻落地的判断

- **判断 1:Spec-Kit vs BMAD 怎么选?**
  - **看团队规模** — 一两人/小团队 → BMAD(补角色)
  - **看团队基建** — 强/规范健全 → Spec-Kit(强秩序)
  - **看项目复杂度** — 单一 Feature → Spec-Kit;多仓/重构/迁移 → 都要扩展
- **判断 2:多仓 Spec-Kit 怎么落地?**
  - **submodule 主仓+子仓**(作者验证过)
  - **按端规则文件**(让 AI 知道当前处理的是前端/后端/共享规格)
  - **按端自动切片的 tasks**(前端后端各拿各的清单)
  - **三类卡点**(启动前 API 冲突、写完后合规、发布前全量)
- **判断 3:BMAD 圆桌什么时候特别有价值?**
  - **重构/迁移/替换类项目** — 容易"做完发现跟线上对不齐"
  - **方案阶段有歧义时** — 多角色挑战可以避免"自以为是合理的成功标准"
  - **跨领域决策时** — 前端审后端方案吃力,圆桌能补足知识盲区

### 3. 这篇文章踩过的 3 个真实坑

- **坑 1:"先把代码搞通"导致漂移**
  - 短期省事,长期变"代码新/文档老",后续喂给 AI 的信息偏差
  - 解决:回到规格主仓重走 1234,改动成本高但保一致性
- **坑 2:成功标准建立在"没被验证的前提"上**
  - "切换后对方研发几乎无感"——老系统行为基线都没采,拿什么比?
  - 解决:**"没有基线,不切换"** 写进硬约束
- **坑 3:把 BMAD 当"提效工具"而不是"补足工具"**
  - BMAD 的优势是拉下限,不是压缩时间
  - 前期方案设计阶段,审阅工作量不亚于甚至超过人类会议评审
  - 心态上要准备"代码速度提上去了,审阅量没减少"

### 4. 对 Seetong 团队的启示

- **Seetong 目前是典型"小团队 + 复杂业务 + 多端"组合**,跟作者用 Spec-Kit 的场景高度相似
  - iOS / Android / HarmonyOS / Web / 后端 / 设备 SDK 都有
  - 历史需求文档不完整,设备能力不断演化
  - 异地组网、套餐失效、云组播等场景都有"历史包袱"
- **可立刻试的 3 件事**:
  1. **用 Spec-Kit 跑一个跨 iOS/Android 的小 Feature** — 验证多仓 submodule + 按端切片是否真能跑通
  2. **对核心业务(Bug 离线/预览/报警链路)做"行为基线"采集** — 把现状用埋点/日志固化成可对比的事实,后续 AI 改造有锚点
  3. **.omx/skills 下补"多视角评审"skill** — 不一定要上 BMAD,但可以借鉴圆桌思路,让多个 Agent(架构/后端/QA/产品)对一个变更分别给评审
- **优先避坑**:
  - 不要直接上 BMAD 全流程(心智负担风险)
  - 选 Spec-Kit 时不要只看主流程,先把"多仓适配/门禁卡点/角色评审"三件套准备好
  - 砍需求要分阶段,不要等"系统做完了再砍"——每阶段砍一波,符合"代价 > 收益"就砍

## 关键术语

- **SDD**(Spec-Driven Development):规格驱动开发
- **Spec-Kit**:GitHub 2025 开源,标准化规格流程(`/specify` / `/plan` / `/tasks` / `/implement`)
- **OpenSpec**:轻量级规格变更
- **BMAD**:AI-driven development framework,角色化 Agent + 引导式工作流 + 圆桌评审
- **Party 模式**:多 Agent 同会话互呛
- **圆桌评审**:多角色 Agent 对同一问题分别分析
- **BPO**(Business Process Outsourcing):把公司内部资源包装做对外业务输出,与标准 SaaS 相对
- **行为基线**:老系统历史演化出来的行为事实,没有基线不能切换
- **心智负担**:跟 AI 协作时大量审阅和决策带来的认知压力
- **硬约束**:落成可验证条款的强约束(如"没有基线不切换")
- **AI 原生**:让 AI 真正参与团队协作流程,而非"人人有 AI 工具"
- **拉上限 vs 拉下限**:Spec-Kit 把团队能力上限拉高,BMAD 把团队能力下限托住
- **门禁卡点**:Feature 启动前/写完后/发布前的合规检查
- **多仓适配**:主仓 spec 怎么被前/后端子仓各自消费的工程机制
- **多视角评审**:架构/后端/QA 等多角色 Agent 分别评审同一份产物
