# 多Agent使用边界与并行判定

## 核心结论（一句话）

多 agent 不是默认更强；只有当任务可拆分、边界清楚、互不踩写、并且并行结果能真正缩短路径时，才值得启用。

## 为什么需要这页

很多任务表面看起来“复杂”，但并不适合多 agent。

常见误区有两个：

1. 任务一复杂就想并行  
2. 觉得多 agent 天然比单 agent 更高级  

实际上，多 agent 的收益只在以下条件成立时才明显：

- 任务可拆
- 子任务之间低耦合
- 写入边界清楚
- 需要的结果可以并行生成

否则只会带来更多上下文同步成本。

## 什么时候适合单 agent

以下情况优先单 agent：

- 任务范围小，几步内能闭环
- 下一步动作强依赖上一步结果
- 核心难点在推理，不在并行收集
- 写入文件高度集中，容易冲突
- 用户要的是直接结果，不是并行过程

**典型例子**
- 单文件 bugfix
- 一页 wiki 编译
- 一次明确的配置修改
- 需要边看边改的任务

## 什么时候适合多 agent

以下情况值得考虑多 agent：

### 1. 信息探索可以并行

例如：
- 一个 agent 查本地代码结构
- 一个 agent 查外部官方文档
- 主线程做集成判断

### 2. 写入边界天然分离

例如：
- 一个 agent 负责知识库页面 A
- 一个 agent 负责知识库页面 B
- 主线程只负责索引和汇总

### 3. 审查与执行能拆开

例如：
- 主线程负责实现
- 子 agent 负责 review 或 verifier

### 4. 任务是明确的多产物输出

例如：
- 同时生成多份不互相覆盖的材料
- 多个独立模块的平行处理

## 并行前必须回答的 4 个问题

1. 子任务之间是否互相依赖？  
2. 子任务是否会改同一批文件？  
3. 主线程能否在等待期间做非重叠工作？  
4. 并行完成后，谁负责最终整合？  

只要有两项回答不清楚，就不该轻易开并行。

## 推荐分工模型

### 模型一：主线程集成，子线程探索

**适合**
- 复杂但信息来源多样的任务

**分工**
- 子 agent：查资料、查代码、找模式
- 主线程：判断、决策、实施、收尾

### 模型二：主线程实现，子线程审查

**适合**
- 风险高但写入范围集中的任务

**分工**
- 主线程：写代码或写页面
- 子 agent：找风险、找漏测、找回归点

### 模型三：主线程统筹，子线程分产物

**适合**
- 多份独立产物输出

**分工**
- 每个子 agent 拥有明确写入范围
- 主线程统一改索引、统一总结

## 不推荐的并行方式

### 1. 同时改同一文件

最容易制造冲突，也最容易让主线程花更多时间收拾残局。

### 2. 把主路径任务扔给子 agent

如果下一步动作必须依赖这个结果，最好自己做，不要把关键路径外包出去。

### 3. 为了“看起来高级”而多开 agent

并行不是表演。没有吞吐收益的并行，只是在增加协调成本。

## 一个简单判定公式

只有同时满足以下 3 条，才建议启用多 agent：

1. **可拆**：任务能切成两个以上边界清晰的子任务  
2. **可并**：子任务彼此不阻塞  
3. **可合**：主线程能低成本整合结果  

否则，默认单 agent。

## 当前最适合多 agent 的场景

1. 外部资料研究 + 本地代码结构分析  
2. 主线程实现 + 子线程 review  
3. 多篇独立知识页并行编译  

## 当前不适合多 agent 的场景

1. 单篇文章知识库编译  
2. 小范围配置修改  
3. 单文件 bugfix  
4. 需要边看边改的探索性实现
