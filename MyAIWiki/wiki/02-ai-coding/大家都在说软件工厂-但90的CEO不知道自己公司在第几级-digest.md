## 一句话总结

软件工厂的本质不是"工程师用 AI 写得更快"，而是**把人从生产线的具体步骤里按安全网同步的速度移出去**。Alex Lieberman 的五级梯子（Level 0-4）给了一个能在一分钟内定位的阶段模型，**真正的分水岭是"拿走所有 AI 工具后流程会不会崩"**——会崩 = 至少 L2-L3；不会崩 = 还停留在"更贵的个人工具"层。

## 关键观点

1. **"软件工厂"被三个力量同时推到台前**：Anthropic 90% / Google 75% 等不可忽视的证据、微软/8090/Chamath/Garry Tan 等大公司把语言带进主流、"X% 是 AI 写的"成为新的炫耀指标
2. **L1 和 L3 的差距不在 AI 覆盖率，在"人还在守哪几个关卡"**：
   - L1 = AI 加速某个步骤，流程不变
   - L2 = agent 拥有工单，但人审每个 PR
   - L3 = 三道护栏（低风险定义 + 第二 agent 审 + 自动化测试）全过自动合并
   - L4 = bug 在人知道前已修复上线
3. **升级的最大阻碍不是技术，是习惯**：
   - 0→1：组织问题（允许 + 清除阻力）
   - 1→2：让团队停止"自己写每一行"的习惯
   - 2→3：最难一跃，要内部平台 + 风险分级 + 第二 agent
   - 3→4：监控和信任，不是更好的模型
4. **第 6 问是组织诊断神器**："如果明天把所有 AI 工具拿走，我们的流程会变，还是只是变慢？"只是变慢 = L1
5. **下一个工厂的战争是争评估系统**——Karpathy："传统软件自动化你能规定的事，AI 自动化你能验证的事"。所有不是软件工程、QA、DevOps 的职能要进 L2/L3，先要解决"什么算对"的自动评估
6. **节奏原则**：跑得比安全网（测试 / 版本控制 / 内部平台）快 = 出生产事故；跑得比安全网慢 = 浪费工厂化收益
7. **DORA 报告的关键变量**：内部平台是"AI 赢家和落后者的最大单一变量"——比模型选型、组织变革都重要

## 我的理解

- **这文章给的不是"我们多 AI"的炫耀指标，是"我们少人"的流程清单**。角度比"AI 写 X% 代码"高一个维度
- **L2 → L3 那一步的三道护栏**（低风险定义 + 第二 agent 审 + 自动化测试）是真正稀缺能力——大多数团队卡在这里不是模型不够，是审查机制没重构
- **"评估系统先于 agent 能力"** 是给非工程职能最关键的提醒：客服、运营、销售这些职能要进 L2/L3，先建"什么算对"的可测量框架，agent 能力本身反而次要
- **"软件工厂"作为一个类比有点危险**：容易被解读成"上人/上系统就能工业化"，但 Alex 强调的是流程层面的重构，不是工具层面的升级
- **"激进派"和"保守派"两边都常见错误**：激进派跑得比安全网快（出生产事故），保守派跑得比安全网慢（浪费收益）。真正的能力是"和你的安全网同步"
