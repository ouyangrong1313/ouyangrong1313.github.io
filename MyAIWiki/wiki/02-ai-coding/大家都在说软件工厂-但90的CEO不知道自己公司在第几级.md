## 核心结论（一句话）

"软件工厂"不是装了 AI 工具就成立，而是把人从生产线的具体步骤里按"安全网同步"的速度移出去——Alex 给出 Level 0-4 五级梯子，**真正能区分一个公司是 Level 1 还是 Level 3 的不是"用了多少 AI"，而是"拿走所有 AI 工具后流程会不会崩"**。

## 五级阶梯（核心方法论）

贯穿案例：**客户遇到 bug，"下单"按钮失效**

| Level | 名字 | 关键特征 | 人的角色 | 典型位置 |
|---|---|---|---|---|
| 0 | 工匠 | 写、审、测、部署全人工 | 做活儿的人 | 极少数传统公司 |
| 1 | 辅助 | 流程不变，工程师用 AI 写得快一点 | 把关者 | **大多数公司实际所在** |
| 2 | 委托 | agent 写修复开 PR，人审完才合并 | 每件事的关卡 | **多数"AI 先进"公司卡在这** |
| 3 | 有监督的工厂 | 监控 agent 发现→编码 agent 修→第二 agent 审→测试通过→低风险自动合并 | 设置护栏 / 处理异常 | Ryan Carson 代码工厂、StrongDM |
| 4 | 自主工厂 | bug 在人知道前已修复上线 | 决定产品下一步是什么 | 几乎没有公司真正在这 |

> L2 和 L3 的**唯一区别是"人是不是还必须在每个 PR 合并前读一遍"**。这是流水线工业化的真正分水岭。

## 自测 6 问（拿来就能用）

1. **多少 PR 无人审就合并？** 零 = Level 1
2. **第一版代码谁写？** 人写 = L0/L1；人指挥 agent = L2+
3. **有没有"低风险自动合并 / 高风险上报"的规则？谁定的？** 这是 L3 的核心问题
4. **生产出问题谁先发现？** 人发现 = 早期；系统发现并分类 = L3
5. **有没有内部平台让 agent 接入？** DORA 报告：这是"AI 赢家和落后者的最大单一变量"
6. **拿走所有 AI 工具，流程会变还是只是变慢？** 只是变慢 = L1；流程会崩 = 真工业化

**编者注**：第 6 问是分水岭。如果 AI 工具撤走流程不变，那只是更贵的个人工具，不是工厂。

## 升级路径（4 步）

- **0→1**：组织问题，不是技术问题。给工具 + 允许在真实代码用 + 清除法务/安全阻力
- **1→2**：让 agent 完整拥有一张工单。**最大阻碍是让团队停止"自己写每一行"的习惯**，不是技术
- **2→3**：杀掉审查瓶颈。三道护栏（低风险定义 + 第二 agent 审 + 自动化测试）全过自动合并。**最难一跃**
- **3→4**：去掉最后一个人工触发点。靠监控和信任，不是更好的模型

**节奏原则**：跑得比安全网（测试 / 版本控制 / 内部平台）快 = 出生产事故；跑得比安全网慢 = 浪费工厂化收益。两边都是错的。

## 关键引用（保留原文力度）

> "代码不能由人来写。代码不能由人来审。" —— StrongDM 的两条原则

> 「传统软件自动化你能规定的事；AI 自动化你能验证的事。如果一件任务有自动成功信号，机器就能练习它，你就能信任输出。」 —— Karpathy

> 工厂的本质不是"工程师用 AI 写代码更快了"，而是**人升了一级——从做活儿的人，变成了设计生产线的人**。

## 我的判断（编译者注）

1. **这文章最值钱的是 L2 → L3 那一步的"三道护栏"**：低风险定义 + 第二 agent 审 + 自动化测试。这三件事是当前 AI Coding 团队真正缺的能力，不是模型
2. **第 6 问是组织诊断神器**：可以用作"我们到底是不是 AI-native"的最快判断。答案"只是变慢" = 还停留在工具层；答案"流程会崩" = 至少已经做到 L2-L3
3. **"下一个工厂的战争是争评估系统"** 是隐藏的元观点：所有不是软件工程、QA、DevOps 的职能，要进 L2/L3 都要先解决"什么算对"的自动评估问题
