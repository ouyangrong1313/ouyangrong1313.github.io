## 一句话总结

**React 比 HTML 更适合作为 AI 设计稿格式**——因为 React 的组件化、树形结构、数据驱动、文本化交付和 Git 版本管理，能让 Coding Agent 在设计稿基础上低成本理解、修改和扩展；而巨大 HTML 扁平、非结构化、错误不可见，已不适合 Agent 时代。

## 关键观点

1. **设计稿 ≠ 静态展示**
   - Claude Design 是 UX 交互设计工具，要求设计稿具备真实交互和数据能力
   - HTML 做不到 React 这种"数据驱动的 UI"

2. **组件化是复杂度必然选择**
   - 巨大 HTML 不可加载、不可修改、不可重用
   - React 天然支持小组件化——修改只需加载一个组件

3. **树形结构对 Coding Agent 友好**
   - 层级清晰 vs 扁平 HTML
   - 设计与实现不割裂，修改能映射回代码

4. **Agent 时代错误可被快速修正**
   - React/JS 有明确报错和组件定位
   - HTML 样式错误：没有明显报错，debug 才真的要命
   - **关键转变**：Chat 时代追求一次生成通过；Agent 时代靠错误反馈快速修正

5. **Claude Design 四件套交付**
   - HTML（结构骨架）+ CSS（设计系统）+ React（组件结构）+ **data.jsx**（数据结构）
   - **data.jsx 是隐藏的关键**——传统设计工具普遍缺失，对开发特别重要

6. **文本格式 + Git = 高效协作**
   - 所有交付物都是文本，git diff 一眼看清改了什么
   - 团队协作 + AI 协作都受益

7. **设计稿驱动的迭代流程**
   - UI 有问题先改设计稿，再让 Agent diff + 改代码
   - 配合截图辅助理解，"它就知道怎么改了"

## 核心论断

> **React + 结构化 + 数据驱动 + 文本化 + 版本管理 = AI 时代最合适的设计稿格式**

## 我的理解

- **这是 2026 年 AI Coding 范式在设计侧的延伸**——和 [[54万行代码的顿悟-Markdown才是新编程方式]] 同一主线：可读、可 diff、可版本管理的"结构化文本"是 AI 时代一等公民
- **与 [[Claude-Code动态工作流-让AI自己写Harness-这事靠谱吗]] 呼应**：都强调"结构比提示词更重要"——React 的树形结构就是设计稿侧的"harness"
- **"design.jsx" 是设计工具的隐藏必需品**——传统工具只关心"长什么样"，Claude Design 关心"长什么样 + 数据怎么流"
- **Git 友好 + diff 协作** 模式可推广到所有 AI 协作资产：文档、配置、设计稿、测试用例——只要是文本，都应该进 Git
