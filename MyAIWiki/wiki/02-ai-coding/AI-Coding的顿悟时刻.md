## 核心结论（一句话）

**当执行成本越来越低，研发团队最大的瓶颈只剩两个 —— 需求定义和架构设计。当稀缺资源从工程师时间变成正确判断，围绕旧资源建立的 Scrum / 站会 / Story Point / PM 自然到该重构的时候。AI 裁员的真实财务模型：用不到 4% 的 Agent 成本换取接近 100% 的产出提升，组织结构向两端收缩。**

## 5 节核心方法论

### #01 工厂模式成为标配
软件研发从手工艺模式（人写、人 review、人测试）→ 工厂模式（AI/Agent 跑标准生产线，人退到后面设计生产线 + 设定规则 + 处理例外）。
- **作者断言**：Software Factory 接下来半年内一定成为所有软件公司的标配
- 和 [[大家都在说软件工厂]] 互文（那个是 Level 0-4 阶梯，这个是宣告"工厂模式"成主流）

### #02 团队落地流程：Spec → 反向采访 → LDD → tickets → PR

```
1. 先写 spec
2. Agent 反向采访人（把需求问清楚）
3. Agent 生成 LDD（Lightweight Design Document）
   - 参考公司过去的设计文档
   - 保持同一种工程风格
4. 系统自动生成 tickets
5. 系统自动生成 PR
```

**关键原则**：可以沉淀成 skill 的有 —— 分支命名、API 的 service-repository pattern、analytics 埋点、测试、QA。

> **"这里的 skill 不是随便拿别人写好的模板来用，因为不同公司有自己的工程口味和产品手感。"**

**这是本文最实操也最重要的金句**。直接对应 [[Codex配置原则总览]] 的"配置必须符合团队语境"原则。

### #03 SCRUM 失效论

> "Scrum 并不是在管理项目，它本质上是在管理不确定性。"

作者把 Scrum 的几大组件逐个解构：

| Scrum 组件 | 存在前提 | Agent 时代是否成立 |
|---|---|---|
| 站会 / Standup | 信息不同步 | 大量自动化流水线消除不同步 |
| Sprint | 产能有限，需要提前规划 | 执行可预测后无需长 sprint |
| Story Point | 执行成本难以预测 | 提效 100% 都不夸张 |
| PM | 大量依赖关系需要协调 | Agent 协调依赖关系 |

**核心结论**：

> "Scrum 诞生的时候，软件行业最稀缺的资源是工程师的时间。**Agent 时代，最稀缺的资源已经变成了正确的判断。**"

> "当稀缺资源发生变化，围绕旧资源建立起来的管理体系，自然也到了该重构的时候。"

### #04 未来研发最大瓶颈 = 需求定义 + 架构设计

> "决定一个系统未来几年命运的，从来都不是代码量。而是架构。"

- 数据库怎么设计
- 服务怎么拆分
- 模块边界怎么划分
- 哪些地方需要解耦
- 哪些地方需要提前预留扩展能力

> "**方向错了，执行越快，反而离正确答案越远。**"

> "Agent 可以在极短时间内生成大量代码。但如果架构方向错了，那几十万行代码很多时候只是技术债务。"

**作者明确表态**：架构设计**暂时还不愿意完全交给 Agent**。

### #05 AI 裁员的精确财务模型

**单工程师成本**：
- 工程师年成本：50 万人民币
- Agent 额度（Codex）：200 美元/月 = 2400 美元/年 ≈ 1.7 万人民币
- 提效：100%

**公司视角**：

> "公司实际上是在用不到 4% 的额外成本，换取接近 100% 的产出提升。这笔账对于老板来说太容易算了。"

**组织结论**：**组织结构开始向两端收缩**

> "中间大量负责执行和传递信息的岗位会越来越少。留下来的，要么是负责方向和决策的人。要么是能够驾驭 Agent 的人。"

> "很多公司裁员，本质上不是因为业务变差了。而是因为过去需要十个人完成的工作，现在五个人加一堆 Agent 就能完成。"

## 我的判断（编译者注）

1. **和 [[54万行代码的顿悟]] 是同期不同侧重**：
   - 那个从"工程师个人范式"出发（Markdown 是新编程方式）
   - 这个从"团队流程 + 组织重构"出发（Spec/LDD + Scrum 反思 + 裁员）
   - **合起来看**：AI-native 团队需要的是个人范式 + 团队流程 + 组织结构三层都变

2. **"Spec → 反向采访 → LDD" 是新增的实操模式**：
   - 和 [[54万行代码的顿悟]] 的"Skillify it"循环互为补充
   - **Skillify**：构建 → 技能化（结果导向）
   - **Spec/LDD**：意图 → 文档化设计（过程导向）
   - 两者结合 = "用 LDD 把意图固定化 → 技能化执行" 的完整闭环

3. **"Skill 必须本地化" 这一条是最容易踩坑的**：
   - 很多团队直接套别人的 prompt 模板 / skill 包
   - 失败的根本原因就是"口味"不对
   - 验证：拿到一个 skill，先问"这是否符合我们团队的工程风格？"，不通过就重写

4. **"Scrum 失效"论不是否定 Scrum，而是否定它依赖的前提**：
   - 前提变了（执行可预测），体系就要重构
   - **不是"废除站会"，而是"重新设计信息同步机制"**
   - 和 [[Claude-Code负责人谈AI原生工程组织]] 的"重构信息流"思路一致

5. **"AI 裁员的精确财务模型"是说服老板的最强武器**：
   - 4% 成本换 100% 产出，不是道德判断，是经济判断
   - "组织向两端收缩" 是组织设计的真正方向
   - 留给我们的问题：**你属于哪一端？方向决策？还是驾驭 Agent？**

6. **"架构设计不愿交给 Agent" 是清醒的判断**：
   - 和 [[你不知道的 Agent]] 等的"Agent 擅长执行、不擅长判断"一致
   - 真正的护城河：需求定义能力 + 架构设计能力 + 业务判断力

7. **AI 裁员对 Seetong 团队的启示**：
   - 我们也在用 Codex / Claude Code，提效是有的
   - 是不是要重新看团队规模和角色构成？
   - 中间的"传递信息岗位"（PM / Scrum Master / 协调者）会被 Agent 消解
   - 我们要培养的是"做决策"和"驾驭 Agent"两端的人
