# Codex工具入口与能力边界

## 核心结论（一句话）

Codex 的真实能力，不由模型参数单独决定，而由“当前可用的工具入口”与“这些入口的边界条件”共同决定。

## 分类提炼

- 场景：Codex CLI / 浏览器抓取 / MCP / GUI 自动化 / 外部系统接入
- 标签：#主题/AI-Coding #主题/AI-Agent #主题/效率 #手法/工程实践 #场景/技术博客
- 类型：工具索引页 / 能力边界页 / 执行参考

## 为什么要有这页

很多失败并不是任务本身难，而是入口不对。

例如：

- 微信文章抓取失败，常常不是页面不可达，而是用了 `curl` 去碰一个需要浏览器态的入口
- 某些网页任务失败，不是模型不会，而是当前没有浏览器执行面
- 某些系统操作失败，不是逻辑有误，而是系统级 GUI 权限没有打开

因此，做任务前要先判断：“这个任务最适合从哪个入口进？”

## 当前常用工具入口

### 1. Shell / Terminal

**适用场景**
- 文件读取
- 小范围文本搜索
- 命令行工具执行
- 本地脚本运行

**优点**
- 快
- 可组合
- 适合做最小验证

**边界**
- 不适合需要登录态网页渲染的任务
- 不适合复杂 GUI 操作
- 不适合把真实浏览器行为简化成 HTTP 抓取

### 2. WebFetch / Web Search

**适用场景**
- 普通网页、文档、开放页面
- 官方文档查询
- 轻量网页抽取

**优点**
- 成本低
- 快速拿正文
- 适合公开可读页面

**边界**
- 对依赖客户端渲染、登录态或平台风控的页面不稳定
- 微信文章经常会返回验证页，而不是正文

### 3. Playwright + Chrome

**适用场景**
- 微信文章正文抓取
- 浏览器渲染结果读取
- 需要真实页面上下文的抽取

**优点**
- 最接近真实用户浏览行为
- 能直接读取 DOM、正文和元信息
- 可复用本机浏览器能力

**边界**
- 需要浏览器可启动
- 可能依赖系统权限、提权和运行环境
- 不应默认用于所有网页任务

**当前经验**
- 微信文章优先级最高的路径是：`Playwright + 本机 Chrome`
- `curl` 对微信文章是低优先级兜底，不是主路径

### 4. MCP / 外部连接器

**适用场景**
- 外部系统查询
- 结构化接口调用
- 多工具协同

**优点**
- 结构化
- 稳定
- 适合重复任务

**边界**
- 必须先有可用服务器或插件
- 没有暴露的工具，不能假装存在

### 5. GUI 自动化

**适用场景**
- 只能通过桌面界面完成的收尾动作
- 浏览器菜单、系统设置、上传下载等图形操作

**优点**
- 能补 CLI 和 API 的死角

**边界**
- 高度依赖系统权限
- 容易受语言、本地化、窗口状态影响
- 应作为补位手段，而不是首选主路径

## 任务入口选择原则

### 原则一：优先选最接近真实工作面的入口

- 普通文档 -> WebFetch
- 微信正文 -> Playwright + Chrome
- 本地文件 -> Shell
- 系统级操作 -> GUI 自动化

### 原则二：不要拿低保真入口硬做高保真任务

错误示例：

- 用 `curl` 硬抓微信正文
- 用纯文本搜索代替浏览器渲染验证
- 用聊天记忆代替共享知识页

### 原则三：入口选择本身就是任务设计的一部分

在动手前，先回答：

1. 这个任务最接近哪个真实工作面？
2. 当前会话有没有这个入口？
3. 没有的话，是切入口，还是降级目标？

## 当前最值得固定的工具策略

1. **公众号文章抓取**
   - 主路径：Playwright + Chrome
   - 兜底：用户提供正文或截图

2. **知识库编译**
   - 主路径：本地文件 + `apply_patch` + 最小验证

3. **配置改造**
   - 主路径：本地文件读取 + wiki 编译

4. **普通公开网页资料**
   - 主路径：WebFetch / Web Search

## 失败时的判断框架

任务失败后，先别急着怀疑模型，先判断属于哪类问题：

- 入口不对
- 权限不够
- 工具不存在
- 工具存在但不稳定
- 任务定义本身太模糊

这一步做对，能减少很多无意义试错。
