## 核心结论(一句话)

**AI 原生 ≠ 人人都用 AI 工具。AI 原生 = 让 AI 真正参与团队协作流程,把视角从"AI 能不能帮我写代码"切换为"AI 依据什么写代码"。在 SDD(Spec-Driven Development)工具链里,Spec-Kit 是"强秩序/依赖人上限",BMAD 是"强兜底/依赖人审阅",两者没有优劣,匹配的是不同团队规模与基建。**

## 一、AI 原生组织的"割裂感"——为什么个人提效 ≠ 组织提效

> **个人要求是把 AI 能力发挥到极致;组织要求是为 AI 准备好稳定的环境。**

文章用一个真实团队案例开场(AI 接入群聊 + 日志 + 会议纪要 + 项目文档,自动识别问题、整理 todolist、能写代码就提 PR、能出方案就出方案,复杂项目 20%+ 代码 AI 全流程完成,部分 PR 用两个模型交叉 Review)来说明"酷"和"不酷"是并存的:

| 外部视角(AI 一线使用者) | 内部视角(AI 组织建设者) |
|---|---|
| 感叹效率高 | 到处是规则、人工兜底、半自动流水线 |
| 觉得是未来 | 错了怎么办、流程崩了怎么办 |

**真正要做好"AI 原生"必须处理清楚的暗面清单**:

- 权限怎么设
- 什么任务能自动做、什么任务必须人审
- 哪些群聊可以监听、监听内容格式有无要求
- AI 生成的 todo 怎么去重
- PR 怎么限制改动范围
- 单测失败怎么办、谁兜底
- 两个模型意见冲突怎么办
- AI 催人补材料的话术、频率、升级机制
- 上下文太长怎么压缩
- AI 误判需求优先级怎么办
- 自动合并造成事故谁负责
- ......

> **结论**:AI 原生 = 人人都用 AI 工具,但 人人都用 AI 工具 ≠ AI 原生。
> 核心是"**让 AI 参与团队的协作流程**"。

## 二、SDD:Spec-Driven Development(规格驱动开发)

GitHub 在 2025 年开源 Spec-Kit 把 SDD 理念推到更多开发者面前。沿这条线社区出现了三股不同风格的工具:

| 工具 | 风格 | 偏重 |
|---|---|---|
| **Spec-Kit** | 标准化规格流程 | 强秩序,流程清晰 |
| **OpenSpec** | 轻量级规格变更 | 改动/演进 |
| **BMAD** | 完整 AI 研发流程编排 | 角色化 Agent + 圆桌 |

SDD 回答的是底层问题:**当 AI 开始进入研发流程之后,团队应该如何组织需求、方案、任务和代码?**

核心思想是"**维护 AI 稳定运行需要的上下文环境**":需求边界、代码边界、特殊业务规则都被强制标准化。

## 三、Spec-Kit:秩序感很强的轻流程骨架

### 3.1 标准 4 步流程

```
/specify   →  /plan   →  /tasks  →  /implement
需求+约束     方案拆解      可执行任务      AI 写代码
```

**价值主张**:没有 Spec-Kit 也能用 AI 写代码,但在组织角度会很乱、很多返工内耗;Spec-Kit 把散乱的信息强制标准化到一起。

### 3.2 在"在线系统重构迁移"项目里的真实体会

**典型场景**:把旧系统能力迁到新系统上,涉及:
- 旧系统在线跑着不能停
- 历史需求和业务逻辑文档不全
- 旧系统早接了一堆外部系统
- 新系统对外 API 必须兼容

**Spec-Kit 在多仓环境的硬骨头**:

| 痛点 | 表现 |
|---|---|
| **多仓适配** | 主仓沉淀 spec,前/后端子仓消费同一份规格时,字段改一处影响哪些地方? |
| **顺序 vs 并行** | 前后端实际是并行的,但 Spec-Kit 是 1234 顺序流程,跟现实冲突 |
| **来回拉扯** | 实际不是 1234 一次走完,而是 123 → 234 → 12341234 反复,需求/方案/任务/代码不断互相推 |
| **漂移风险** | 选"先把代码搞通"会变成代码是新的、文档是老的,后续喂给 AI 的信息就偏差了 |

**作者为了"补足 Spec-Kit 链路之外的事"做了三类扩展**:

1. **多仓适配** — 封装"按端自动切片"的实现命令,前端/后端各拿各的 tasks
2. **多视角评审** — 搞一组角色化评审命令(架构/后端/QA 各看各的),汇总成报告
3. **门禁卡点** — Feature 启动前跑 API ID 冲突、版本纳管检查;写完后跑合规验证;发布前全量检查

> **关键判断**:Spec-Kit 管的是它那条链路内部的秩序;**链路之外的事,必须靠团队自己补**。

### 3.3 Spec-Kit 的适用边界

- ✅ 适合:**项目产品沉淀好 + 工程事实源不错 + 单一 Feature**
- ❌ 不适合:**基建薄弱、需要大量人为治理去补足它欠缺能力** 的复杂项目

## 四、BMAD:更像一个 AI 团队在围着项目追问

### 4.1 是什么

- **AI-driven development framework** = 一组角色化 Agent + 引导式工作流
- 4 大阶段:**分析(可选)→ 规划 → 方案设计 → 实现**
- 内置角色:产品 / 架构师 / 开发者 / QA
- 特色功能:**Party 模式**(多角色同会话互呛)+ **圆桌评审**(多 Agent 审同一问题)

### 4.2 为什么是"补足"

作者最初用 Spec-Kit 时最大感受是:**它把 workflow 设计好了,你只要按部就班地填内容走流程就好**。

但有个根本问题——**把 PRD 转成前后端实现需要非常全面的能力**,前端很难 hold 住后端技术方案。

而 BMAD 的设计理念就是"**给你提供一套模拟的 AI 团队**",预置产品/架构师/开发者/QA 等角色:

> **"你不是在用一个工具,而是在跟一个 AI 虚拟产研团队合作推进一个事情,跟真实的人类协作方式非常像。所以你的认知和思维模式不用做调整,用起来就比较舒服。"**

### 4.3 在重构迁移项目里两个被点醒的瞬间

**瞬间 1:从 SaaS 错位成 BPO**

作者最初把这个系统当成标准 SaaS 来做,但 BMAD 引导他发现:**这是个 BPO 系统**——把公司内部资源包装后做对外业务输出,而不是完全标准 SaaS。

> **如果按标准 SaaS 思路做出来,有些功能是过度设计的,有些实际业务需求又没有录进系统里。** ——建设方向从源头就被改了。

**瞬间 2:圆桌挑战"行为基线"**

作者写了一个成功标准"切换后对方研发几乎无感",架构视角的 Agent 追问:**"你拿什么证明?老系统的行为基线你采了吗?"**

这一问把问题问穿了:

- 老系统跑了好多年,行为是历史演化出来的
- 根本没完整文档
- 原来的成功标准建立在**没被验证的前提**上

**硬约束最终落成一句**:**没有基线,不切换。**

> **"如果最后东西都做出来了再反过头去调整,改的东西一多,你都不确定即使是 AI 改的是不是全部改到位了,整体返工成本非常大。"**

### 4.4 砍需求要分阶段持续砍

不是上线前一次性砍,是每个阶段砍一波:

- PRD 阶段一次性砍 13 项
- 架构砍一轮
- epics 又砍一轮

每波逻辑都一样:**"这东西到现在还没用上,上线日越来越近,留它写它测它维护它的代价已经大于它能带来的收益了"**。

### 4.5 BMAD 的最大问题:心智负担

- 工作量翻倍 — 强迫症 + 怕 AI 脑补 = 细节全审
- 流程长、内容多、审阅累
- 不擅长的领域(前端审后端方案)要先吃透再做判断,很痛苦
- AI 是"断断续续"的协同,不像人类"拉个会评完各自分头干"
- 整体时间其实**不见得比原始人类协同快**——前期方案设计投入非常大

> **关键感受**:**输入确定时,BMAD 输出代码比人类高很多;但前期审阅过程比人类会议评审多得多。**

## 五、一句话的差异

| 维度 | **Spec-Kit** | **BMAD** |
|---|---|---|
| 拉什么 | **拉你的上限** | **拉你的下限** |
| 依赖 | 团队基建 + 规范 + 能力 | Agent 内部封装的能力 + 圆桌挑战 |
| 适合 | 团队能力强、基建完善 | 一人团队 / 角色缺失的小团队 |
| 主要风险 | 真实项目复杂度不在主流程里,需要大量补链路外机制 | 心智负担重,审阅工作量不亚于人类评审 |
| 协作形态 | 顺序流程(单人按部就班) | 多 Agent 圆桌(模拟团队) |

> **最终拍板都要靠人,差别在于 AI 帮你补的是上限还是下限。**
