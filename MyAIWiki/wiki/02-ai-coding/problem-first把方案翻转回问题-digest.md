## 一句话总结

**每个"需要做 XX"都是一个没说清楚的问题的"压缩版本"；PM 用 /problem-first 技能，90 秒、8 个部分（方案跳跃诊断 / 底层问题 / 假设挑战+验证 / 问题陈述 / 三个替代框架 / 想法筛选漏斗 / 草稿信 / 强制全部输出）把方案解压回问题；同样适用于点子太多的 PM 筛选 backlog（50 个想法 90% 死于"证据状态：无"）。**

## 关键观点

1. **每个方案都是问题压缩版** — 团队感知到痛点，但跳过定义问题直接给答案；PM 工作 = 把答案"解压"回问题

2. **不否定方案，把方案当研究材料** — 政治上更安全（"挖进路线图底下找问题"而非"挡在路线图前面"）

3. **Munger 翻转** — Invert, always invert；"这个方案能解决什么"不如"这个方案在响应什么问题"

4. **8 个部分输出** — 90 秒完成；强制全跑 = 系统性消除遗漏

5. **假设挑战带验证** — 每条假设 = "如果错了风险" + "验证方案"；没验证方法的假设是"更精致的猜测"

6. **三个替代框架的威力** — 团队需求"通知系统"折叠了 3 个完全不同的产品（UI 状态模式 / 信任与状态问题 / Agent 告警）

7. **逆向筛选漏斗** — 50 个想法 → 90% 死于"证据状态：无" → 3-5 存活 → 1 个下周带证据包提案

8. **草稿信 = 拉回同一对话起点** — 跳过草稿信 = 制造沟通真空

9. **协议 > 提示词** — 每次都给你同样的 8 个部分不管输入什么

10. **AI 的主要价值 = 系统性消除遗漏，不是速度** — 手写 1 小时 vs AI 90 秒，主要差异是"每次都跑全"
