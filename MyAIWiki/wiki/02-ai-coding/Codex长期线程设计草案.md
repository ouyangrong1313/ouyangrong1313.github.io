# Codex长期线程设计草案

## 核心结论（一句话）

长期线程不是“保存聊天记录”，而是为高频、长期、反复推进的任务建立固定职责、输入源、输出位置和 stop condition。

## 分类提炼

- 场景：Codex 工作台设计 / 高频任务沉淀 / 长任务协作
- 标签：#主题/AI-Coding #主题/AI-Agent #主题/效率 #手法/工程实践 #场景/技术博客
- 类型：线程设计草案 / 工作流模板 / 长任务容器

## 线程设计原则

1. 每个线程只服务一类稳定问题  
2. 线程必须有固定输入源和固定输出位置  
3. 线程必须定义 stop condition  
4. 线程要配最小 verifier，而不是靠记忆和感觉收尾  
5. 长期线程保存的是职责和上下文，不是无限堆聊天记录  

## 推荐的 5 类长期线程

### 1. 知识库编译线程

**用途**
- 收到公众号 / 网页 / 长文后，编译进 `MyAIWiki`

**默认输入**
- 用户提供的 URL 或正文

**默认输出**
- `raw/`
- `digest/`
- `wiki/`
- `master-index.md`
- 分类 `index.md`
- `log.md`

**Stop condition**
- `raw -> digest -> wiki -> index -> log` 五步完成

**Verifier**
- 文件存在
- 索引存在
- 日志存在
- wiki 读回正常

### 2. 配置改造线程

**用途**
- Codex / OMX / AGENTS / skills / 知识库配置持续演进

**默认输入**
- 新方法论、新问题复盘、已有配置页

**默认输出**
- 配置清单页
- 短规则页
- 相关索引和日志

**Stop condition**
- 形成可执行配置页，并明确下一步动作

**Verifier**
- 页面可读
- 索引可检
- 日志已记

### 3. 代码审查线程

**用途**
- 针对高风险改动做持续 review

**默认输入**
- diff
- 变更文件
- 调用路径

**默认输出**
- 风险点
- 缺失验证
- 开放问题

**Stop condition**
- 给出 grounded finding，或明确“未发现问题但有哪些残余风险”

**Verifier**
- 至少有文件/行级引用
- 明确验证覆盖范围

### 4. 发布跟踪线程

**用途**
- 跟踪一个功能或版本从实现到收尾的状态

**默认输入**
- 需求、变更、反馈、发布评论

**默认输出**
- 当前状态
- 未决问题
- 下一步动作

**Stop condition**
- 进入明确发布完成 / 暂停 / 回滚状态

**Verifier**
- 当前状态有证据
- 未决项清楚

### 5. 外部信息监控线程

**用途**
- 跟踪某类外部资料、技术趋势或重点主题

**默认输入**
- 文章、官方文档、外部链接

**默认输出**
- 新知识页
- 对现有知识页的补充或修正

**Stop condition**
- 有新增结论被沉淀进知识库

**Verifier**
- 有实际写入，不是只停留在聊天分析

## 线程元模板

```text
Thread name:
Purpose:
Default input:
Default output:
Stop condition:
Verifier:
Escalation rule:
What not to do:
```

## 哪些任务不该进入长期线程

- 一次性、小而确定的修补
- 没有复用价值的临时问题
- 不需要滚动上下文的单步任务
- 需求本身还没澄清的任务

## 什么时候该从普通任务升级为长期线程

满足 3 条中的 2 条即可考虑升级：

1. 一周内重复出现 3 次以上  
2. 每次都要重新解释背景  
3. 输出不止一个文件或不止一次交互  

## 当前最适合先固定的线程

1. 知识库编译线程  
2. 配置改造线程  
3. 代码审查线程  

这三类已经反复出现，而且都高度依赖上下文连续性。
