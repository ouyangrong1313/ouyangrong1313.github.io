# Codex配置优化清单：从Harness视角改造AGENTS、Skills、知识库与验证闭环

## 核心结论（一句话）

改善 Codex 配置的关键，不是继续堆更长的提示词，而是把配置重心从 Prompt 转向 Harness：让规则更薄、知识可检索、验证可执行、边界更清晰。

## 要点列表

1. **把 `AGENTS.md` 当成路由层，不要当成知识仓库**
   - 顶层 `AGENTS.md` 只保留目标、边界、流程顺序、技能路由、禁止项
   - 细节规范下沉到更深层 `AGENTS.md`、skill、脚本或知识库页
   - 避免一个总控文档同时承担规则、知识、操作手册三种职责

2. **把 `MyAIWiki` 变成单一事实来源，而不是素材堆放区**
   - 正式知识只落到 `/Users/topsee/ouyangrong1313/MyAIWiki`
   - `.omx/wiki` 只视为运行时内部状态区，不作为正式知识库
   - 每篇文章尽量沉淀成三层：
     - 原文：保留上下文
     - digest：保留结构化提炼
     - wiki：保留可执行结论
   - 真正给 Codex 用的是“短规则页”“清单页”“索引页”，不是长文原稿

3. **为 Codex 建立最小验证闭环，禁止“没验证就算完成”**
   - 把“完成”定义为“验证完成”，不是“模型自称完成”
   - 按任务类型定义最小验证模板：
     - 文档类：链接、索引、日志
     - 脚本类：帮助信息或最小输入样例
     - 代码类：最小 smoke test，不默认全量构建
   - 优先补“最低成本但最能拦错”的验证，而不是上来就全量 lint / test

4. **把 skills 当成标准操作手册，不要把 SOP 塞进系统提示词**
   - 高频任务优先沉淀为 skill：例如文章编译、代码评审、知识卡片生成、变更收尾
   - skill 内容应回答：
     - 什么时候用
     - 用它前要确认什么
     - 具体步骤是什么
     - 最后如何验证
   - 这样比反复在 Prompt 里解释做法更稳定，也更节省上下文

5. **用 Harness 思维重新划分四层职责**
   - Prompt 层：任务表达与高优先级约束
   - Context 层：知识获取与上下文治理
   - Harness 层：权限、验证、反馈、流程、恢复
   - Memory 层：项目级长期记忆与索引
   - 一旦职责清楚，Codex 配置就不容易互相污染

6. **为复杂任务增加“轻量多角色”，不要裸跑长任务**
   - 小任务：单 agent 直接做
   - 中任务：主 agent + 明确验证步骤
   - 大任务：主 agent 负责集成，子 agent 只处理边界清晰的侧任务
   - 关键不是“多 agent 越多越好”，而是避免一个 agent 同时承担规划、执行、审查三种角色

7. **配置原则：模型越强，提示词越薄，但验证要更硬**
   - 不要继续无限增长总控规则文档
   - 要把时间花在：
     - 路由是否清楚
     - 知识是否易取
     - 验证是否真实
     - 权限边界是否稳
   - 未来真正拉开差距的，是 Harness 质量，不是提示词花活

## Codex 改造清单

### 一、`AGENTS.md` 清单

- [ ] 顶层只保留原则、边界、流程顺序、技能路由
- [ ] 把工具使用细则从总控文档迁到 skill 或 repo 内规则
- [ ] 显式写清哪些信息属于“运行时状态”，哪些属于“正式知识”
- [ ] 对“何时直接做、何时计划、何时验证、何时多 agent”写成短判定规则

### 二、`skills` 清单

- [ ] 给高频任务建立 skill，而不是重复口头说明
- [ ] 每个 skill 都要有触发条件和停止条件
- [ ] 每个 skill 都要有最小验证动作
- [ ] skill 负责操作手册，避免大段 SOP 长驻在 Prompt 里

### 三、`MyAIWiki` 清单

- [ ] 建一页“Codex 配置原则”
- [ ] 建一页“默认知识库入口与禁止写入位置”
- [ ] 建一页“验证命令索引”
- [ ] 建一页“任务类型 -> 验证模板”
- [ ] 建一页“多 agent 使用边界”
- [ ] 所有长文都补一版“可执行结论页”

### 四、验证闭环清单

- [ ] 每类任务都有最小验证，不凭模型自评
- [ ] 文档类任务默认检查索引、链接、日志
- [ ] 知识库写入类任务默认检查写入路径是否正确
- [ ] 脚本/配置类任务默认给出一条最小验证证据
- [ ] 复杂任务默认补一个“完成定义”

## 推荐实施顺序

1. **先清路径**：明确 `MyAIWiki` 是正式知识库，`.omx/wiki` 不是。
2. **再清职责**：收缩顶层 `AGENTS.md`，把操作细节下沉。
3. **再补验证**：给文档、脚本、代码三类任务各补一个最小验证模板。
4. **再补 skills**：把最高频、最容易出错的任务沉淀为 skill。
5. **最后补知识索引**：把零散经验编译成短页，供 Codex 按需读取。

## 一个简单判断标准

如果一条配置同时满足下面三点，它大概率该从 Prompt 中拿出来，放进 Harness：

1. 需要反复执行  
2. 需要验证结果  
3. 失败成本高

比如：知识写入路径、完成前检查、权限边界、任务收尾动作。
