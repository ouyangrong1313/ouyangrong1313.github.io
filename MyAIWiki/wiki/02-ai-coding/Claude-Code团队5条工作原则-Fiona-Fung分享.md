## 核心判断：瓶颈没有消失，只是转移了

过去所有软件工程流程（瀑布/敏捷/规范/方法论）**本质上是围绕"写代码太贵"展开的**——工程师时间最贵，所以花大量时间做规划/写需求文档/做评审/开会。

**AI 时代前提变了**：在 Claude Code 团队，**写代码已经不是拖慢速度的环节**。

> "瓶颈没有消失，只是转移了。**转移到了验证、代码评审、安全。**"

> "代码生成太快了，新问题变成了，这些代码对不对，怎么维护，人到底该如何跟得上 review 代码的节奏。"

类比：**从马车到汽车，不只是把马换成发动机**——整个公路系统、交通规则、城市规划都得重新设计。

## 5 个旧领域正在失效 + 5 个新规范

| 旧领域 | 失效原因 | 新规范 |
|---|---|---|
| **规划方式** | 工程速度和产出量完全不同 | **JIT 规划**（对的时间做恰好足够的规划，原型先行，文档后补）|
| **代码所有权** | 谁写的代码变奇怪问题 | 角色模糊化，**人类判断力聚焦在真正需要的地方** |
| **代码评审** | 规模/形态/工具全变 | **Trust but verify**（Claude 干 60-70%，人 review 真正需判断处）|
| **团队构成** | 技能组合需求变 | **Taste is scarce, typing is not**（招聘看创造力和判断力）|
| **知识共享** | 文档不再是唯一真相 | 共享 artifacts 而非文档 |

**附加**：
- **新人入职成本大幅降低**（甚至一周就可以直接开始产出代码）
- **少做前期规划，多做原型**
- **组织架构更扁平**，每个管理者都先从一线干活开始

## 五节详解

### 一. 规划变化：JIT 规划

- **现象**：Fiona 刚加入时写的六个月路线图，**三个月就过时了**（Claude Code 迭代太快）
- **新做法**：**JIT 规划，Just-In-Time，像 JIT 编译一样，在对的时间做恰好足够的规划**
- **砍掉**：写代码之前必须先写设计文档的仪式（Fiona 称之为 "theater，做戏"）
- **加码**：验证（唯一能保证质量的方式是不断把验证流程往前推）
- **讨论方法**：

> "**在技术讨论中，代码赢才牛逼。**"

> "**Building is cheap，做东西很便宜。Arguing is expensive，争吵才昂贵。**"

> "以前争方案要各写一份 PPT、开两轮会讨论，现在十分钟两个原型都出来了，看着实物聊比对着 PPT 吵高效一万倍。"

**结论**：**在 AI 时代，过度规划就是浪费。**

### 二. 自动化变化：肌肉记忆

- **现象**：每遇到一个问题就追问"能不能自动化"
- **Fiona 例子**：每天早上端着咖啡手动总结客户反馈 → 后台自动任务（**重点不是这件事，是这个习惯**）
- **目标**：把"能不能自动化"变成**肌肉记忆**

> "以前自动化成本高，所以只有高频、高重复度、高价值的事情才值得自动化，但现在自动化成本几乎为零，**逻辑就反过来了，几乎所有重复超过 3 次的事情都应该自动化。**"

- **新工具**：**触发器 hook**（Agent+hook 自动化）—— 几分钟就能跑起来
- **效果**：**一个一个小的自动化攒起来，会在你可能都没反应过来的时候，一起长成了一颗苍天大树。**

**行动建议**：
- 别想太大（不要一上来就搭建完整自动化体系）
- **就从今天开始，找一件今天重复做的事情，花十分钟让 Claude Code / Codex 帮你自动化掉**
- 明天再一件，后天再一件，**一个月后回头看，工作方式完全不一样**

### 三. 代码评审变化：Trust but verify

- **核心问题**：过去六个月被其他工程 leader 问得最多的问题 = "**你们人怎么跟得上代码 review 的速度**"
- **答案**：**Trust but verify（信任但验证）**

**Claude 接管的部分（占 review 工作量 60-70%）**：
- 风格检查
- linting
- PR 反馈
- bug 捕捉和修复
- 补充测试

**人类 review 不可替代的部分**：
- 法律合规
- 信任边界
- 安全敏感代码
- 产品方向
- **品味判断**

> "**trust 和 verify 之间的平衡是动态的。今天需要人来做的事情，下一个模型可能就能做了，所以你必须得不断重新评估这条线。**"

类比：**打游戏，每个版本的版本答案都不一样，你不能拿上个版本的攻略打新版本。**

### 四. 团队角色变化：Taste is scarce, typing is not

- **现象**：角色界限变得很模糊
  - PM 在大量写代码
  - 工程师做内容和设计
  - 跨职能 gap 不再是瓶颈
- **具体例子**：工程师修完 bug → Claude 起草文案初稿 → 人类做最终判断 → 当天就能发（以前要等排期几天）
- **本质**：**人类还是做最终决策的那个人，只是不再是写初稿的那个人了**

**招聘两种人**：
1. **有产品 sense 的创意 builder**：识别该做什么，能快速做出原型
2. **有深厚系统背景的工程师**：负责 trust but verify 里最需要人的部分

> "**Taste is scarce, typing is not.** 品味是稀缺的，打字不是。"

> "**subtly wrong is still wrong**，微妙的错误仍然是错误。"

> "**我根本不在乎你一个小时能写多少行代码，我在乎的是你选择去做什么，以及你怎么知道它是对的。**"

> "**当 AI 能把执行速度提升 10 倍的时候，决定性的因素变成了你知不知道应该做什么，以及什么样的结果叫真正的优秀。这，就是品味。**"

### 五. 推动团队变化：团队级 Harness

**团队原则分两类**：
- **左边灰色 = 必须做的硬性要求**
- **右边黑色 = 大家自己摸索的空间**

> "其实本质上，**就是给团队设计了一个 harness，核心就是大的方向统一，具体怎么落地各团队自己定。**"

**Fiona 三条最看重的事**：

1. **保持团队尽可能扁平**，管理者支持各个小组的工作，但保持灵活让人能流动到工作需要的地方
2. **如果 Claude 能做的事情，就让 Claude 做**，这能让我们腾出手来做更难的工作
3. **人不会主动去删除流程，只会在旧流程上面继续叠新流程，所以你得主动站出来，指名道姓地说出哪些流程可以走了**

**第三条最难**——Fiona 的具体例子：

> 每周 review 会议，一大堆人坐在会议室里，所有人都在看电脑，只有轮到自己汇报的时候才抬头说两句 status，说完又低头继续看电脑。
> 
> 然后她问了一句，**我们为什么还在开这个会。**
> 
> 这时候，所有人才意识到，好像，这个会根本不需要。
> 
> 于是，从此，这个会就取消了。

> "无数的流程和会议，当初设立的时候都有道理，但环境变了、工具变了，它们早就失去了存在的意义，只是因为惯性还在那里被迫转着。"

> "**AI 在你的组织里介入的越深，你会发现，很多过去的步骤和流程，其实可以自动化了，如果我们不主动去审视，那这些步骤就会一直在那里，最后，变成纯粹的形式主义。**"

## 3 个 Fiona 也没答案的问题（重要！）

1. **你还需要单独的 iOS 和 Android 团队吗？**（因为现在工程师已经可以更灵活地跨平台工作了）
2. **全自动化的 review 到底能推到多远，在「够快了」和「我们漏掉了什么重要的东西」之间那条线在哪里？**
3. **当角色越来越模糊的时候，怎么确保所有角色都对自己的产出有信心？**

> "即使是 Claude Code 的亲爹团队，也没有把所有事情都想明白。他们也在摸索，很多时候，这就不是一个有标准答案的事情。"

## 我的判断（编译者注）

1. **5 条原则和 [[54万行代码的顿悟]]、[[YC如何进行AI-Native组织改造]] 高度互补**：
   - 那两篇偏"组织/范式"层
   - 本文偏"具体实操"层（JIT 规划 / 自动化肌肉记忆 / Trust but verify / 团队 harness）
   - **三层合并**：范式 → 组织 → 实操 = 完整的 AI-native 蓝图

2. **"JIT 规划"是 PR/原型先行 + 文档后补的最强背书**：
   - 和 [[Codex配置原则总览]] 的"配置必须符合团队语境"原则一致
   - 适用于我们的 Seetong 团队：**别再写长篇设计文档了**，有想法先做原型，能用了再说

3. **"Trust but verify"是 AI 时代 Code Review 的金标准**：
   - 我们 Seetong 团队的 PR review 比例还很低
   - 可以借鉴：**Claude 先干 60-70% 风格/lint/PR/bug/test，人 review 真正需判断处**
   - 立刻可行动：把 Code Review 工具（如 CodeRabbit / Sourcery）集成到我们 Seetong 项目的 PR 流程

4. **"Taste is scarce, typing is not" 是招聘新标准**：
   - 我们 Seetong 团队招聘要不要重新看这个标准？
   - 不再是"几年 iOS 经验 / 写过多少个 APP"
   - 而是："**能不能识别该做什么，能不能快速做出原型，能不能判断对错**"

5. **"3 个 Fiona 也没答案的问题" 是最重要的部分**：
   - **iOS/Android 团队** - 跨端后是否还需要？KMP/Flutter/AI Coding 都让边界变模糊
   - **自动化 review 边界** - 推到哪里为止？漏掉重要东西怎么办？
   - **模糊角色信心** - 当 PM 在写代码、工程师在做设计，每个人怎么对自己的产出有信心？
   - **Fiona 敢公开"没答案"这一点**比很多 AI 公司都强——我们是不是也敢？

6. **"团队级 harness"（灰+黑）是我们 Seetong 团队可以立刻借鉴的**：
   - **灰色 = 硬性要求**（如"所有 PR 必须有 review"、"所有功能必须有埋点"）
   - **黑色 = 团队自摸索**（如"用什么 IDE 写代码"、"怎么组织任务"）
   - 关键：灰色不能太多，否则又变回 command and control

7. **"Pick your noisiest workflow" 是个可立即行动的口号**：
   - 让每个团队成员找一件今天重复做的事
   - 花 10 分钟让 Claude Code 自动化
   - 一个月后看效果
   - **这是 AI 转型的"最低门槛"行动**

8. **Fiona 的 "6 个月前别人问得最多的问题 = 跟不跟得上 review 速度" 是个路标**：
   - 6 个月后，**Fiona 已经能给出标准答案**（Trust but verify，Claude 干 60-70%）
   - 意味着 6 个月前这个问题还是个开放问题
   - **现在问 Seetong 团队：我们的"6 个月前没答案、现在可能有答案"的问题是什么？**
