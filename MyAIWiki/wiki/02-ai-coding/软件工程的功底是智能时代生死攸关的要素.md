# 「软件工程的功底」是智能时代生死攸关的要素

- 原文链接：https://mp.weixin.qq.com/s/Tj_M1OjNbJ9HeJlfMX6qMg?scene=1
- 来源：微信公众号
- 获取时间：2026-05-14

## 核心观点

这篇文章的核心论点很明确：**大模型降低了写代码的门槛，但没有降低软件交付的工程门槛，反而把软件工程能力从“加分项”抬升成了“生存能力”。**

文章通过 Anything 被苹果下架、Cursor 压力测试失效、AI 被用于零日武器化等案例，强调一个事实：

- AI 可以加速代码生成
- 但**代码数量 ≠ 软件质量 ≠ 系统能力**
- 如果没有需求澄清、架构设计、规格约束、测试、评审、治理与反馈闭环，AI 只会更快地产生技术债、安全漏洞和系统性风险

作者把这个问题放回软件工程历史脉络里看：软件工程从 1968 年提出开始，本质任务就是**对抗复杂性**。无论是结构化方法、面向对象、SOA、微服务，还是今天的大模型辅助开发，真正的问题都不是“能不能更快写出代码”，而是“能不能在复杂系统里持续、可靠、安全地交付价值”。

文章特别批判了两种常见误区：

1. **把敏捷误解成不要文档、不要设计、不要架构**  
   结果是用速度掩盖技术债。
2. **把 Vibe Coding 误解成软件工程本身**  
   结果是把理解债务、验证债务和治理债务留到后面爆炸。

作者提出“软件工程 3.0”的概念：在大模型时代，工程并没有过时，反而需要更强。新的研发系统应该是：

- 人
- 模型
- 工具链
- 数据
- 知识库
- 规范体系
- 反馈机制

共同组成一个可验证、可治理、可持续演进的软件生产系统。

## 文章结构梳理

### 1. 为什么 AI 编程热潮反而更需要软件工程

作者先用 2026 年的几个案例开场：

- **Anything** 被苹果两次下架：暴露大量安全漏洞、密钥泄漏、隐私泄露
- **Cursor CEO** 用数百个 GPT-5.2 智能体生成 300 万行 Rust 代码，但连稳定加载 Google 首页都做不到
- **谷歌威胁情报团队** 确认攻击者利用 AI 自动化武器化零日漏洞

这些案例共同说明：

> AI 放大的不只是生产力，也会放大缺陷传播速度和攻击能力。

因此，“能生成代码”不意味着“能交付软件”。

### 2. 软件工程诞生的根本原因：对抗复杂性

文章回到 1968 年北约会议提出“软件工程”的历史背景，指出软件工程不是流程装饰，而是为了对抗复杂性。

作者引用了：

- Fred Brooks《没有银弹》
- Dijkstra
- Hoare

核心观点是：

- 软件天然具有不可见性、耦合性、易变性和一致性要求
- 软件系统失控，几乎都来自对复杂性的低估
- 工程方法、架构范式的演进，本质上都是为了更好管理复杂性

### 3. 国内敏捷实践为什么容易走样

作者认为很多团队的问题不是不会做功能，而是：

- 工程基础还没立住
- 速度已经被强行拉满

比如：

- 没有模块化设计
- 没有持续重构
- 没有自动化测试
- 没有评审文化
- 没有架构治理

却已经追求“两周一个 Sprint”。

这种“敏捷”并不是真正的敏捷，而是**快速积累技术债**。

### 4. Vibe Coding 的真正代价

作者把 Vibe Coding 的风险拆成三层：

#### （1）技术债务
AI 生成的代码可能表面上清晰、完整，但隐藏：

- 不合理抽象
- 重复逻辑
- 隐式耦合
- 边界处理错误
- 安全缺陷

#### （2）理解债务
手写代码时，人是边写边建立心智模型。  
AI 大量生成代码后，团队往往变成“事后理解系统”。

这意味着：

> 不是单纯在 review 代码，而是在 reverse engineer 一个自己没参与设计的系统。

#### （3）治理债务
即使：

- CI 是绿的
- 单测是过的
- Demo 是通的

系统里仍可能埋着：

- 并发问题
- 权限绕过
- 性能瓶颈
- 数据泄露风险

也就是说，AI 生成代码最危险的地方，恰恰是它“看起来非常专业”。

### 5. 历史事故为什么今天仍然 relevant

文章列举了几类经典事故：

- **Therac-25**：软件缺陷导致过量辐射和死亡
- **阿丽亚娜 5**：软件复用脱离上下文验证导致爆炸
- **波音 737 MAX**：系统设计、流程、安全文化共同失效
- **CrowdStrike 2024**：一次更新失误导致全球级蓝屏事故

作者的意思不是回顾历史，而是强调：

> AI 时代并没有让这些教训过时，反而让它们更危险、更容易被放大。

### 6. 软件工程 3.0：不是放弃工程，而是强化工程

作者提出，大模型时代的软件工程重点转向三件事：

#### （1）上下文工程
模型质量高度依赖上下文，需求、规则、约束、架构决策、质量属性、安全规范等都要显式化、结构化。

#### （2）规格驱动开发
通过 Spec-Driven Development，把：

- 业务语义
- 架构边界
- 验收标准
- 质量目标

先定义清楚，再让 AI 在约束下生成代码和测试。

#### （3）驾驭工程
通过：

- 测试门禁
- 安全扫描
- 代码评审
- 反馈闭环
- 运行时监控
- 发布治理

把 AI 输出纳入可控边界。

文章的落点很清楚：

> AI 的输出可以非确定，但软件交付不能非确定。

## 我的理解

我觉得这篇文章最有价值的地方，不是“AI 不是银弹”这种口号，而是它把问题说到了一个很具体、很疼的层面：

### 1. 大模型正在把“写代码能力”商品化
过去，很多团队把“能不能很快写出功能”视为竞争力。  
但现在，生成代码的成本越来越低，这部分能力正在快速被商品化。

所以真正稀缺的，开始变成：

- 你能不能定义清楚问题
- 你能不能划清边界
- 你能不能识别风险
- 你能不能把一个系统长期维护下去

这其实就是软件工程的基本功。

### 2. Vibe Coding 最大的问题不是乱写，而是“看起来像对的”
如果 AI 生成的是明显错误的代码，反而好处理。  
真正麻烦的是：

- 它能跑
- 它像样
- 它通过了初步验证
- 但它把复杂性、耦合和风险藏到了系统深处

这类问题会让团队在短期内很爽，在中期开始焦虑，在长期付出极高代价。

### 3. 工程的重心正在从“生产代码”转向“治理系统”
以前工程师主要是生产代码的人。  
以后更重要的角色可能是：

- 需求澄清者
- 架构边界设计者
- 上下文维护者
- 质量守门人
- 知识系统建设者

也就是说，**人类的价值正在往“定义、判断、治理、演化”迁移。**

### 4. 软件工程 3.0 本质上是“AI 时代的系统工程”
我很认同文中“软件工程 3.0”这个提法。  
它不是简单说“加个 AI 工具”，而是说：

> 研发系统本身要升级。

这和很多团队现在的问题正好对应：

- 只上了 AI IDE
- 没有知识库
- 没有规格体系
- 没有 prompt 约束
- 没有质量门禁
- 没有回流机制

结果就是 AI 只是在旧流程里加速制造新问题。

## 对 AI Coding / Agent 落地的启发

这篇文章对实际落地有几个很强的启发：

### 1. 不要把 Agent 当成“自动程序员”
更合理的定位应该是：

- 帮你做信息整理
- 帮你做方案展开
- 帮你做测试生成
- 帮你做执行加速

但关键决策仍然需要工程约束来收口。

### 2. 知识库与规范库会越来越重要
如果上下文决定生成质量，那知识库就不是“可有可无的文档区”，而是模型输出质量的地基。

### 3. Skill / Workflow / Spec 是治理 AI 的关键抓手
真正成熟的 AI Coding 系统，不会只靠 prompt 灵感，而会逐步沉淀成：

- Skill
- Workflow
- Spec
- Guardrail
- Eval

这些才是把“偶尔有效”变成“可持续复用”的关键。

### 4. 评估 AI Coding，不能只看速度
至少要同时看：

- 正确性
- 可维护性
- 安全性
- 可验证性
- 上下文一致性
- 演进成本

否则会掉进“短期效率高，长期总成本更高”的陷阱。
