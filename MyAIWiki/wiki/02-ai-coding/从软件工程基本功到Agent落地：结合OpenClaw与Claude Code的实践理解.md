# 从软件工程基本功到 Agent 落地：结合 OpenClaw 与 Claude Code 的实践理解

## 为什么要写这篇

《[[软件工程的功底是智能时代生死攸关的要素]]》把一个关键问题说透了：**AI 提升了代码生成效率，但没有替代软件工程。**

这篇延伸笔记想回答更落地的问题：

> 如果我们已经在用 OpenClaw、Claude Code、Codex CLI、Skills、知识库、MCP 工具，怎么把“软件工程基本功”真正落到 Agent 体系里，而不是停留在口号上？

我的结论很直接：

> Agent 落地的真正门槛，不是模型够不够强，而是团队有没有把“上下文、规范、流程、验证、记忆、反馈”这些软件工程能力产品化、系统化。

也就是说，**Agent 不是软件工程的替代品，而是软件工程能力的放大器。**

---

## 一、为什么很多 AI Coding 看起来很强，落地却很脆

在个人体验和团队实践里，很多 AI Coding 系统都容易掉进同一个陷阱：

- Demo 很惊艳
- 单次任务很快
- 但一进入真实项目就开始漂移

常见表现包括：

1. **上下文漂移**
   - 这个回合说的是 A，下一回合开始按 B 理解
   - 不知道项目已有代码风格和架构边界
   - 不知道哪些接口是真实的、哪些只是需求草稿

2. **局部正确，全局失真**
   - 单个函数写得像样
   - 但放进现有系统后边界不对、状态不对、耦合变重

3. **能写，不能收口**
   - 会生成代码
   - 不会主动验证
   - 不会判断哪些改动需要测试、哪些结论需要证据

4. **会回答，不会积累**
   - 这一轮解决了
   - 下一轮又从零开始
   - 教训没有沉淀成 skill、memory、知识库、规范

这些问题看起来像“模型不够聪明”，其实更像：

> **工程系统没有把模型纳入可控边界。**

---

## 二、把“软件工程基本功”翻译成 Agent 系统语言

如果把传统软件工程基本功翻译成 Agent 体系里的对象，我会这样对应：

| 软件工程基本功 | Agent 体系里的对应物 |
|---|---|
| 需求澄清 | Prompt 约束、任务分解、Skill 输入模板 |
| 架构边界 | 子 agent 边界、工具权限边界、目录结构、职责划分 |
| 编码规范 | Skill、AGENTS.md、IDENTITY.md、项目约束 |
| 测试与验证 | completion contract、tool evidence、lint/test/build/check |
| 配置管理 | gateway/config、MCP 工具白名单、环境分层 |
| 版本经验沉淀 | memory/、MEMORY.md、知识库 wiki、技能库 |
| 发布治理 | 审批、幂等操作、外部写操作前确认、可回滚策略 |

这个映射很重要，因为它说明：

> 今天做 Agent 系统，不是在发明一套完全不同的东西，而是在把传统工程控制点重新编码成“Agent 可执行的结构”。

---

## 三、OpenClaw / Claude Code 真正有价值的地方，不是“会写代码”

很多人第一次看 OpenClaw 或 Claude Code，会把注意力放在：

- 能不能自动改代码
- 能不能自动调工具
- 能不能一轮做很多事

但如果结合软件工程视角看，真正有价值的地方其实是这些：

### 1. 它们让“约束”变成可执行对象

比如 OpenClaw 里：

- `AGENTS.md`
- `SOUL.md`
- `IDENTITY.md`
- `USER.md`
- `SKILL.md`
- `completion_contract`
- `tool_discipline`

这些本质上都不是装饰文档，而是在把团队经验、边界条件、质量要求、行为守则变成“每轮都会被加载和执行的约束系统”。

这很像把：

- 开发规范
- 评审 checklist
- 架构原则
- 操作手册

从静态文档，变成了运行时治理层。

### 2. 它们让“上下文工程”变成了一等公民

传统 IDE 的问题是：
- 上下文大多靠人脑记
- 或者靠一次性 prompt 输入

而 OpenClaw 这种体系里，上下文被拆成了多层：

- 长期记忆：`MEMORY.md`
- 当日记忆：`memory/YYYY-MM-DD.md`
- 项目知识：wiki / raw / entities
- 行为规则：`AGENTS.md` / `SKILL.md`
- 当前任务上下文：chat transcript + tool results

这就意味着，系统不是只在“回答问题”，而是在持续维护一个认知底座。

### 3. 它们让“验证前完成”成为默认流程

很多 AI Coding 最大的问题，是太快进入“我已经做完了”。

但好的 Agent 系统会不断提醒：

- 最终结论要有证据
- 能 test 就 test
- 能 build 就 build
- 能 code search 就别靠猜
- 外部写操作前先确认

这本质上就是把软件工程里的 **质量门禁** 内嵌到了 agent loop 里。

---

## 四、为什么 Skill 是 AI 时代最像“工程资产”的东西

如果只让我选一个最值得长期沉淀的对象，我会选 **Skill**。

因为 Skill 同时具备几个关键特征：

1. **可复用**  
   不是某一轮对话里的灵感，而是下次还能直接用。

2. **可演进**  
   做错一次，就可以把修正写进 Skill，未来持续复利。

3. **可传递**  
   一个团队内部可以共享，让经验不再只存在于最强那个人脑子里。

4. **可约束**  
   它不只是“建议”，而是 agent 真实执行前会读取的操作规范。

从这个角度看，Skill 很像软件工程里的：

- 框架脚手架
- 最佳实践库
- 领域模板
- 操作 runbook
- 自动化流程定义

也就是说：

> Skill 不是“prompt 小抄”，而是 AI 时代的软件工程资产。

---

## 五、知识库为什么会成为 Agent 系统的护城河

文章里说“上下文工程”很重要，这件事在真实实践里最直接的落点就是：**知识库质量决定 Agent 上限。**

原因很简单：

### 1. 模型强，不代表项目上下文完整
模型知道通用知识，但不知道：

- 你的真实代码结构
- 团队历史决策
- 哪些需求踩过坑
- 哪些接口文档不可信
- 哪个方案曾经被否掉以及为什么

这些东西不进知识库，就只能不断在聊天里重复补课。

### 2. 知识库不是存资料，而是存“可调用语义”
真正有效的知识库，不只是堆文章，而是能支持：

- 检索
- 关联
- 汇总
- 回溯
- 复用

像 Obsidian / wiki / entities / raw / digest 的组合，本质上是在构建一个“可被 Agent 调用的长期语义层”。

### 3. 知识库与 Skill 组合后，才会产生复利
单独有知识库，可能只是能查资料。  
单独有 Skill，可能只是能按流程执行。  
两者结合，才会变成：

- 用知识库提供高质量上下文
- 用 Skill 约束推理与执行路径
- 用 memory 持续沉淀经验
- 用工具补事实证据

这就是一个真正会“越用越强”的系统。

## 六、怎么避免把 Agent 用成“高速制造技术债的机器”

如果回到《[[软件工程的功底是智能时代生死攸关的要素]]》的主线，我觉得最该警惕的是：

> 不要把 Agent 当成“绕过工程控制”的捷径。

更稳妥的做法是：

### 1. 让 Agent 先查证，再输出
比如：
- 先读代码，再写结论
- 先查文档，再给方案
- 先跑工具，再下判断

### 2. 让 Agent 默认留下痕迹
比如：
- 更新 memory
- 生成文档
- 把教训写进 skill
- 把复盘沉淀到 wiki

### 3. 让 Agent 受到明确边界约束
比如：
- 哪些操作可直接做
- 哪些操作必须确认
- 哪些目录能写
- 哪些工具可调用
- 哪些外部动作不能越权

### 4. 让 Agent 结果必须能被验证
比如：
- 引用真实代码位置
- 引用真实接口名
- 引用真实日志 / 查询结果
- 不允许靠“应该”“大概”“通常”冒充结论

这四条看起来像在“限制 AI”，其实是在**保护 AI 产出能够进入真实工程系统**。

## 七、对团队建设的启发：以后拼的不是谁更会下 Prompt

很多人会把 AI 落地竞争，理解成：

- 谁 Prompt 更会写
- 谁模型更贵
- 谁 IDE 更先进

但我越来越觉得，长期真正拉开差距的是：

1. **谁更会沉淀 Skill**
2. **谁更会维护知识库**
3. **谁更会做上下文工程**
4. **谁更会设计验证闭环**
5. **谁更会把经验转成可执行约束**

换句话说：

> 未来团队的竞争力，不是“谁先用了 AI”，而是“谁先把 AI 纳入工程系统并形成复利”。

---

## 八、一个我认可的判断框架

如果以后要判断一个 Agent 系统是不是靠谱，我会看这几个维度：

### 1. 上下文质量
- 有没有长期记忆
- 有没有项目知识库
- 有没有历史决策可回溯

### 2. 约束质量
- 有没有行为边界
- 有没有技能规范
- 有没有明确的完成标准

### 3. 验证质量
- 会不会主动找证据
- 会不会运行必要检查
- 会不会区分事实、推断、建议

### 4. 复利能力
- 会不会把一次经验沉淀下来
- 会不会把教训写进 Skill / Memory / Wiki
- 会不会越用越稳

### 5. 协作能力
- 能不能拆分任务
- 能不能管理多个子 agent
- 能不能在长链路任务中保持状态一致

这几个维度，其实对应的仍然是软件工程问题，只不过外壳变成了 Agent。

---

## 九、我的最终结论

如果只用一句话概括这篇延伸笔记，我会写：

> 在 OpenClaw、Claude Code、Codex CLI 这些系统里，真正的护城河不是模型本身，而是把软件工程基本功转译成“可执行上下文 + 可复用 Skill + 可验证流程 + 可积累知识”的能力。

所以，软件工程在 AI 时代没有过时，只是换了一种存在形式：

- 以前体现在代码结构、测试体系、发布流程里
- 现在还进一步体现在 Prompt 结构、Skill 设计、知识库组织、Agent 边界、工具调用纪律和 Memory 机制里

看起来我们是在“做 Agent”，本质上我们仍然是在做一件老而重要的事：

> **用工程方法驯服复杂性。**
