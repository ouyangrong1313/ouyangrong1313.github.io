# Codex配置原则总览

## 核心结论（一句话）

当前这套 Codex 配置的主线，不是继续增加规则密度，而是把规则、验证、线程、工具面和共享记忆拆开，各自放在最合适的位置。

## 这页解决什么问题

当配置页越来越多时，真正缺的不是更多规则，而是一个稳定入口，回答下面几个问题：

1. 顶层 `AGENTS.md` 应该承担什么，不该承担什么  
2. 哪些规则应该留在知识库页，而不是塞回系统提示词  
3. 长任务、验证、工具入口和多 agent 边界应该分别去哪一页查  

这页就是当前 Codex 配置的统一导航。

## 当前配置主线

### 1. 总控文件要薄

`AGENTS.md` 的职责应集中在：

- 路由
- 边界
- 默认工作顺序
- 禁止项

它不该重新承担知识仓库和 SOP 手册职责。

### 2. 正式知识要外置

正式配置知识默认沉淀到 `MyAIWiki`，而不是：

- 聊天记录
- 临时上下文
- `.omx/wiki`
- 越来越长的顶层规则文件

### 3. 完成必须带 verifier

所有高频任务都应该有最小验证模板。  
没有 verifier，默认视为未完成。

### 4. 长任务需要长期线程

高频、重复、滚动推进的任务，不该每次从零重新解释，而应进入固定线程类型。

### 5. 能力边界先看工具入口

很多失败不是模型问题，而是入口选错了。  
浏览器态、MCP、Shell、GUI 自动化各有边界。

### 6. 多 agent 不是默认答案

只有可拆、可并、可合的任务，才值得开并行。  
否则默认单 agent。

## 一个简单判断标准

如果一条做法同时满足以下 3 条，就不该只停留在聊天里：

1. 会重复发生  
2. 容易出错  
3. 结果需要验证  

这类内容，应优先沉淀为知识页或短规则页。
