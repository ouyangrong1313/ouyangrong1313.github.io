# Karpathy LLM Wiki 系统

## 核心观点

**你可能根本不需要 RAG**。
只需要一个**聪明的文件夹结构 + LLM**。

Karpathy 的核心理念：**不要让 AI 每次帮你从零检索答案，而是让 AI 帮你建一个持续生长的知识库**——你喂进去的每一条内容都真正沉淀下来，越积越厚。

就好比图书馆和搜索引擎的区别。搜索引擎每次给你找，找完就结束了。图书馆是你自己建的，每进来一本书都有位置，书和书之间有关系，越建越有价值。

## Raw vs Wiki 双层结构

| | raw/ (clippings/) | wiki/ |
|--|------|-------|
| 本质 | 冰箱里的生食材 | 做好的菜 |
| 谁写的 | 作者原文，你搬运 | LLM 重新组织 |
| 格式 | 混乱无结构 | 统一格式+摘要+索引 |
| 关联 | 文件孤立 | [[链接]] 形成知识网络 |
| 能否提问 | 不能 | 能，直接问 LLM |
| 谁维护 | 你（往里丢就行） | LLM（编译、更新） |

---

## 三层架构

```
my-vault/
├── clippings/ ← 原始内容（Web Clipper 存入），只读不改
├── wiki/
│   ├── sources/ ← 每篇文章/视频的摘要页
│   ├── entities/ ← 人物、公司、工具的专属页面
│   ├── concepts/ ← 概念主题页
│   ├── index.md ← 总目录（所有页面的导航入口）
│   └── log.md ← 操作日志
└── CLAUDE.md ← 本文件
```

**目录职责**：
- `clippings/`：原始内容区，只读不改
- `wiki/sources/`：每篇文章/视频的摘要页
- `wiki/entities/`：人物、公司、工具的专属页面
- `wiki/concepts/`：概念主题页

## 三核心操作

```
Ingest（摄入）→ Query（查询）→ Lint（体检）
```

### Ingest（收录新内容）

当说「ingest」或「收录这篇文章」时：
1. 读取 `clippings/` 中指定的文章（或所有新文章）
2. 在 `wiki/sources/` 创建摘要页，文件名与原文一致
3. 提取文章涉及的人物、公司、工具 → 创建或更新 `wiki/entities/` 对应页面
4. 提取文章涉及的概念、方法 → 创建或更新 `wiki/concepts/` 对应页面
5. 更新 `wiki/index.md`，将每个新页面加入对应分类，附一句话摘要
6. 在 `wiki/log.md` 追加一条记录：`## [日期] ingest | 文章标题`

> 一篇文章通常会创建 **5–15 个 wiki 页面**

### Query（查询知识库）

当问问题时：
1. 先读 `wiki/index.md` 定位相关页面
2. 精读相关的 sources / entities / concepts 页面
3. 综合回答，引用来源用双链格式 `[[页面路径]]`
4. 如果回答产生了有价值的新总结，询问是否存入 `wiki/concepts/`

### Lint（健康检查）

当说「lint」时：
1. 检查是否有页面之间存在矛盾的描述
2. 检查是否有页面没有被任何其他页面引用（孤立页面）
3. 检查文章中提到但还没有独立页面的重要概念
4. 输出检查报告，等确认后再执行修复

## 什么时候用 RAG

| Obsidian 方案够用 | 需要 RAG |
|--------------|----------|
| <10 人小团队 | 数万~百万级文档 |
| <1000 文档 | 毫秒级响应需求 |
| 个人研究者 | 多用户并发查询 |
| 需要可视化、可编辑 | 企业级部署 |

> **为什么不先从 Obsidian 开始？如果发现规模超出了它能处理的范围，再迁移到 RAG。**

## 核心原则

| 原则 | 说明 |
|------|------|
| LLM 写，你读 | wiki 是 LLM 的领域 |
| 累积式增长 | 每次查询都增强 wiki |
| 索引即检索 | 文件结构替代检索算法 |
| 透明可解释 | 所有数据人眼可读 |
| 不留孤立页面 | 每个 wiki 页面至少被一个其他页面引用 |

---

## 传统 RAG vs Obsidian 方案对比

| 维度 | 传统 RAG | Obsidian 方案 |
|------|----------|---------------|
| 技术复杂度 | 向量数据库 + 嵌入模型 + 检索流程 | 只有 Markdown |
| 成本 | 服务器 + API 调用 | 几乎免费 |
| 可解释性 | 黑盒 | 完全透明 |
| 维护成本 | 需要技术背景 | 任何人都能用 |
| 设置时间 | 数小时~数天 | 10 分钟 |
