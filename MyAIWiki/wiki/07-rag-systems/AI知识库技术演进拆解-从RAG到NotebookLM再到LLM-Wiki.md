# AI 知识库技术演进拆解：从 RAG 到 NotebookLM，再到 LLM Wiki

- 原文链接：https://mp.weixin.qq.com/s/UtN4_bhOYBV3OnYIUGN4LQ
- 来源：微信公众号
- 获取时间：2026-05-14

## 核心观点

这篇文章试图回答一个很多做 AI 知识库的人都会关心的问题：

> NotebookLM 到底是不是 RAG？如果是，为什么它的体验和传统 RAG 产品完全不像一回事？而再往后，LLM Wiki 又代表了什么新的演进方向？

作者给出的判断很明确：

1. **NotebookLM 本质上仍然是 RAG**
2. 但它不是常见的“低配 RAG”，而是一套被高度产品化、黑盒化、高工程化包装起来的高阶知识库系统
3. **LLM Wiki 则不是在查询时拼答案，而是提前把知识编译成可持续演化的结构化 Wiki**

所以文章最后把技术路径总结成了三阶段演进：

- **第一阶段：低配 RAG**
  - 切块 → 向量化 → 检索 → 回答
- **第二阶段：NotebookLM 类产品化 RAG**
  - 文档理解 → 多索引 → Retrieval and Ranking → Context Engineering → Source Grounding → 知识产品输出
- **第三阶段：LLM Wiki / 深度知识库**
  - 知识抽取 → 实体识别 → 主题页生成 → 关系链接 → 冲突检测 → 增量更新 → 持续演化

作者最想说明的，其实不是“RAG 过时了”，而是：

> 真正好用的 AI 知识库，不是把向量库接起来就行，而是要把文档理解、检索、排序、上下文组织、引用追溯和知识沉淀全部工程化、产品化。

## 文章结构梳理

### 1. 问题起点：NotebookLM 为什么看起来不像 RAG

作者一开始就说自己很推崇 NotebookLM，因为它：

- 只基于用户上传资料回答
- 大幅降低幻觉
- 检索和输出质量已经接近“数字分身 / 同事 skill”的最低要求

但一个让人困惑的地方是：

- 如果 NotebookLM 是 RAG
- 为什么在产品体验里几乎看不到传统 RAG 那些明显痕迹？
  - TopK
  - 向量召回
  - rerank
  - score threshold
  - chunk 参数

作者的判断是：

> 不是没有这些东西，而是 Google 把这些工程链路自动化、黑盒化了。

所以 NotebookLM 的本质不是“不是 RAG”，而是：

> 一套被长上下文、检索排序、上下文工程、结构化笔记和 source grounding 包装过的高阶 RAG 系统。

### 2. NotebookLM 的可能技术路径：七层黑盒

作者根据 Google 官方披露线索和自己的实践，把 NotebookLM 拆成七层：

1. Source 接入
2. 文档理解
3. 多粒度 Chunk
4. 多索引
5. Retrieval and Ranking
6. Context Engineering
7. 答案生成

这个拆法非常有价值，因为它把 NotebookLM 从“神秘产品”还原成一套可理解的工程系统。

#### （1）Source 接入
这里的关键不是“上传文件”，而是“把资料作为一个可引用的事实来源加入 Notebook”。

作者认为 NotebookLM 的对象模型更像：

- Notebook
  - Sources
  - Notes
  - Conversations
  - Indexes
  - Settings

这意味着系统的基本抽象不再是“单个聊天上传文件”，而是“围绕一组 sources 构建一个可追溯的研究空间”。

#### （2）文档理解
这是全文最重要的一节之一。

作者反复强调：

> 很多人以为 RAG 的难点在向量库、embedding、rerank，但真正决定知识库上限的，往往是前面的文档理解。

因为真实文档里有价值的信息不只在正文，还有：

- 标题层级
- 章节关系
- 页码
- 表格
- 图片
- 图注
- 目录
- 脚注
- 参考文献
- 作者、来源、时间等元数据

如果只是粗暴把 PDF 转文本，就会把这些结构全部打平。  
后面无论切块、检索、排序、引用都会变得不稳定。

作者举了很典型的例子：

- 只抽出一句“2024 年公司毛利率为 38.4%”
- 却不知道它属于哪个章节、是不是管理层讨论、有没有限定条件

所以真正要做的不是“转文本”，而是：

> 尽可能还原文档原本的知识结构。

包括：
- 章节树
- 表格结构化
- 页码绑定
- 上下文关系
- 引用可追溯性

作者甚至明确说：

> 文档理解可能是整个 RAG 产品里最难的一层。

我觉得这是全文最重要的判断之一。

#### （3）多粒度 Chunk
传统 RAG 常讨论 chunk size，作者认为高阶 RAG 系统不会只有一种 chunk，而会维护多种粒度：

- Source 级别
- Chapter 级别
- Section 级别
- Paragraph 级别
- Chunk 级别
- Sentence 级别

因为不同问题需要不同粒度的证据：
- 问细节 → 小粒度
- 问章节观点 → section 粒度
- 问多文档总结 → source summary / chapter summary

所以 chunk 不能只是“切一刀”，而是要成为：

- 带 source_id
- 带 section_id
- 带页码
- 带位置
- 带 citation_ref

的可溯源逻辑单元。

#### （4）多索引
作者强调一个很关键的观点：

> 高阶 RAG 不是不要向量库，而是不能只有向量库。

NotebookLM 一类系统应该同时维护：

- Vector Index
- Keyword / BM25 Index
- Metadata Index
- Document Tree Index
- Citation Index
- Conversation / Note Index

为什么？因为：

- 向量检索适合语义相似
- 关键词检索更适合专有名词、标题、页码、公式
- 元数据适合过滤
- 文档树适合章节级扩展
- 引用索引适合回溯

所以 Retrieval and Ranking 很可能是一套混合检索体系，而不是单一路径。

#### （5）Retrieval and Ranking
作者认为 NotebookLM 不可能把用户问题直接丢给向量库搜一下，而是会先做问题理解，生成 query plan。

比如用户问：
“这几份资料里，对 AI 知识库未来发展最重要的判断是什么？”

系统背后需要拆成：
- 每份资料里的核心判断
- 共同点
- 冲突观点
- 趋势提炼
- 综合结论

然后再做：
- 多路召回
- 候选合并
- 去重
- 多条件排序

排序标准不只是相关性，还包括：
- 来源可信度
- 覆盖度
- 引用质量
- 上下文充分性
- 是否适合作为证据

这部分揭示了 NotebookLM “看起来很靠谱”的原因：

> 它不是简单找到相似片段，而是在做问题理解、证据组织和候选排序。

#### （6）Context Engineering
这一节很重要，因为它把高阶知识库和低配 RAG 区分开了。

作者认为 NotebookLM 进入生成前，会先构造一个 Context Package，里面可能包括：

- 用户问题
- 问题意图
- 候选证据
- source 信息
- 章节上下文
- 前后文扩展
- 多份资料摘要
- 历史对话
- 用户笔记
- 引用映射
- 回答约束
- 资料不足时的处理规则

作者特别提醒：

> 百万 Token 上下文不是让你无脑塞全文，而是让你在证据之外放入更多结构化辅助信息。

这句话很关键。因为很多人误以为长上下文就可以替代检索和结构化，其实不是。

#### （7）答案生成
最后生成阶段，NotebookLM 一类产品需要遵守：

- 只基于资料回答
- 资料不足时保守回答
- 关键结论绑定证据
- 有冲突要指出冲突
- 推断不能冒充事实

所以它真正的核心是：

> 可溯源的资料生成，而不是“看起来会回答”。

### 3. Karpathy 的 LLM Wiki：从查询式知识库走向编译式知识库

文章中段把 Karpathy 的 `LLM Wiki` 拉进来，形成一个更完整的对照。

作者总结得很清楚：

#### 传统 RAG / NotebookLM 类路径
- 上传资料
- 查询时召回片段
- 临时综合回答

#### LLM Wiki 路径
- 原始资料不动
- LLM 读取资料
- 增量维护结构化 Wiki
- 不断更新实体页、主题页、交叉引用、矛盾点和综合结论

也就是说：

> 不是问一次拼一次答案，而是提前把知识编译成一个持续演化的结构。

作者因此区分了两层：

- RAG 工程自动化
- Wiki 化知识沉淀

NotebookLM 更接近前者，并带一点 Wiki 化倾向；LLM Wiki 则更强调长期知识结构沉淀。

### 4. 文章的最终结论：AI 知识库的三阶段演进

作者最后把整条路线归纳为：

#### 第一阶段：低配 RAG
- 资料切块
- 向量化
- 检索
- 回答

#### 第二阶段：NotebookLM 类产品化 RAG
- 文档理解
- 多索引
- Retrieval and Ranking
- Context Engineering
- Source Grounding
- 知识产品输出

#### 第三阶段：LLM Wiki / 深度知识库
- 知识抽取
- 实体识别
- 主题页生成
- 关系链接
- 冲突检测
- 增量更新
- 持续演化

这不是替代关系，而是逐层叠加：

- 低配 RAG 是底座
- NotebookLM 把底座做成产品级系统
- LLM Wiki 再把问答系统推进为长期知识资产系统

作者最后一句很有力量：

> 各位依旧逃离不了 RAG。

意思不是说只有 RAG 一条路，而是说：

- 知识库这件事绕不开检索、结构化、证据组织和 grounding
- 只是产品成熟后，这些工程细节不再暴露给用户

## 我的理解

我觉得这篇文章最有价值的地方，是它把“AI 知识库为什么难”这件事，真正说到了工程深处，而不是停留在向量库调参层面。

### 1. RAG 的核心难点，不在召回，而在资料有没有被正确理解

这一点我非常认同。  
很多团队一上来就在：
- 换 embedding
- 调 TopK
- 加 rerank
- 改 prompt

但如果前面的文档解析就错了，后面都是在垃圾上做优化。

作者那句逻辑链很有杀伤力：

- 垃圾解析
- 垃圾切块
- 垃圾向量化
- 垃圾召回
- 模型一本正经地基于垃圾回答

这几乎就是很多企业知识库效果差的根源。

### 2. NotebookLM 的优势，不是“没有 RAG”，而是“把 RAG 产品化了”

这是一个特别重要的澄清。  
很多人看到 NotebookLM 体验好，就以为它跳过了 RAG。其实不是，它只是把：

- 文档理解
- 多索引
- 检索排序
- 上下文装配
- 引用追溯

这些东西做得足够深、足够黑盒、足够自动。

所以真正值得学的不是“它不用 RAG”，而是：

> 它把 RAG 从开发者工具链，做成了用户无感的知识产品。

### 3. LLM Wiki 的意义，在于把“回答问题”推进到“沉淀知识”

这也是我觉得文章后半段最有启发的点。  
传统 RAG 更像一个“查询系统”，每次问都临时拼答案。  
而 LLM Wiki 更像一个“编译系统”，把知识提前组织进：

- 实体页
- 主题页
- 关系链接
- 冲突点
- 综合结论

这个方向非常适合长期知识库建设。

### 4. AI 知识库的未来，不会停在问答，而会走向知识编译与持续演化

这篇文章实际上在提示一个趋势：

- 初级阶段：能问答
- 中级阶段：能引用、能回溯、能生成知识产品
- 高级阶段：能持续维护知识结构本身

这和 Obsidian、Wiki、Second Brain、实体记忆这些方向其实正在汇流。

---

## 对 AI 知识库 / RAG 系统建设的启发

### 1. 文档理解优先级要高于向量库调参
先解决“文档结构还原”，再谈召回质量。

### 2. 不要只做单一路径向量检索
真实知识库一定是混合检索。

### 3. Chunk 必须可溯源、可回到原文结构
否则引用、审核、追踪都会不稳定。

### 4. Context Engineering 是高阶知识库的关键差异点
不是把资料塞进去，而是把证据组织成模型可用的上下文包。

### 5. 长期目标应该从“问答系统”升级到“知识沉淀系统”
问答只是入口，知识结构化和持续演化才是护城河。
