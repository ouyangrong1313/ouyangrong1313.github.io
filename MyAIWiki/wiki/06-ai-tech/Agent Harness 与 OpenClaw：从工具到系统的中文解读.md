# Agent Harness 与 OpenClaw：从工具到系统的中文解读

## 为什么这篇文章对 OpenClaw 特别重要

如果只把 OpenClaw 看成一个「能调用模型、能连几个工具」的聊天壳子，那会低估它。

从 Agent Harness 的视角看，OpenClaw 的价值不在于“又一个 AI 客户端”，而在于它正在把一个原本无状态的 LLM，组织成一个**可以持续工作、可以调用外部能力、可以沉淀经验、可以跨会话延续、可以被验证和被约束的系统**。

换句话说：

> OpenClaw 的重点不是把模型接进来，而是把模型放进一套可运行的 Harness 里。

这也是为什么同样一个模型，在普通聊天窗口、在 Cursor、在 Claude Code、在 OpenClaw 里的表现会明显不同。

差异通常不只来自 Prompt，而是来自 Harness 设计。

---

## 用一句话理解 OpenClaw 在做什么

**OpenClaw 不是“给模型套壳”，而是在构建一个可复利的 Agent 运行时。**

这里的“复利”很关键：
- 今天分析过的问题，明天可以从 memory 里继承
- 今天踩过的坑，后天可以写进 skill
- 今天磨过的流程，下次可以变成自动化路径
- 今天的一个会话，不只是一次回答，而是在增强未来的系统能力

这和文章里说的 Harness 本质高度一致。

---

## 对照文章看 OpenClaw：哪些能力已经是 Harness 了

下面按文章里的核心组件，逐个映射到 OpenClaw。

### 1. 编排循环：OpenClaw 的主循环就是 Agent 的心跳

文章里说，Harness 的起点是一个 TAO / ReAct 循环：
- 思考
- 行动
- 观察
- 再行动

OpenClaw 的实际运行方式也非常像这个模式：
- 用户发出请求
- 系统注入上下文、规则、技能、环境信息
- 模型判断要不要调用工具
- 调工具拿结果
- 再基于结果继续推理
- 最后输出答案或继续进入下一轮

表面看是“会调工具的对话”，本质上已经是一个运行时循环。

更重要的是，OpenClaw 不是只做单轮工具调用，它支持：
- 多轮推进
- 背景任务
- 子 Agent 分派
- 长任务继续执行
- Heartbeat 主动推进

这说明它已经不只是“函数调用增强聊天”，而是具备了 Harness 的运行骨架。

### 2. 工具系统：OpenClaw 的工具层已经不是玩具

文章提到，生产级工具层要解决的不只是“把函数暴露给模型”，而是：
- 如何注册
- 如何约束参数
- 如何描述能力边界
- 如何区分读写风险
- 如何组织不同来源的能力

OpenClaw 在这件事上已经很像一个成熟 Harness：
- 本地工具：`read`、`write`、`edit`、`exec`
- 运行态工具：`process`、`session_status`
- 协作工具：`sessions_spawn`、`sessions_send`
- 外部 MCP：TAPD、神策、友盟、企微等
- 媒体工具：图像、音频、视频生成与分析

这里最有意思的一点是：
**OpenClaw 的工具不只是“功能更多”，而是已经开始区分工具语义。**

比如：
- `read` / `write` / `edit` 是文件系统操作
- `exec` 是高自由度执行能力
- `sessions_spawn` 是子 Agent 编排能力
- MCP 是外部业务系统能力

这种语义分层，本身就是 Harness 设计的一部分。

### 3. 记忆系统：OpenClaw 已经把“会话”升级成“连续性”

文章里说，记忆系统是让 Agent 从“单次问答”走向“持续工作”的关键。

OpenClaw 这点很典型：
- `memory/YYYY-MM-DD.md`：日记式短中期记忆
- `MEMORY.md`：长期记忆
- `SOUL.md`：人格与行为基调
- `USER.md`：用户画像与偏好
- `AGENTS.md`：工作规范与操作守则
- 项目专属记忆目录：把上下文按项目沉淀

这其实非常像文章提到的多层记忆结构：
- 常驻加载的轻量信息
- 按任务加载的项目上下文
- 更长期的经验沉淀

更妙的是，OpenClaw 这里的记忆不是“模型自动学会了什么”，而是**用文件系统做了可读、可写、可维护的外部记忆**。

这非常符合 Harness 思路：
- 把记忆从模型内部幻想里拿出来
- 做成外部、可审计、可更新的持久层

### 4. 上下文工程：OpenClaw 的真正价值之一就是“上下文装配”

很多人低估了 OpenClaw 最强的地方：它不是只会把用户问题发给模型，而是会在调用前把大量上下文组织进去。

例如：
- 系统规则
- 当前 channel 信息
- 当前工作目录
- 可用工具清单
- Skills 目录与触发规则
- memory 文件
- 项目上下文
- 运行环境信息

文章里说，上下文工程不是“塞得越多越好”，而是要找到**最小但高信号**的信息集。

OpenClaw 现在已经部分具备这个能力，但也正因为如此，它会天然面临文章里提到的一个大挑战：

> 上下文腐烂

也就是说，OpenClaw 越强，越容易遇到：
- 规则越来越多
- memory 越来越长
- skills 越来越密
- 项目背景越来越重

这会带来两个直接问题：
1. 有价值的信息被埋在上下文中间
2. 模型越来越“知道很多”，但不一定更聚焦

所以文章这部分对 OpenClaw 的启发非常直接：
- 要持续做上下文压缩
- 要做按需加载，而不是全量注入
- 要让 skill 成为可选模块，而不是固定噪声
- 要把项目上下文变成检索式装载，而不是每轮硬塞

### 5. 状态管理：OpenClaw 其实已经有“运行中状态”了

普通聊天产品没有真正的状态管理，只有消息记录。

但 OpenClaw 不一样。它已经开始处理：
- 后台 exec 进程
- process 轮询与恢复
- 子 Agent 生命周期
- 跨 session 的消息路由
- 当前工作目录与 repo 上下文
- 长任务中的中间产物

这意味着 OpenClaw 的状态不只存在于“对话框历史”里，还存在于：
- 文件
- 进程
- session
- background task
- 外部服务

这正是 Harness 和简单 Chat UI 的根本差异之一。

### 6. 错误处理：OpenClaw 已经比很多 Agent Demo 更接近生产

文章里强调，生产级 Harness 必须能正确处理错误。

OpenClaw 目前已经体现出几个重要方向：
- 工具调用失败会显式返回
- 权限受限时会要求审批
- exec 可以后台继续并通过 process 管理
- 子 Agent 可以隔离执行，降低主线程污染
- 失败案例可以沉淀到 memory / skill

这意味着错误不再只是“一次回答失败”，而是有机会变成：
- 可恢复的中间状态
- 可被记录的经验
- 可被固化成 Skill 的改进

这一点很像你现在在做的 HL（Heuristic Learning）路线：
**不是指望模型永不犯错，而是让系统对错误越来越有记忆。**

### 7. 安全与权限：OpenClaw 已经在做“模型建议，系统裁决”

文章里一个我很认同的点是：

> 模型负责提出动作，系统负责决定能不能执行。

OpenClaw 的权限模型已经体现出这个思路：
- 有些命令可直接执行
- 有些命令需要 ask / approval
- 外部动作要先确认
- 敏感操作要遵守 workspace 规则

这就把“行动能力”和“权限边界”分开了。

这类设计很重要，因为越往后，模型越像一个积极的执行者；如果没有权限系统兜住，系统就会很危险。

### 8. 验证循环：这是 OpenClaw 最值得继续强化的方向

文章里说，验证循环是 demo 和生产系统的分水岭。

我觉得这句话放在 OpenClaw 身上尤其准。

OpenClaw 现在已经有验证的基础：
- 改代码后可以跑测试
- 查问题后可以读日志、再追证据
- 生成文档后可以落文件检查
- 背景任务可以轮询确认

但如果要更像一个强 Agent Harness，还可以继续强化：
- 默认要求“最小可验证闭环”
- 每个 Skill 附带验证步骤
- 把失败自动回写到经验库
- 对长任务形成“结果 + 证据 + 风险”的结构化输出

你其实已经在往这个方向走了：
- 周期性经验沉淀
- skill failures 目录
- 决策日志
- dry-run 思路

这几件事，本质上都是在给 OpenClaw 增加验证闭环。

### 9. 子 Agent：OpenClaw 已经触到多 Agent Harness 的门槛了

文章提醒，多 Agent 很强，但不是默认答案。

OpenClaw 的 `sessions_spawn`、`sessions_send`、`subagents` 很有代表性：
- 可以开干净上下文的子 Agent
- 可以让子 Agent 专门做一个探索性任务
- 可以保留主线程聚焦当前目标

这类能力说明 OpenClaw 已经不只是“单体助手”，而是在接近一个可调度的 Agent 运行平台。

但文章的提醒也很适合现在的 OpenClaw：
- 多 Agent 会增加调度成本
- 会丢上下文
- 会增加结果汇总负担

所以一个很稳的原则是：
**先把单 Agent Harness 做厚做稳，再把多 Agent 当成增量能力，而不是默认路径。**

## 从 OpenClaw 往前看：真正该优化的，不只是模型

如果接受这篇文章的视角，那么 OpenClaw 的优化重点就会发生变化。

很多团队一遇到问题，第一反应是：
- 换更强模型
- 加更多工具
- 写更长 prompt

但从 Harness 视角看，更关键的问题常常是：
- 上下文是不是太脏了
- skill 是不是太散了
- memory 有没有真正可复用
- 工具权限是不是太粗
- 验证闭环是不是不够
- 失败有没有沉淀成系统经验

这也是为什么你最近在做的几件事，其实都很对路：
- memory 化
- skill 化
- 周回顾
- failures 沉淀
- 决策日志
- 主动 heartbeat 工作流

这些动作看上去不像“增强模型”，但实际上是在增强 Harness，而 Harness 才是系统复利的主体。

## OpenClaw 和 Claude Code / Codex 的差异，可以怎么理解

文章里提到，Claude Code、Codex、LangGraph 这些系统，本质上都是在做不同风格的 Harness。

如果硬要概括风格差异，我会这样理解：

### Claude Code 风格
- 偏薄 Harness
- 强依赖模型本身能力
- 强调工具调用、验证、收集-行动-验证闭环

### Codex 风格
- 偏统一运行时
- 多前端共享一个 Harness
- 比较强调 app server 和 runtime 的一致性

### LangGraph 风格
- 偏显式状态机
- 开发者可控性强
- 适合需要可视化和确定性流程的场景

### OpenClaw 风格
- 更像一个**把聊天、工具、记忆、技能、外部系统、跨会话调度、主动任务**揉在一起的日常操作系统
- 它不只是 coding agent
- 也不只是 workflow engine
- 更像一个个人 / 团队级的 Agent OS 雏形

这也是 OpenClaw 很有意思的地方：
它不是单纯模仿 Claude Code，而是在往“第二大脑 + Agent 运行时 + 工具编排平台”混合体演进。

## 对你这套体系的直接启发

如果把这篇文章套到你现在的系统，我觉得最重要的不是抽象概念，而是 5 个非常具体的判断。

### 1. 你在做的不是“会聊天的 AI”，而是可复利的 Harness

真正值钱的，不是单次回答质量，而是：
- 是否能沉淀
- 是否能复用
- 是否能减少未来重复劳动
- 是否能让下一次更强

### 2. memory 和 skill 不是附件，而是 Harness 的主结构

很多人把 memory、skills、AGENTS.md 当成提示词补丁。

其实从 Harness 视角看，它们才是系统主体的一部分：
- memory 决定连续性
- skill 决定复用性
- AGENTS / SOUL / USER 决定行为边界与人格一致性

### 3. 长期竞争力不在“工具多”，而在“上下文干净 + 闭环完整”

这点特别重要。
未来真正拉开差距的，不是谁接了 100 个工具，而是谁能做到：
- 该加载什么就加载什么
- 不该带的上下文不带
- 每次输出都尽量有验证
- 失败能沉淀，成功能复用

### 4. Heartbeat、cron、subagent 这些能力，不是花活，是 Harness 的运行机制

这些东西如果只是“功能清单”，价值不大。
但如果把它们看成 Harness 的器官，就会明白：
- Heartbeat = 主动性机制
- cron = 精确定时执行机制
- subagent = 并行探索与任务隔离机制
- memory = 持久化认知层
- skill = 复用工作流层

一旦这样理解，系统就不再像一个 bot，而更像一个操作系统。

### 5. 未来真正该做的是“让 Harness 变薄但更稳”

文章最后一个观点我很喜欢：
随着模型越来越强，Harness 应该逐步变薄，但不会消失。

这对 OpenClaw 也成立。
不是说以后不需要 skill、memory、验证，而是说：
- 重的部分应该逐渐模块化
- 重复规则应该压缩
- 低价值上下文应该淘汰
- 高价值结构应该保留

理想状态不是把系统越堆越厚，
而是把它做成：
**薄路由、厚技能、厚数据、强验证。**

这和你现在认同的 Garry Tan 式复利思路，其实是同一个方向。

---

## 最后一句话

如果只看表面，OpenClaw 像一个带很多工具的聊天助手。

但从 Agent Harness 的视角看，它更像是：

> 一个把模型、工具、记忆、规则、任务、外部系统和主动工作流编织在一起的个人/团队级 Agent OS。

模型当然重要，但真正让系统越来越强的，不只是模型升级，而是 Harness 的持续演化。

这也是为什么：
**会不会“用模型”，和会不会“做系统”，最终是两件事。**

OpenClaw 真正有潜力的地方，恰恰在后者。
