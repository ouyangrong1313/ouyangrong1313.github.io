# Harness 不是目的，知识才是护城河：一个 AI 工程交付团队的知识沉淀实践

## 基本信息
- 来源：微信公众号（腾讯技术工程）
- 原文链接：https://mp.weixin.qq.com/s/JV4-oPP0jjsBCZ4tW3Gy1g
- 作者：stevenpxiao
- 主题：讨论 Harness Engineering 落地时，为什么工作流和 Agent 编排不是终点，真正的长期护城河是团队知识的持续沉淀与复用。

## 一句话总结
这篇文章的核心观点非常明确：**Harness 很重要，但 Harness 只是管道；真正能形成长期复利的，是团队私域知识库。**

也就是说，工作流、Agent、Skill、工具链都可能随着模型演进而被重写，但一个团队在真实业务里积累下来的架构决策、踩坑记录、领域模型、流程规则和最佳实践，才是不会轻易贬值的资产。

## 文章的核心论点

### 1. 工作流是手段，知识才是目的
文章反复强调一个判断：

> 工作流只是管道，知识才是流过管道的活水。

这句话很值得记住。因为很多团队在做 Agent 系统时，注意力容易全放在：
- 状态机怎么设计
- 多 Agent 怎么分工
- 工作流怎么编排
- 模型怎么切换

但这些更像“基础设施设计”。
真正决定未来竞争力的，不是你的工作流是不是 16 阶段，
而是：**你每跑完一次工作流，系统有没有变得更聪明。**

### 2. 没有知识闭环的工作流，只是昂贵的一次性自动化
文章指出一个非常典型的反模式：
- 团队花很多时间搭复杂工作流
- 每个需求都能自动跑一遍
- 但每次还是从零开始
- 上一次踩过的坑，下一次继续踩
- 上一次做过的架构决策，下一次重新推导

这说明工作流虽然“自动化”了，但没有形成知识闭环。

换句话说：
**如果系统不能把结果沉淀为可复用知识，它就只是重复劳动的加速器。**

### 3. 知识才是真正的复利资产
文章把知识分成三类：
- 散点型知识：孤立事实
- 因果型知识：原因与结果之间的推理链
- 时空型知识：只在特定业务背景、时段、条件下成立的经验

越往后两类，越依赖团队真实实践，越不是通用模型能稳定给出的内容。

这也是文章的一个底层判断：
**模型会越来越强，但模型不能替代你的业务知识。**

## Harness Engineering 的三个支柱
文章把 Harness Engineering 总结为三个支柱：

1. **上下文工程（Context Engineering）**
   - 长期 / 短期记忆
   - 知识检索注入
   - 渐进式披露
   - 上下文防火墙

2. **架构约束（Architecture）**
   - Agent 编排模式
   - 状态机设计
   - 降级策略
   - 安全边界

3. **持续治理（Governance）**
   - 质量门禁
   - 知识生命周期
   - 自动衰减
   - 持续进化

作者的关键提醒是：
**知识管理本身不是 Harness 的附属模块，而是 Harness 的核心能力之一。**

## 这篇文章最有价值的部分：知识分层架构

文章最有落地价值的内容，不是理论定义，而是它提出了一套很完整的知识体系设计。

### 一、三个维度：存储层 × 类型 × 成熟度
它把知识库设计成一个三维结构：

#### 1）存储层：知识存在哪里
- Layer 0-P：个人偏好
- Layer 0-T：团队约定
- Layer 1：技术知识
- Layer 2：业务知识
- Layer 3：项目知识

这个分层很实用，因为它解决了“共享边界”问题：
- 有些知识只属于个人
- 有些知识适合整个团队
- 有些知识跨项目通用
- 有些知识只在当前项目有效

#### 2）知识类型：知识描述的是什么
文章定义了 5 种类型：
- `model`：实体、结构、关系定义
- `decision`：技术选型、架构决策和理由
- `guideline`：推荐做法或禁止做法
- `pitfall`：已知风险、故障模式、排查经验
- `process`：业务流程、状态机、操作步骤

这个分类很强，因为它能逼着知识条目更清晰，不再把所有东西都写成一篇杂文。

#### 3）成熟度：知识有多可信
- `draft`：刚提取出来，单一来源
- `verified`：在单项目里验证过
- `proven`：被多个项目验证过

这一步特别关键，因为它解决了知识库常见的一个问题：
**知识不是非黑即白，它有置信度。**

### 二、知识自动衰减：防止“过时经验”污染系统
文章提出了一个很好的机制：
- `proven` 长期不被引用，降级为 `verified`
- `verified` 长期不被引用，降级为 `draft`
- `draft` 持续不用，归档退出活跃索引

这点非常重要。
因为知识库最大的风险不是“不够多”，而是“旧知识还假装自己有效”。

所以作者的观点我很认同：
**知识不是写进去就永远有效，知识也要有生命周期治理。**

## 工作流如何真正服务于知识沉淀
文章有一个很强的落地设计：
把工作流拆成“知识注入 → 执行消费 → 归档提取”三个阶段。

### 1. INIT：知识注入
工作流一开始先拉取团队知识仓库，把知识全景目录注入给 Agent。

这意味着每次新任务启动时，都不是从零开始，而是自动站在前人的经验上。

### 2. 各阶段执行：按需消费知识
不同 Agent 在不同阶段查询不同知识：
- 需求分析查业务知识
- 技术分析查决策和反模式
- 架构设计查模型和历史决策
- 实现阶段查 guideline 和 pitfall

这很像“按角色分配阅读预算”，而不是把所有知识一次性塞给模型。

### 3. ARCHIVE：知识提取与提升
流程结束后，把全过程中的：
- 架构决策
- 失败经验
- 推荐做法
- 业务规则

自动整理成结构化知识条目，并判断是否应该从项目知识提升到团队技术知识或业务知识。

这个设计的妙处在于：
**工作流不再只是交付需求，而是在顺便生产知识。**

## 按需消费知识：这比“全量喂上下文”高明得多
文章很强调一个点：
不要把知识库全量推给 Agent，而要让 Agent 主动查询。

他们设计了三级索引：

### Layer A：全景目录
先告诉 Agent“知识库大致有什么”。

### Layer B：分类清单
再让 Agent 看某个类别下有哪些条目，每条一行摘要。

### Layer C：完整条目
只有在需要时，才读单条知识的完整内容。

这个思路本质上是在解决一个很现实的问题：
**上下文是稀缺资源。**

相比“上来就塞 50 条知识全文”，这种分层索引方式更接近一个真正成熟的 Harness：
- 先定位
- 再筛选
- 最后按需深入

这和 Karpathy 提到的 LLM Wiki Pattern 很接近，也和很多高质量 AI Coding 系统的做法一致。

## 人机协作瓶颈：文章补上了一个容易被忽视的现实问题
这篇文章有个挺新鲜的点，不只是讲知识和工作流，还专门讲了一个工程现实：

> 很多工作流不是卡在 Agent 不会做，而是卡在人不在场。

比如：
- 需要人工审批时，人正在开会
- 需要确认架构方案时，人不在电脑前
- 需要 review 产物时，刚好人在路上

如果没有跨设备接管和异步审批能力，再漂亮的工作流也会因为“等人”而失速。

作者提出的解决思路包括：
- 远程操控
- 跨设备接管
- 文件系统即状态机
- 持久化阶段产物
- 异步审批与通知触达

这部分给我的感觉是：
**他们已经不再把 Harness 当成单机命令行工具，而是在把它当成一个真正的团队协作系统来设计。**

## 这篇文章对你的体系特别有价值的地方
如果把这篇文章和你现在在做的 OpenClaw / HL / 第二大脑体系放在一起看，会发现它们非常契合。

### 1. 它证明了“厚数据”路线是对的
你现在强调的：
- `memory/`
- `MEMORY.md`
- 实体知识
- 技能沉淀
- 失败案例沉淀

本质上都属于这篇文章说的“知识资产化”。

它和“只升级模型、只换工具”不是一个维度的竞争。
它做的是：**把团队经验外化成可复用结构。**

### 2. 它强化了“Skill 不是目的，知识沉淀才是目的”
这篇文章很像是在提醒：
- Skill 只是操作封装
- Workflow 只是执行路径
- Agent 只是调度形态
- 真正要沉淀的是：为什么这么做、踩过什么坑、什么情况下有效

这点特别适合拿来约束以后做 Skill 的方向。

### 3. 它把“知识生命周期治理”讲得很实在
不是所有知识都应该永远保留在活跃区。
过时知识、低引用知识、冲突知识、孤儿知识，都应该被治理。

这对个人知识库和团队知识库都很重要。
否则系统越用越大，最后不是越来越聪明，而是越来越吵。

### 4. 它让“工作流复利”这个概念更具体了
所谓复利，不只是：
- 下次更快
- 下次少踩坑

而是：
- 知识被提取
- 知识被验证
- 知识被提升
- 知识被引用
- 知识被衰减或归档

这是一整套闭环，而不只是“留个总结”。

## 我自己的理解
我觉得这篇文章最有价值的，不是提出了一个全新的理论，而是把一个非常容易被忽略的事实说透了：

> 大多数 Agent 系统最终比拼的，不是 orchestration 花不花，而是知识有没有沉淀成资产。

如果站在更长期的视角，团队真正要建设的不是“最复杂的 Harness”，而是：
- 一个能持续吸收经验的知识底座
- 一个能把项目经验提升为团队资产的机制
- 一个能让未来每次工作都自动受益于过去的系统

这才是护城河。

## 可复用的稳定观点
1. **Harness 是必要的基础设施，但不是最终目的。**
2. **没有知识闭环的工作流，只是昂贵的一次性自动化。**
3. **知识库设计必须同时考虑存储层、知识类型和成熟度。**
4. **知识需要生命周期治理，不能只积累不衰减。**
5. **真正优秀的 AI 工程系统，不只是会执行，还会沉淀。**
6. **团队长期竞争力不在模型本身，而在被验证过的领域知识。**
7. **工作流的价值，不只是交付任务，更是顺手生产团队资产。**
