# Agent Harness 解析：智能体架构深度拆解

## 基本信息
- 来源：微信公众号（AI寒武纪）
- 原文链接：https://mp.weixin.qq.com/s/H8_U4vENXJuiojXtXbCF5w
- 主题：解释什么是 Agent Harness，以及为什么它是生产级智能体系统的核心。

## 一句话总结
文章的核心观点是：**真正决定 Agent 上限的，往往不是模型本身，而是包裹模型的 Harness（脚手架/运行基础设施）**。Harness 负责把一个无状态的 LLM 变成可持续工作、可调用工具、可持久记忆、可恢复、可验证、可控风险的生产级智能体。

## 核心概念
### 1. 什么是 Agent Harness
Harness 是围绕 LLM 的整套软件基础设施，典型包括：
- 编排循环
- 工具调用
- 记忆系统
- 上下文管理
- 状态持久化
- 错误处理
- 安全护栏
- 验证与恢复机制

文中一个很形象的比喻：
- 裸 LLM 像 CPU
- 上下文窗口像内存
- 外部存储像硬盘
- 工具像设备驱动
- Harness 像操作系统

### 2. Agent 和 Harness 的区别
- **Agent**：最终表现出来的“有目标、会用工具、能纠错”的行为体
- **Harness**：产生这种行为的底层机器和运行机制

换句话说，很多人说“我做了个 Agent”，其实更准确地说，是“我做了一个 Harness，并让它驱动某个模型”。

## 三个工程层级
文中把围绕模型的工程分成三层：
1. **提示词工程**：设计模型收到的指令
2. **上下文工程**：决定模型什么时候看到什么
3. **Harness 工程**：在前两者基础上，再补齐工具、状态、记忆、错误恢复、安全、生命周期管理

结论是：**Harness 不是提示词外面再套一层壳，而是完整的应用运行时。**

## 生产级 Harness 的关键组件
文章标题写“12 个组件”，但正文实际展开了 11 个主要模块：

### 1）编排循环
本质上是 TAO / ReAct 循环：
思考 → 行动 → 观察 → 再思考。

看上去只是个 `while` 循环，但真正难的是循环管理：
- 什么时候停
- 什么时候重试
- 什么时候切子任务
- 什么时候做压缩

### 2）工具系统
工具不是简单“给模型几个函数”，而是要完整处理：
- schema 定义
- 注册与暴露
- 参数校验
- 沙盒执行
- 结果捕获
- 返回给模型的可读格式

### 3）记忆系统
记忆分层非常关键：
- 短期记忆：当前会话历史
- 长期记忆：跨会话可持久化内容
- 分层记忆：轻量索引 + 按需展开 + 原始档案

文中提到一个重要原则：**记忆不是事实本体，而是提示线索，行动前仍要回查真实状态。**

### 4）上下文管理
这是最容易悄悄失效的部分。
核心风险是“上下文腐烂”：
- 关键信息掉在中间位置后，模型表现明显下降
- 窗口再大，也会随着上下文变脏而退化

常见策略：
- 摘要压缩
- 屏蔽旧观察结果
- 即时检索
- 子 Agent 探索后只返回压缩摘要

### 5）提示词构建
不是简单拼字符串，而是做优先级编排：
- 系统提示
- 工具定义
- 项目记忆
- 用户指令
- 对话历史

### 6）输出解析
现代 Harness 越来越依赖原生工具调用，不再靠脆弱的自由文本解析。
关键判断只有两个：
- 有工具调用 → 继续循环
- 没工具调用 → 输出最终答案

### 7）状态管理
需要支持：
- 持久化
- 中断恢复
- 检查点
- 调试回放
- 长任务续跑

文中提到的做法包括：session、checkpoint、git 提交、进度文件等。

### 8）错误处理
这是生产化的分水岭之一。
哪怕单步成功率 99%，10 步任务整体成功率也会明显下降。
所以要把错误分类：
- 瞬时错误 → 自动重试
- 模型可恢复错误 → 返回给模型自修
- 用户可修复错误 → 中断并请求人工
- 意外错误 → 直接上抛调试

### 9）安全护栏
文章强调一个重要分层：
- 模型负责“建议做什么”
- 工具系统负责“是否允许做”

所以权限校验不能交给模型自己“自觉”。

### 10）验证循环
这是把 demo 变成生产系统的关键。
验证可以来自：
- 测试 / lint / 类型检查
- UI 截图等视觉反馈
- 独立评判 Agent

文中给出的结论很硬：**让模型具备验证自己结果的通路，质量能提升 2~3 倍。**

### 11）子 Agent 编排
当任务规模变大，就需要：
- fork 子上下文
- 专家 Agent 作为工具
- 任务移交
- worktree / 隔离执行环境

但文章也提醒：多 Agent 不是越多越好，它本身有成本。

## 一个完整 Harness 循环长什么样
文章把运行流程拆成 7 步：
1. 组装提示词
2. 调模型
3. 判断输出类型
4. 执行工具
5. 打包结果
6. 更新上下文
7. 继续下一轮直到终止

终止条件通常包括：
- 模型不再发工具调用
- 达到轮次上限
- token 预算耗尽
- 命中安全护栏
- 用户中断

## 主流框架的实现方式
文章用几个主流体系做对照：
- **Anthropic / Claude Code**：偏薄 Harness，强调“笨循环 + 模型负责智能”
- **OpenAI Agents / Codex**：代码优先，Runner 驱动，多个前端共享一个 Harness
- **LangGraph**：把 Harness 显式建模为状态图
- **CrewAI**：角色化多 Agent 协作
- **AutoGen / Microsoft Agent Framework**：对话驱动编排，多种协作模式

## 七个关键架构决策
文章最后给出每个 Harness 都绕不开的 7 个选择：
1. 单 Agent 还是多 Agent
2. ReAct 还是计划-执行
3. 上下文窗口怎么管
4. 验证循环怎么设计
5. 权限与安全怎么做
6. 工具范围暴露多少
7. Harness 做厚还是做薄

其中几个很值得记：
- **先把单 Agent 做到极致，再拆多 Agent**
- **工具不是越多越好，最小必要集通常更优**
- **模型越来越强，但 Harness 不会消失，只会变薄**

## 我自己的理解
这篇文章对做 AI Coding / Agent 落地特别有价值，因为它把一个常见误区讲透了：

> 不是“模型不够强”，而是“模型周围的运行时系统太弱”。

如果拿 OpenClaw、Claude Code、Codex、Cursor 这些工具来理解，会非常贴切：
- 同一个模型，在不同产品里效果差很多
- 差距往往不是 Prompt 本身
- 而是上下文装载、工具暴露、状态恢复、验证闭环、权限控制这些 Harness 设计

## 对知识库的启发
这篇文章适合挂到 AI Agent / AI Coding / 系统设计几个主题下，尤其适合以后反复引用这些判断：
- Agent ≠ 模型
- Agent 的本质很大程度上是 Harness
- 生产级智能体 = 模型能力 × Harness 设计 × 验证闭环
- 上下文管理和验证机制比“多堆几个工具”更重要
- 多 Agent 是后期优化，不是起步默认答案

## 可沉淀的稳定观点
1. **Harness 是智能体产品真正的工程内核。**
2. **上下文管理是 Agent 稳定性的第一现场。**
3. **没有验证闭环的 Agent，基本只能算演示系统。**
4. **权限控制必须在工具层做，而不是相信模型自律。**
5. **随着模型进化，Harness 会变薄，但不会消失。**
