## 核心问题

如何把 AI 协作从对话式编码，升级为可控、可审计、可复用的工程过程？

---

## 一、从 "Vibe Coding" 到 Harness Engineering

### Vibe Coding 的三个结构性缺陷

| 维度 | vibe coding 典型表现 | 生产级工程要求 |
|------|---------------------|---------------|
| 信息损耗 | 同一句话多次执行给出不同实现；AI 按自己理解"猜"需求 | 需求→设计→代码每步都要有显式产出和可追溯关系 |
| 知识孤岛 | AI 只知训练语料里的通用知识，不懂团队历史决策和私有约束 | 团队知识需要被持久化为 AI 可消费、可演进的工程制品 |
| 验证断档 | "能跑"就直接提交，概率性错误顺着 MR 滑进主干 | 每个关键节点都要有可机读的质量门禁和审计记录 |

### 核心矛盾
- **生成快、验证慢、错误累积** —— 这是 AI 时代软件工程的核心矛盾
- 解决这个矛盾，靠的不是更长、更复杂的 prompt，而是工程化

### Harness Engineering 核心理念
AI 参与问题分析、方案设计、编码实现、审查和验证，但**最终判断权始终留在工程师手中**。

## 二、核心公式：代码产出 = AI 能力 × 上下文质量

### 为什么是乘法，不是加法？
- 当上下文质量趋近于零时，模型再强，产出也是零
- 瓶颈不在模型，而在上下文

### 真实业务仓里的上下文缺口（五类）

| 缺口类型 | 典型问题 | AI 的"盲区" |
|----------|----------|-------------|
| 隐性规范 | 团队约定的锁机制、埋点规则、错误码空间 | AI 不知道这些规范存在 |
| 历史决策 | "为什么当时选了 A 方案不选 B" | 训练语料里没有团队内部决策记录 |
| 服务契约 | IDL 字段的冻结状态、下游是否强依赖 | AI 看到的是文本，不理解哪些字段动不得 |
| 跨服务依赖 | 同一个需求要改哪几个服务、谁调谁 | AI 缺乏全局视角 |
| 演进轨迹 | 某个模块上次大改的坑、灰度策略 | AI 没有跨会话记忆 |

### 工程化的杠杆点：系统性提升上下文质量

- 隐性规范 → 写进 `context/team/`，所有 AI 会话自动继承
- 历史决策 → 沉淀到 `context/project/{module}/experience/`
- 服务契约 → 编码进 `.service-matrix/dependencies.yaml` 和 IDL 门禁
- 跨服务依赖 → 由服务矩阵自动解析，影响面分析从"搜索"变成"查表"
- 演进轨迹 → 通过 Self-Refinement 闭环，让每次纠错都沉淀为团队资产

## 四、五阶段 + 四门禁

### 五阶段流程

```
阶段1 初始化 → 阶段2 需求定义⭐ → 阶段3 设计⭐ → 阶段4 开发⭐⭐ → 阶段5 交付
                  (需求评审门禁)    (设计门禁)    (Dev门禁 + 服务仓库检查门禁)
```

### 四道强制门禁

| 门禁 | 位置 | 阻塞条件 |
|------|------|----------|
| 需求评审门禁 | 阶段 2.2 | 需求文档不合格 / 评审未通过 |
| 设计门禁 | 阶段 3.3 | 设计评审未通过 / 追溯链不达标 |
| Dev 进入门禁 | 阶段 4.2 | `tasks/features.json` 缺失或不合法 |
| 服务仓库检查门禁 | 阶段 4.3 | 三仓分支不一致 / 服务仓库未就位 |

### 核心理念：错误代价递增曲线
- ⭐ 门禁设在"代价最低的拐点上"
- 需求错了 → 改文档
- 设计错了 → 改设计文档
- 开发错了 → 改代码+可能需要回滚

## 五、三层知识体系 + 三仓联动

### 三层知识架构

| 层级 | 位置 | 范围 | 典型内容 |
|------|------|------|----------|
| 团队级 | `context/team/` | 所有项目必须遵循 | Git 规范、错误码空间、日志规范 |
| 框架工程级 | `context/harness-framework/` | 所有需求研发必须遵循 | 五阶段流程、门禁规则、文档模板 |
| 服务级 | `context/project/{project}/{module}/` | 特定服务 | 架构图、踩坑经验 |

### 三仓联动

每条 TAPD 单，在三个仓里用完全相同的分支名：
- **Harness 仓**（脑）：规范、知识、需求状态、工具链
- **业务代码仓**（手脚）：代码和测试
- **IDL 契约仓**（神经）：跨服务协议

```
        一条 TAPD 单 T12345
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
Harness仓   业务仓    IDL仓
feature/Base/T12345（完全相同的分支名）
```

## 六、Skill Agent Command 三件套

| 类型 | 定位 | 数量 |
|------|------|------|
| Skill | 可复用的工作流、规范、最佳实践 | 34 |
| Agent | 自主子任务执行者（可调工具、可调 Skill） | 24 |
| Slash Command | 固定入口 + 标准化参数 | 35 |

### 典型 Command
```
/requirement:new        # 新建需求
/requirement:continue    # 恢复上下文
/requirement:next       # 进入下一阶段
/requirement:gate-check  # 门禁自检
/agentic:code-review     # 多维度代码审查
```

## 七、Self-Refinement 闭环

### 闭环流程
1. 用户纠正 AI 某个错误
2. AI 识别：这是"模式性教训"还是"一次性 diff"？
3. 模式性 → AI 主动提议沉淀层级（团队级/框架工程级/服务级）
4. 用户确认
5. 生成 experience 文档 / 更新 Skill / 修订规范
6. 下次同类场景，AI 主动引用

### 核心价值
错误不再"走一次算一次"，而是成为团队资产。
