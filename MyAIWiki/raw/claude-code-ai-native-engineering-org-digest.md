# Claude Code 负责人：AI-native 工程组织怎么跑起来｜digest

## 一句话总结

当 AI 把“写代码”变便宜后，真正昂贵的部分会迁移到验证、评审、跨职能协作和组织流程本身；AI-native 工程组织的核心不是让工程师写得更快，而是重写规划、评审、角色分工和流程治理方式。

## 5 个核心观点

1. **瓶颈迁移**：从“写代码慢”转向“验代码、评代码、跨职能对齐慢”。
2. **规划后移**：路线图和设计文档不再越早越好，而要更靠近真实问题与真实反馈。
3. **争论材料从 PPT 变成 PR**：既然生成多个实现方案已经便宜，技术争论应尽量建立在真实 diff 和调用影响面上。
4. **自动化验证前移**：样式、lint、基础 review、测试补齐可先交给 Claude；安全、法律、信任边界、品牌体验仍要专家把关。
5. **组织结构要减层和去旧流程**：AI-native 不是堆新工具，而是持续清理旧流程、压平层级、把判断权交给离现场最近的人。

## 文章结构拆解

### 1. 瓶颈从写代码移到验代码
- 代码生成速度上来后，团队吞吐量提升，但验证、评审、回归风险和跨职能沟通成了新瓶颈。
- 过去很多规则（如各种 SLA）是围绕旧瓶颈长出来的，在新环境下可能变成新的阻塞。

### 2. 规划像 JIT 编译一样靠近现场
- 半年路线图在 AI 变化速度下很快失真。
- 更有效的做法是让规划更晚、更接近真实用户反馈和真实原型。
- 设计文档没有消失，但不应成为所有开发前置条件。

### 3. 技术争论先跑三个 PR
- 与其在白板上争 API，不如直接生成多种实现方案，看 diff 和影响面。
- 便宜的是实验，不是协作失控；因此团队文化要允许快速试错，也要防止“谁先提交谁赢”。

### 4. 评审交给 Claude，边界交给专家
- Claude 适合做样式、lint、基础 code review、补测试、PR babysitting。
- 安全敏感、法律风险、品牌体验、信任边界仍要人类专家把关。
- 核心不是“全自动”，而是 **trust but verify**。

### 5. 重新定义招人和角色边界
- 更看重产品感、系统能力、判断力，而不是原始手速。
- PM、设计、非工程角色会越来越多地直接 shipping code。
- 工程师也会借助 Claude 补齐原本需要跨角色排队的工作。

### 6. 经理先做 IC，组织尽量扁平
- 经理应先作为 IC 加入，建立产品和代码信用。
- 组织层级越厚，在快速变化环境下的对齐成本越高。
- 团队原则统一，但具体流程应由离现场最近的人自主调整。

### 7. 代码库成为事实来源
- 代码、spec、用户反馈、自动总结流程都应尽量进入可被 Claude 读取和验证的系统。
- 指标上看 ramp-up time、PR cycle time、Claude-assisted commits，但最终还是要看产品质量和用户体验。

### 8. 要定期“杀旧流程”
- 流程不会自己消失，只会不断叠加。
- AI-native 转型的真实动作，往往是取消或缩短那些已经失去存在理由的老流程。

## 对工程团队最有价值的 7 个问题

1. 现在团队真正的瓶颈在哪里？
2. 哪些环节必须由人类专家亲自审核？
3. 哪些评审或验证已经可以先让 Claude 跑一遍？
4. 哪些文档/评审/会议只是历史遗留，并不再服务当前瓶颈？
5. 争论是否还停留在口头，还是已经进入真实原型/PR 对比？
6. 团队是否把“代码仓库 + 规格 + 反馈”作为统一事实来源？
7. 是否给团队明确授权去取消旧流程，而不是只会叠新流程？

## 可直接迁移的组织动作

- 把最烦、最贵、最慢的一条流程挑出来，先改这一条。
- 用真实 PR/原型替代一部分会前争论。
- 把 lint / 样式 / 基础 review / 测试补齐前移给 Claude。
- 只保留少数团队原则，其余流程由小组现场优化。
- 定期做一次“流程清仓”，明确哪些规则已经应该退休。
