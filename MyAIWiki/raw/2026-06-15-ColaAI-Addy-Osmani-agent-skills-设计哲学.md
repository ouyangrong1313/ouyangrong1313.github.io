## 原文

> **导语**:如果你也用 Claude Code、Cursor 写过代码,大概率遇到过这种事:它写得飞快,但跳过了测试、跳过了规格、跳过了安全检查。这篇聊的,就是一个专门治这个毛病、已经拿下 58.9k 星的开源项目。

### AI 写代码,默认会抄近路

你让它实现一个功能,它会用最短的路径给你跑通——然后顺手跳过写规格、跳过补测试、跳过安全评审,跳过所有让软件真正可靠的步骤。能跑,但一上生产就原形毕露。

Chrome 团队的知名工程师 Addy Osmani,把这件事做成了一个开源项目,思路简单粗暴:既然 agent 爱偷懒,那就给它一套资深工程师的工作流,每一步都不许跳。这个叫 agent-skills 的项目,星标涨得飞快——光是我写这篇文章的工夫,就从 4 万多涨到了近 6 万。

> 📦 **addyosmani/agent-skills** —— 给 AI 编程 agent 用的"生产级工程技能包"——把资深工程师的工作流、质量关卡和最佳实践打包,让 agent 每个环节都照着做。

**核心数据**:⭐ 58.9k Stars | 🍴 6.4k Forks | ⚖️ MIT | 🔖 v0.6.0 | 23 技能 / 7 命令 / 8+ 工具支持

### 一、为什么要"给 agent 立规矩"

它解决的不是"AI 写不出代码",而是"AI 写得出代码,但写不出靠谱的代码"。

agent 天生倾向走最短路径,这恰恰是资深工程师和新手最大的区别——什么时候该先写规格、该测什么、怎么评审、什么时候才能上线,这些判断力,agent 默认是没有的。agent-skills 做的,就是把这套判断力变成一步步的流程,塞进 agent 的工作里。

这件事的隐性成本,写过项目的人都懂:当时图快省下的那点时间,后面会以"线上事故""没人敢动的祖传代码""半夜被叫起来修 bug"的方式连本带利还回来。让 agent 在前面多走几步,就是为了在后面少踩几个坑。

> ⚠️ **它不是什么**:不是一堆"你是一个资深工程师"那种泛泛的提示词。它是有步骤、有检查点、有退出条件的结构化工作流,agent 必须照着走完才算数。

### 二、7 个命令,就是一条完整的开发流水线

上手最直接的方式,是它给的 7 个斜杠命令。每个命令对应开发的一个阶段,背后会自动激活对应的技能。你不用记 23 个技能叫什么,记住这 7 个动作就行:

| 你在做什么 | 命令 | 核心原则 |
|---|---|---|
| 定义要做什么 | `/spec` | 先有规格再写代码 |
| 规划怎么做 | `/plan` | 拆成小而独立的任务 |
| 增量构建 | `/build` | 一次只做一小片 |
| 证明它能用 | `/test` | 测试就是证据 |
| 合并前评审 | `/review` | 改善代码健康度 |
| 简化代码 | `/code-simplify` | 清晰胜过聪明 |
| 上生产 | `/ship` | 越快越安全 |

除了手动敲命令,技能也会根据你在干什么自动激活——比如你在设计接口,它会自动拉起 `api-and-interface-design`;你在写界面,`frontend-ui-engineering` 就上场了。

### 三、23 个技能,按开发生命周期排好了队

7 个命令是入口,背后是 23 个技能(22 个生命周期技能 + 1 个元技能)。它们不是散装的,而是顺着"定义→规划→构建→验证→评审→发布"这条线排好的:

- **定义 Define**:`interview-me`(一问一答逼出真需求)、`idea-refine`、`spec-driven-development`(先写 PRD)
- **规划 Plan**:`planning-and-task-breakdown`(把规格拆成可验证的小任务)
- **构建 Build**:这一档最重,**7 个技能** —— 增量实现、测试驱动、上下文工程、源码驱动、怀疑驱动、前端工程、接口设计
- **验证 Verify**:`browser-testing-with-devtools`(接 Chrome DevTools 看运行时数据)、`debugging-and-error-recovery`
- **评审 Review**:代码评审、代码简化、安全加固、性能优化(4 个)
- **发布 Ship**:git 工作流、CI/CD、废弃与迁移、文档与 ADR、发布与上线(5 个)

### 四、真正的杀手锏:4 个设计哲学

技能多不稀奇,难的是怎么让 agent 真的照做。这个项目里我觉得最值钱的,是它背后这 4 个设计选择:

**1 流程,而不是文档** —— 技能是 agent 照着走的工作流,不是它读完就忘的参考资料。每个都有步骤、检查点、退出标准。

**2 反合理化(最妙的一点)** —— 每个技能都内置一张"借口→反驳"表。agent 最爱说"测试我后面再补",这张表直接把这类偷懒的借口和对应的反驳写死,堵死它跳步的退路。

**3 验证不可妥协** —— 每个技能都以"拿证据"收尾——测试通过、构建输出、运行数据。"看起来对"永远不算数。

**4 渐进式披露** —— SKILL.md 是入口,详细的引用文件只在需要时才加载,把 token 占用压到最低。这点对长会话特别友好。

> **核心金句**:"AI agent 默认走最短路径——而最短路径,往往跳过了让软件可靠的所有步骤。" —— agent-skills 的设计出发点

### 五、23 个技能,都长成同一个样

这套东西能保持一致、不散架,靠的是一套统一的骨架。你打开任意一个技能的 SKILL.md,结构都一模一样,固定 7 块:

1. **名称 + 描述** —— 告诉 agent 这个技能是干嘛的、什么时候该用
2. **概述** —— 这个技能解决什么问题
3. **何时使用** —— 明确的触发条件
4. **流程** —— 一步步的工作流,带检查点
5. **反合理化** —— 常见的偷懒借口,以及怎么反驳
6. **危险信号** —— 哪里开始不对劲的征兆
7. **验证** —— 必须拿出来的证据

正是这种近乎死板的模板化,让 23 个技能不会各写各的风格。agent 理解了一个,就理解了全部——这也是为什么它敢说自己是"生产级",而不是一堆零散的提示词拼盘。

### 六、除了技能,还有 3 个专家和 4 张清单

技能之外,项目还配了两样很实用的东西。

**3 个预设专家角色**,专门用来做针对性的评审:

| 角色 | 视角 |
|---|---|
| 代码评审员 | 资深员工级工程师,按"staff 工程师会不会放行"的标准做五维评审 |
| 测试工程师 | QA 视角,看测试策略、覆盖率和"先证明"模式 |
| 安全审计员 | 安全工程师视角,做漏洞检测、威胁建模、OWASP 评估 |

**4 张速查清单**:测试模式、安全检查、性能检查(含 Core Web Vitals 目标)、无障碍检查。需要的时候自动拉进来,平时不占上下文。

### 七、它把哪些 Google 工程智慧打包进去了

Addy 明确说,这套技能借鉴了 Google 的工程文化(《Software Engineering at Google》和它的工程实践指南)。这些不是抽象口号,而是直接嵌进了每一步流程里:

- **Hyrum 定律** —— 用在接口设计
- **Beyoncé 规则 + 测试金字塔(80/15/5)** —— 用在测试
- **变更规模约 100 行 + 评审速度规范** —— 用在代码评审
- **Chesterton 栅栏** —— 用在代码简化
- **主干开发 + 功能开关** —— 用在 git 和 CI/CD
- **把代码当负债** —— 专门有个废弃技能

### 八、怎么用上它

官方最推荐的是 Claude Code,一行命令装好:

```
/plugin marketplace add addyosmani/agent-skills
/plugin install agent-skills@addy-agent-skills
```

> ℹ️ **不止 Claude Code**:技能本质上就是纯 Markdown,所以 Cursor、Gemini CLI、Windsurf、OpenCode、GitHub Copilot、Kiro、Codex 都能用,仓库 docs 目录里每个工具都有对应的接入说明。

### 九、我的判断:它适合谁

**👍 很适合**
- 已经在用 AI 写真实项目的人
- 受够了 agent"能跑就行"的人
- 想给团队定统一开发规范的人

**🤔 暂时不必**
- 只是偶尔让 AI 写写小脚本
- 完全不碰工程化流程的
- 嫌"立规矩"拖慢节奏的

> 说到底,它的价值不在"让 AI 更聪明",而在"让 AI 更靠谱"。当你写的是要上生产、要别人维护的代码,靠谱比聪明值钱得多。
