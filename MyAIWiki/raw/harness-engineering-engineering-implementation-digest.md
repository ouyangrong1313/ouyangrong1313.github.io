# Harness Engineering 拆解：AI工程化落地的从0到1

## 核心观点

**Harness Engineering 的本质：如何让 AI 在你的项目里，持续、稳定、规范、顺畅地做出你真正想要的结果。**

不同项目落地的 Harness 表面完全不一样——有的 CI 很强，有的规范很强，有的多 Agent 很强。但往下挖，解决的都是同一个问题。

---

## 角度一：核心概念澄清（Rule/Skill/Agent/Workflow/Scripts/MCP）

### 核心洞察

六个核心概念，解决六个不同问题：

| 概念 | 问题 | 角色 |
|------|------|------|
| Rule | 什么事绝对不能乱来 | 基础规矩、红线、底线 |
| Skill | 这件事具体应该怎么做 | 标准操作手册 |
| Sub Agent | 复杂任务由谁分工 | 不同阶段专业角色 |
| Workflow | 按什么顺序接力 | 前进、暂停、打回、重跑规则 |
| Scripts | 谁来判断做没做好 | 统一门禁和事后验证 |
| MCP | 怎么接外部工程系统 | 外接能力接口 |

### 开头钩子

1. "你还在把所有 AI 规则都塞进一条系统提示词？难怪它不听话。"

2. "Rule、Skill、Agent、Workflow、Scripts、MCP——这六个词搞不清楚，做 Harness 就是瞎子摸象。"

3. "为什么你的 AI 总是'懂了但没做'？因为你只给了它规则，没给它操作手册。"

---

## 角度二：Rule 的局限——软约束为什么不够

### 核心洞察

Rule 是软约束，不是硬门禁。AI 会出现三种典型情况：
1. 忘了某条 Rule
2. 认为某条 Rule 和需求"无关"
3. 知道 Rule 但偷懒没做

Rule 重要，但不能解决所有问题。

### 开头钩子

1. "你以为给 AI 写完 Rule 就万事大吉了？它下一秒就能忘得一干二净。"

2. "Rule 是软约束，不是硬门禁——这个区别，决定了你的 Harness 能走多远。"

3. "为什么 AI '懂了但没做'？因为你只告诉它什么不能做，没告诉它不做会怎样。"

---

## 角度三：多 Agent 不是设计出来的，是被问题逼出来的

### 核心洞察

多 Agent 的拆分不是设计出来的，而是被问题一步一步逼出来的：

遇到问题 → 拆分新 Agent → 继续遇到问题 → 继续拆分

最终形成的 Agent 数量和职责，是根据实际情况逐步演化的结果。

### 开头钩子

1. "七个 Agent 不是我拍脑袋拆出来的，是被七个真实问题一个一个逼出来的。"

2. "如果你在设计阶段就把所有 Agent 职责定好了，那大概率是你在自我满足，而不是在解决问题。"

3. "多 Agent 的本质不是'分任务'，而是'分角色'——这个区别很多人没想清楚。"

---

## 角度四：三层补稳机制

### 核心洞察

多 Agent 落地后需要三层补稳：

1. **第一层：流程定义文件** —— 把流程结构单独抽出来
2. **第二层：角色契约** —— 把每个 Agent 的输入输出抽成契约
3. **第三层：流程校验脚本** —— 再补一个总验证脚本

### 开头钩子

1. "多 Agent 跑起来了，然后呢？大多数团队死在了'没有第三层'。"

2. "Agent 之间的协作不能只靠'说好了'，要有契约，要有校验，要有脚本。"

3. "你以为定义了 Workflow 就完了？流程跑起来之后，谁来判卷？"

---

## 角度五：Scripts 是最后的兜底

### 核心洞察

**当 Rule 开始管不住、Agent 开始出现随机性时，需要 Scripts 来兜底。**

Scripts 要解决的三类问题：
- A 类：静态规范问题（代码风格、命名）
- B 类：基础交付门槛（编译、测试）
- C 类：工程一致性（构建产物、版本号）

总验证脚本的作用：把"是否完成"从 Agent 的责任变成系统的责任。

### 开头钩子

1. "Rule 管不住的，Agent 靠不住的，最后都要 Scripts 来兜底。"

2. "95%的团队死在'没有判卷人'——AI 做完就完了，没人知道到底做没做成。"

3. "Scripts 不是可选项，是必选项。没有 Scripts，你的 Harness 就是裸奔。"

---

## 角度六：AI 需要"整个项目的记性"

### 核心洞察

AI 不能只依赖当前对话，需要带上"整个项目的记性"。

dev-map + 任务看板：
- dev-map：技术债务、决策记录、历史上下文
- 任务看板：进度、阻塞点、下一步

别把「项目级上下文」理解成堆字，而是"项目级记忆"。

### 开头钩子

1. "你的 AI 只记得这一轮对话，上一轮做的决定它早就忘了——这就是团队级 Harness 最大的坑。"

2. "为什么 AI 总是在重复问同样的问题？因为你没有给它'项目记忆'的机制。"

3. "Memory 在团队级 Harness 里可以靠边站，因为团队的记忆要靠文档和工具，不是靠 AI 的对话上下文。"

---

## 角度七：Harness 全貌——四块拼图

### 核心洞察

Harness 的四块拼图：

1. **Rule + Skill** → 约束与规范
2. **Sub Agent + Workflow** → 执行与协作
3. **Scripts** → 验证与判断
4. **Memory/Context** → 记忆与上下文

四块一起转，才能让 AI 在项目里持续、稳定、规范、顺畅地做出想要的结果。

### 开头钩子

1. "你缺的不是规则，不是 Agent，不是脚本——你缺的是让它们一起转起来的机制。"

2. "Harness Engineering 不是堆概念，是让四个拼图一起转。"

3. "看完这篇文章，你知道 Harness 长什么样了：四块拼图，缺一块都转不起来。"
