# 做出改变世界的Claude Code，Cat Wu揭秘AI时代PM的唯一护城河

---

A厂从过完年开始发布迭代节奏，让整个行业都看呆了。

有人做了一个日历，把Anthropic最近的发布全部标注出来，结果发现：几乎每一天都有重大功能或产品上线。

这背后是谁在推动？

Cat Wu，Claude Code和Cowork的产品负责人，负责Anthropic旗下两款最核心的开发者产品。她刚刚接受了Lenny's Podcast的访谈，谈了很多外界几乎听不到的内幕。

包括：为什么他们能做到这个发布速度？Claude Code的代码为什么会泄露？PM这个职位未来还有价值吗？以及，Anthropic到底靠什么从一个几乎没有胜算的后来者，做到现在这个地步？

原采访自行搜索：
How Anthropic's product team moves faster than anyone else | Cat Wu (Head of Product, Claude Code)

以下是访谈中最核心的内容。

---

## 发布周期从6个月压缩到1天，他们怎么做到的

过去做产品，一个功能从立项到上线，正常周期是6到12个月。

现在在Anthropic，很多功能的周期已经压缩到1个月，有时候是1周，有时候甚至是1天。

Cat说，这不是因为他们用了什么神奇的新模型，而是流程和文化的问题。

具体来说，他们做了三件事：

**第一，把目标定清楚。**
LLM太通用了，这反而带来很多模糊性，容易不知道在给谁做产品、解决什么问题。一个好的PM必须能说清楚：我们的核心用户是谁、这个功能要解决什么具体问题、什么叫做成功。目标清晰了，工程师就不用等PM拍板，可以自己做决定。

**第二，几乎所有功能都以Research Preview形式上线。**
这一招降低了发布门槛。功能出去的时候会明确标注"这是早期版本，我们在收集反馈，不保证长期支持"。不需要把每件事做到完美才发布，可以先出去，再根据反馈迭代。

**第三，工程、市场、文档三个团队之间有一套极度精简的协作流程。**
工程师觉得功能准备好了，就发到一个叫做"evergreen launch room"的频道。然后负责文档的、负责PMM（产品营销经理）的，以及团队的成员，第二天就能交出对应的上线公告。整个摩擦力非常低，低到让每一个工程师都敢于直接把想法推出去。

Cat的原话是：PM的角色就是搭建这套框架，让大家知道什么时候该拉哪个跨职能团队进来，以及他们的预期是什么。

## Claude Code源代码泄露，到底发生了什么

几周前，Claude Code的源代码意外泄露到了外部。

Cat的解释是：这是一次人为失误。有人在用Claude协助写一个PR，内容是关于他们如何发布package的流程更新，这个PR经过了两层人工审核，但还是出了问题。

当时外界有些猜测，觉得这可能是有意为之，或者涉及更严重的安全问题。

Cat说不是的，就是流程上的疏漏。发现之后，他们立刻排查，加固了相关的流程，大部分改进措施已经上线。涉事人员还在公司，因为这是流程失败，不是个人失败，最重要的是学到教训、不再重蹈。

## 为什么关掉了OpenClaw的权限

近期Anthropic限制了第三方工具（比如OpenClaw）通过Claude订阅访问API的方式，引发了不少开源社区的不满。

Cat的解释是：这些第三方产品的使用模式和Anthropic自己的第一方产品不同，对基础设施的消耗方式也不一样。在需求量极大、算力非常紧张的情况下，他们需要优先保证自己的第一方产品和API。

作为过渡方案，所有用户都获得了一定量的credits。

这背后的逻辑其实很简单：Anthropic本质上是在用月费200美元近乎无限地补贴第三方产品的算力使用，这在商业上行不通。

## PM这个职位还有没有前途

Cat在访谈中说，她采访了几百个想进Anthropic做PM的人，发现大多数人的思维方式都不对。

他们还在用旧时代的思路：做多季度路线图、跨团队协调、每个需求都对齐所有相关方。

这套东西在AI时代失效了。

功能周期从6个月压缩到1天，意味着PM的首要任务不再是规划和协调，而是帮助团队更快地把东西发出去。

但这不代表PM不重要。Cat的判断是：所有角色都在融合。工程师在做PM的工作，设计师在写代码，PM也在写代码。Anthropic的策略是招募有产品品味的工程师，让他们从用户反馈直接跑通到发布，中间几乎不需要PM介入。

但有一个东西任何角色都取代不了：产品品味。

当代码越来越便宜，决定写什么反而越来越值钱。面对GitHub上数万条Feature Request，能判断哪些值得做、用什么方式做，这个能力来自任何背景，但极其稀缺。

Cat说，Anthropic几乎会无条件录用任何强烈展示出这种能力的人。

## 最被低估的AI技能：让模型反思自己的错误

被问到如何培养对模型能力的理解时，Cat提到了一个很多人忽视的技巧。

当模型做了一个出乎意料的操作，比如修改了前端代码但没有实际测试UI，不要直接去改系统提示词，而是先问模型：你为什么这么做？

模型的回答往往很有价值。它可能会说，系统提示词里有一段让我困惑的内容；或者，我没意识到验证前端是这个任务的一部分；或者，我把验证交给了子智能体，但没有检查它是否真的执行了。

这种对话揭示了模型被什么误导，才能让你知道该从哪里修改Harness，真正缩小能力差距。

Cat把这个称为自己最被低估的AI使用技巧。

## 为什么构建Evals是最被忽视的工作

大多数 人愿意花时间建新功能，但没有人愿意写评估。

Cat认为这是个严重的误解。没有Evals，你就不知道新功能是否真的有效，不知道新模型是否让产品变好了，也不知道哪里在退步。

在Claude Code里，他们的工程师把代码提交进来时，代码评审流程本身就依赖Claude完成。能信任Claude做代码评审，前提是有一套严格的评估体系，能确认Claude在这个维度上达到了标准。

这种能力不是天生的。写好Evals本身是一门手艺，需要你真正理解产品的目标和失败模式，而不是做一堆看起来像指标但实际上什么都测不准的东西。

## Claude的个性为什么重要

有人可能觉得Claude的性格设定是个软性的、不重要的东西。

Cat不这么认为。

她的解释是：Claude有非常独特的个性，这让它在价值观上有自己的立场，会拒绝做某些事情。这种一致性对于用户的信任是至关重要的。一个你不信任的工具，你不会把重要的工作交给它。

Claude的个性也体现在使用体验上。当用户遇到问题，他们不只是看输出是否正确，还在感知整个对话的质量。一个有一致性格的模型，在用户遇到困难时更容易建立起信任。

## Anthropic如何从落后者变成行业领头羊

访谈里有一段关于Anthropic为什么成功的讨论，非常值得注意。

Cat说了两点。

**第一是使命对齐。**
Anthropic招募的是那些真正相信把安全的AGI带给全人类这个使命的人。这个使命不是装饰，而是直接指导决策。当两个优先级相互冲突时，大家会讨论哪个对Anthropic使命更重要，然后所有人都愿意支持这个结论。

最极端的例子：Cat说如果Claude Code失败了但Anthropic整体成功，她会很开心。整个团队都愿意接受这种取舍。

这解释了为什么Anthropic没有做社交网络、没有做资讯Feed，没有做和主线任务无关的产品，因为这些不符合使命，即使可能赚钱，也不做。

**第二是专注。**
使命提供了一把筛子，把所有可能分散注意力的方向都过滤掉了。这让Anthropic在一个同行都在四处扩张的行业里，保持了极高的执行密度。

## Claude Code、Desktop、Cowork，分别该用哪个

很多用户搞不清这三者的区别。Cat给了一个非常清晰的区分方式。

**Claude Code CLI**：适合需要所有最新功能、同时跑多个编程任务的人。大多数新功能最先在CLI上线，功能最全。

**Claude Code Desktop**：适合做前端工作，可以打开实时Preview，边聊天边看网页应用的变化。也适合不熟悉终端界面的人，有图形化界面，可以看到所有正在运行的任务。

**Cowork**：输出不是代码的任务都应该用Cowork。整理邮件、处理Slack、写文档、做演示PPT，这些都是Cowork的场景。Cat自己的使用习惯是：要产出代码，用Claude Code；产出其他东西，用Cowork。

## Cowork的实际使用案例

Cat分享了一个真实例子。

她需要准备一个关于Claude Code从助手到智能体转型的演讲。她把PMM同事整理的草稿、她自己不满意的旧版本、不应该重叠的主题演讲，全部丢进Cowork，告诉它她想传达的叙事主线。

Cowork接下来工作了一个小时，浏览了推上的发布记录，查看了内部Slack的发布频道，研究了团队成员分享的用例，最后生成了一份20页的ppt。

Cat早上醒来看到这份文稿，评价是：基本上不错，有一些要修改的地方，比如文字偏多，但比她自己做快得多。更关键的是，因为Cowork拿到了Anthropic内部的设计系统，视觉上看起来真的像设计师做的。

要用好Cowork，Cat建议先连接所有相关数据源：这是它能够生成有意义内容的前提。

## 关于自动化：95%不够，必须做到100%

很多人会把某件事自动化到90%或95%的准确率，然后觉得差不多了，就不再投入时间。

Cat的看法恰好相反：95%的自动化几乎等于没有自动化。

一个自动化如果不能100%准确运行，你就不能真正依赖它。你反而需要持续监督它，这本身就消耗了时间。

她建议：找到一件你真正想自动化的事情，把它做到100%。花时间教模型你的偏好，给它反馈，让它把这个反馈固化进技能里。这个过程确实慢，但一旦做成了，你才能真正从这件事上解放出来。

---

## 给所有在AI时代感到焦虑的人

访谈最后，Cat给出了一个具体的建议框架。

注意你每天在重复做什么。这些重复性的工作，才是你应该优先自动化的。把那些不喜欢的、机械性的工作交出去，你才能把时间用在真正有创造力的部分。

同时，不要满足于原型。Cat说她看到很多人用AI快速搭出一个原型，觉得很酷，但从来不再打开它。这样的原型对你没有帮助。要构建那些你每天都在用的东西，用真实使用去感知它真正的能力边界。

关于AI的悲观派和乐观派之间的分歧，Cat有一个判断：2024年以前的AI产品是对话式的，Claude Code这一代的产品是行动式的。当你第一次感受到一个智能体真的替你完成了一件事，而不是告诉你该怎么做，那种体验会让你对AI的判断彻底改变。
