# Deep Research Stack 拆解：开源打败闭源

## 核心观点

**开源 Deep Research 可以打败闭源 SaaS，关键是三件事：阶段分离、推理检索、循环反思。**

用 Onyx + CrewAI + Voxtral 三个开源工具，在 DeepResearch Bench 上排名第一，超越 OpenAI Deep Research、Gemini 2.5 Pro、Perplexity。

---

## 角度一：为什么 self-hosting 值得做

### 核心洞察

闭源云服务的真实后果：
- 查询发送到他们的服务器，暴露你在做什么
- 内部数据在他们基础设施上被索引
- 保留和审计由他们决定
- 配额和定价随时变化

对于受监管行业、IP 敏感工作、数据驻留规则下，self-hosting 不是可选项，是必选项。

### 开头钩子

1. "你发给 ChatGPT 的每一次查询，都在别人的服务器上——这对某些团队来说已经是合规问题。"

2. "为什么你的 AI 研究工具总是'不能用'？因为数据主权不是你决定的，是他们决定的。"

3. "Deep Research 能打败 OpenAI 了，而且完全跑在你自己的基础设施上。"

## 角度二：现有工具为什么失败

### 核心洞察

大多数研究工具做一次搜索→交给 LLM 写。这种方式适用于浅层查询，但一旦涉及：
- 跨源综合
- 矛盾检测
- 多跳推理

就崩溃了。

典型失败：
- Agent 找到矛盾来源，选一个继续，矛盾从未浮出水面
- 两个来源说同样的话，被当作独立证据引用
- 关键词匹配不理解"云迁移"="迁移 PostgreSQL 到 AWS"

**根本原因：研究不是一项任务。**

### 开头钩子

1. "你的研究工具找到矛盾来源后，选一个就跑了——这不叫研究，这叫瞎猜。"

2. "关键词搜索不理解'云迁移'和'迁移数据库到 AWS'是同一件事——这就是为什么你的研究总是漏掉关键信息。"

3. "研究不是一项任务，是五个阶段的接力——大多数工具把它们混在一起做。"

---

## 角度三：好研究需要什么

### 核心洞察

无论工具如何，好的深度研究需要五件事：

1. **阶段分离** — 收集/分析/写作之间有硬墙
2. **推理检索** — 不是关键词，是语义理解和多跳
3. **循环反思** — 静态计划遇到新发现要 pivot
4. **统一搜索** — web + 内部文档一条管道
5. **语音层** — 说比写快，听比读轻松

### 开头钩子

1. "好的 Deep Research = 阶段分离 + 推理检索 + 循环反思 + 统一搜索 + 语音层。这五件事，缺一件都不行。"

2. "为什么你的研究总是'浅'？因为你缺了最关键的一步：循环中反思。"

3. "研究不是一次搜索，是一场有计划的探索——遇到意外时要 pivot，而不是无视。"

## 角度四：Onyx 的三阶段架构

### 核心洞察

Onyx 的研究流程分三阶段：

- **Phase 1: Clarification** — 最多 5 个问题，澄清模糊查询
- **Phase 2: Planning** — 最多 6 个探索方向，Planner 无工具权限，只能出计划
- **Phase 3: Iterative execution** — 最多 8 轮，每轮并行 3 个 Agent

两个关键分离：
- Orchestrator 从不直接搜索
- Research Agent 从不看完整查询/计划

防止上下文泄漏。

### 开头钩子

1. "为什么你的 Planner 总是在'找答案'而不是'做计划'？因为它有工具权限——Onyx 把这个权限关了。"

2. "8 轮迭代，每轮 3 个并行 Agent——这个配置不是拍脑袋的，是被问题逼出来的。"

3. "Deep Research 的 Planner 必须没有工具访问权限，否则它就不是在规划，是在执行。"

## 角度五：6步检索管道

### 核心洞察

每个 Agent 在 LLM 综合之前必须经过 6 步：

1. **Query generation** — 并行多路查询
2. **Search recombination** — 混合索引 + Reciprocal Rank Fusion
3. **LLM selection** — LLM 过滤相关块（跳过这步=幻觉进入）
4. **Context expansion** — 读周围块确定上下文大小
5. **Prompt building** — 组装 + 引用
6. **Answer synthesis** — 带引用的 grounded 答案

### 开头钩子

1. "大多数 RAG 死在'第 3 步'——没有 LLM selection 这一步，幻觉就进来了。"

2. "6 步检索，每步都有存在理由，缺一步效果就打折扣。"

3. "为什么你的 RAG 总是不准？因为你跳过了最关键的一步：让 LLM 自己判断哪些块相关。"

## 角度六：CrewAI 的 Flow 编排

### 核心洞察

CrewAI 三个原语解决阶段分离：
- **Flows** — 独立 Crews 接力，只传递干净输出
- **Skills** — SKILL.md 在行动点注入指令
- **MCP Integration** — 直接把 MCP 服务器附加到 Agent

**不要做的事**：一个 Crew 三个顺序任务共享上下文 = "deep frying"（深度油炸）：
- 事实被反复解读变味
- 矛盾被平滑掉
- 源材料到最后一层已经面目全非

### 开头钩子

1. "三个顺序任务共享上下文，这是 Deep Research 的头号杀手——CrewAI 叫它'deep frying'。"

2. "CrewAI 的 Flow 不是工作流，是接力赛——每棒只传干净的 baton，不传累积的上下文。"

3. "你想让研究结果不被'油炸'？CrewAI 的解法：三个 mini-crews，不是三个顺序任务。"

## 角度七：开源 vs 闭源的差距

### 核心洞察

开源系统可以打败闭源：
- Onyx 在基准测试上 #1
- 完全可自托管、可审计、可修改
- 数据永远不离开你的基础设施

三件事你同时得到：
- **能力**：有竞争力或更好的研究质量，引用完整性
- **控制**：每个字节都在你自己的基础设施上
- **透明**：开源代码可读、可审计、可扩展

### 开头钩子

1. "开源 Deep Research 打败了 OpenAI——而且跑在你自己的服务器上。"

2. "如果数据主权不是约束，你的团队的研究工作流会是什么样？从这个问题开始。"

3. "闭源工具给你的：便利 + 失控。开源给你的：能力 + 控制 + 透明。你选哪个？"
