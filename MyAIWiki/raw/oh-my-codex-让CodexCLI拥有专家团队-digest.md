# oh-my-codex：让 Codex CLI 拥有 30 个专家团队 — 拆解

## 核心观点（5个）

1. **oh-my-codex 是 Codex CLI 的多智能体编排层** — 不做新模型、不造新协议，只给 AI 装上"工程化管理"
2. **单 agent 的极限不在模型能力，而在工程化组织** — OMX 补上了记忆、分工、验证、协作缺失的短板
3. **30 个 agent 组成"数字工程队"** — 构建组、评审组、专家组的分工体系，特别是 critic 角色专门唱反调
4. **39 个工作流技能 = 开箱即用的"战术手册"** — 从 $autopilot 到 $ultraqa，覆盖规划到验证的全流程
5. **关键设计：pure add-on** — 基于 Codex CLI 原生扩展点，不 fork，不锁死，永远跟官方主线走

---

## 七大分析角度

### 1. 产品设计视角
"致敬 oh-my-zsh" 的命名策略很聪明——降低用户认知成本，让已知变体。核心逻辑：不是加功能，而是补工程化短板。

### 2. 多 Agent 协作视角
30 个 agent 分工：构建组（7）、评审组（6）、专家组、产口组+协调组。特别值得学习的是 critic 角色——专门挑刺，对抗 AI"讨好用户"的本能。

### 3. 工程化视角
6 个扩展点用至极：AGENTS.md、prompts、skills、config.toml、.omx/。把 Codex CLI 当执行引擎，"纪律"装在外面。

### 4. 技能编排视角
39 个 skill 分执行模式（8种）、规划、其他、魔法关键词。关键洞察：`$ralph` 持续循环 + 架构师把关，是质量保障的简洁有效方案。

### 5. 用户分层视角
明确适合/不适合人群：适合有 Codex CLI 使用经验、需中大型项目协作；不适合纯 Windows 用户或只想"裸用"的人。

### 6. 安全设计视角
`omx --madmax` 危险选项的警告很实在——绕过所有审批和沙箱，要在隔离环境用。这是给高级用户的"双刃剑"提示。

### 7. AI 编码工具竞争视角
核心论点：AI 编码工具的竞争点已经不是"模型多聪明"，而是"能不能把 AI 组织好"。这是对整个赛道的位置判断。

## 开头钩子（14个）

### 角度1：痛点共鸣型
1. "你是不是也遇到过这种情况？用 Codex CLI 写代码，单个 agent 一个 prompt，复杂任务要不停切换上下文，没有记忆、没有分工、没有验证……"
2. "AI 写得飞快，你 debug 更快。"

### 角度2：产品定位型
3. "oh-my-codex（简称 OMX）是 OpenAI Codex CLI 的多智能体编排层。"
4. "名字致敬了 oh-my-zsh —— 就像 oh-my-zsh 让你的终端脱胎换骨，OMX 要做的，是让 Codex CLI 从『单打独斗的 AI』变成『有纪律的作战团队』。"

### 角度3：对比冲击型
5. "装上 OMX 前后，差距有多大？单 agent 单 prompt → 30 个角色专家。"
6. "把 Codex CLI 当成执行引擎，把'纪律'装在外面。"

### 角度4：设计细节型
7. "💡 注意看 critic 这个角色 —— 它专门唱反调。你让 AI 写代码，最大的风险是它'讨好你'。"
8. "OMX 做的不是'加功能'，而是给 Codex CLI 补上了一整套工程化管理。"

### 角度5：数据展示型
9. "30 个 agent + 39 个工作流技能 + 6 个并发 = 一支有纪律的作战团队。"
10. "6 个扩展点用到极致：AGENTS.md → 编排大脑，prompts → 30 个 agent 角色，skills → 39 个工作流技能……"

### 角度6：核心洞察型
11. "单 agent 的极限，不在模型能力，而在工程化组织。"
12. "AI 编码工具卷到现在，真正的竞争点已经不是'模型多聪明'，而是能不能把 AI 组织好。"

### 角度7：推荐行动型
13. "如果你是 Codex CLI 的重度用户，这可能是今年最值得装的一个 npm 包。"
14. "只要说 'ralph'、'别停'、'autopilot'、'build me'，OMX 的编排大脑会自动帮你激活对应技能。"
