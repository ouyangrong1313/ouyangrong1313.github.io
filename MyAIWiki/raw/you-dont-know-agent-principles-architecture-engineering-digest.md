# 你不知道的 Agent：原理、架构与工程实践 — 拆解

**来源：** https://x.com/HiTw93/status/2034627967926825175
**标签：** #主题/AIAgent #手法/工程实践 #场景/技术博客

---

## 核心观点

1. **Harness 比模型更关键**：验证测试质量、反馈信号、回退手段往往比模型本身更能决定系统能否收敛
2. **上下文工程决定稳定性**：Context Rot 是最常见的失效模式，分层管理是解决核心
3. **工具设计按 ACI 原则**：面向 Agent 目标而非底层 API，边界清楚、参数防错、示例驱动
4. **记忆系统要单独设计**：四种记忆分类，工作记忆/程序性记忆/情景记忆/语义记忆各有分工
5. **评测系统先于 Agent 调试**：评测出问题先修评测，不要基于失真信号调整方向
6. **多 Agent 先隔离再协作**：协议 > 协作，隔离边界 > 并行收益
7. **安全边界先于功能**：白名单、路径隔离、审计日志必须在第一版就到位

---

## 7个分析角度

### 角度1：反直觉判断冲击
- 更贵的模型带来的提升没有想象中那么大
- 调试 Agent 时应优先检查工具定义而非模型能力
- 评测系统本身的问题很多时候比 Agent 出问题更难发现
- 稳定的大系统提示比频繁变动的小提示实际成本更低（90%折扣）

### 角度2：工程优先于智能
- Agent Loop 核心不到 20 行代码，循环本身相当稳定
- 新能力通过三种方式接入：扩展工具集、调整系统提示、状态外化
- 模型负责推理，外部系统负责状态和边界，分工明确
- 控制流基本不变，主循环很少需要频繁调整

### 角度3：上下文分层设计
- 常驻层（短而硬）→ 按需加载（Skills）→ 运行时注入 → 记忆层 → 系统层
- 确定性逻辑通过 Hooks/代码规则处理，不进上下文
- Skills 描述符要像路由条件而非功能介绍
- Prompt Caching 依赖前缀稳定性，常驻层越稳缓存命中率越高

### 角度4：工具设计三阶段演进
- 第一代 API 封装：粒度过细，Agent 需要协调多个工具
- 第二代 ACI：面向 Agent 目标，一次搞定而非多次调用
- 第三代 Advanced Tool Use：Tool Search + Programmatic Calling + Examples
- 工具数量要克制，能用 Shell/文档/Skill 解决的不新增工具

### 角度5：记忆整合可回退
- 不删除原始消息，只移动指针
- 触发阈值：tokenUsage / maxTokens >= 0.5
- 失败路径写入 archive/，保留完整历史
- 压缩保留优先级：架构决策 > 关键变更 > 验证状态 > TODO > 工具输出

### 角度6：评测比调试更重要
- 先修评测再动 Agent
- 环境噪声导致的"失败"容易被误判为模型问题
- transcript vs outcome，两类都要覆盖
- Pass@k 验证能力边界，Pass^k 保证上线质量，用途不同不能混用

### 角度7：安全先于功能的工程顺序
1. 单渠道先跑通完整链路
2. 安全边界先于功能
3. 记忆整合要早做
4. Skills 先于新工具
5. 第一个失败就建评测

## 开头钩子（每角度3个）

### 钩子集1：反直觉冲击型
1. 你以为换更贵的模型就能提升 Agent 效果？错，Harness 比模型更关键
2. 调试 Agent 时先检查工具定义，因为多数选错工具的原因都出在描述不准确
3. 评测系统出问题比 Agent 出问题更难发现——你可能一直在错误的方向上优化

### 钩子集2：工程本质型
4. Agent Loop 核心不到 20 行代码，真正的复杂度在循环外部
5. 模型负责推理，外部系统负责状态和边界——这个分工决定了系统能不能跑稳
6. 新能力通过三种方式接入：扩展工具集、调整系统提示、状态外化，不动循环本身

### 钩子集3：上下文工程型
7. Context Rot 是最常见的失效模式——看起来像模型能力不足，实际是上下文组织不当
8. 常驻层越稳定，Prompt Caching 缓存命中率越高，边际成本越低
9. Skills 描述符要像路由条件，"何时该用我"比"我能做什么"重要得多

### 钩子集4：工具设计型
10. 不要分别暴露 create_file、write_content、set_permissions，一个 create_script 一次搞定
11. 工具调用准确率从 72% 提升到 90%，只需要加 1-5 个真实调用示例
12. Programmatic Tool Calling 可以把 token 消耗从 150,000 降到约 2,000

### 钩子集5：记忆整合型
13. 不删除原始消息，只移动指针——整合失败也能回到原始存档继续工作
14. 四种记忆各司其职：工作记忆、程序性记忆、情景记忆、语义记忆
15. 压缩时保留优先级第一条：架构决策，不得摘要

### 钩子集6：评测型
16. 看到 Agent 表现下降，先查环境再动 Agent——可能是环境噪声，不是模型退化
17. transcript 和 outcome 两类都要覆盖，只看一边会漏掉重要信息
18. 20-50 个真实失败案例就够启动评测体系，不用等完整体系

### 钩子集7：安全与顺序型
19. 安全边界要先于功能——白名单、路径隔离、审计日志在第一版就要到位
20. 工程实现顺序：单渠道先跑通 → 安全先于功能 → 记忆整合早做 → Skills 先于工具 → 第一个失败就建评测
21. 多 Agent 协作协议先于协作，隔离边界先于并行，不要反过来
