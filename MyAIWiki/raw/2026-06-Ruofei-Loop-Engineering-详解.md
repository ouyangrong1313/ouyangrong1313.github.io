## 引子

这两天 Loop Engineering 突然热了起来。Peter Steinberger 在 X 上说，重点会从一轮一轮提示 coding agent，转向设计能提示 agent 的 loop。Boris Cherny 也提到，他现在更多是在写 loop，让 loop 去驱动 Claude 判断下一步。

这两句话很适合被写成一个刺激标题：提示词工程要过时了。

但这几天读下来，反而不太关心这个口号。Gergely Orosz 的担心很接地气：很多团队没有无限 token，也没到"手工提示已经卡住工作"的阶段。Garry Tan 也提醒过，不要把 Agent 做成机械重复的工厂。Graham Neubig 给的日常 loop 例子里，还保留了一个普通但关键的动作：人先过一遍任务清单，再决定派给哪个 Agent。

这些讨论放在一起，问题反而更清楚了。

> **Loop Engineering 更值得看的地方，不是"人不再写提示词"。更准确地说，提示词开始进入一个反馈系统：发现工作、分配任务、执行、验证、记录状态，然后决定继续、停止，还是交给人。**

放回最近这条线看：写 Harness 时聊的是状态边界和失败闭环；写 Harness Engineering 时换了说法：模型外面需要一张工作台；今天的 Loop Engineering 再往前走半步——

> **如果 Harness 是让 Agent 跑在一个可检查的工作现场里，Loop 接着问的是：工作现场能不能定期醒来，继续发现问题、处理问题、留下证据。**

## 一句话总结

**提示词解决的是"下一句话怎么说"，loop 解决的是"这件事怎么持续做、怎么知道做对、什么时候停"。**

## 核心判断

1. **Loop 不只是 cron，也不只是让 Agent 一直重复"继续优化"**。它更接近一个带反馈的控制循环。
2. **Prompt 没有消失**，只是从聊天框里的手工输入，变成了循环系统里的一个组件。
3. **一个可用的 loop 至少要有五样东西**：自动触发、隔离工作区、过程资产、外部连接、独立验证。再加一个状态记忆。
4. **先从事实核验、CI 分流、文档检查、重复故障归类、陈旧配置清理这类场景试**，不急着碰核心业务代码。
5. **Loop 的价值取决于验证是否便宜**。如果验证很贵、边界很模糊、后果很重，loop 会把问题放大。
6. **不是所有团队现在都需要 loop**。任务重复、验证自动化、预算撑得住、Agent 有真实工具时，再考虑把它放进循环。
7. **对架构师来说，新的工作不是退出现场**。目标、边界、预算、证据和停止条件，会被放到更靠前的位置。

## 控制层：从 Prompt 到 Loop 的层次关系

Model 负责推理，Tool 负责接触外部世界，Skill 负责沉淀做事方法，Sub-agent 负责分工，**Harness 负责让整套东西跑起来、跑得住、跑完还能检查**。**Loop 再往上走一层**——如果说 Harness 管的是"这一次任务怎么跑"，Loop 管的是"这类任务怎么持续发生"。

Addy Osmani 在原文里把 loop 拆成几块：自动化、worktree、skills、plugins/connectors、sub-agents，再加状态记忆。放到团队现场，若飞把它翻译成**六个问题**：

| 问题 | 对应能力 | 解决的风险 |
|---|---|---|
| 什么时候启动 | 自动化、定时任务、`/goal`、`/loop` | 靠人想起来才做 |
| 在哪里改 | worktree、隔离环境、临时分支 | 并行互相覆盖 |
| 按什么规则做 | Skills、项目规则、计划模板 | 每轮重新解释业务 |
| 能连到哪里 | MCP、插件、连接器、CLI | 只能看本地文件 |
| 谁来复核 | sub-agent、reviewer、测试 | 自写自审过于宽松 |
| 怎么接上下一轮 | 状态文件、issue、看板、日志 | 上下文中断和目标漂移 |

从架构上看，loop 拆成**四个口**：

1. **触发入口**：谁能启动，多久启动一次，输入从哪里来
2. **执行沙箱**：在哪个 worktree、分支或临时环境里跑，能不能回滚
3. **验收出口**：用哪些测试、日志、规则和人工复核来判定结果
4. **状态账本**：这轮尝试、证据、失败原因和下一步写到哪里

## 闭环 vs 开环

- **闭环**：像一个边界明确的小控制系统。目标写清楚，动作范围受限，反馈足够客观，停止条件能验证。比如 CI 分流、链接检查、事实核验、依赖升级预检查。
- **开环**：Agent 可以自己继续发现任务、扩展路径、决定下一步，看起来更"自主"，风险也更高：预算更难控，目标更容易漂移，验证也更依赖人工。

> **我会把 loop 的第一站放在闭环，开环先放一放。** 工程里有个老经验：先让一个小闭环跑稳，再谈更大的自动化。

## 验证：四条件准入判断

不会一上来就把核心业务开发交给无人值守 loop。**它更适合"目标清楚、验证便宜、失败可回滚"的任务。**

四条件准入：
1. 这件事会不会重复
2. 能不能自动验证
3. token 预算能不能承受重试
4. Agent 有没有真实工具可用

如果验证很便宜，loop 会放大收益。如果验证很贵，loop 会放大幻觉、成本和维护压力。

**五项准入小表**（小表很管用）：

| 检查项 | 可以试 | 暂缓 |
|---|---|---|
| 输入 | 日志、issue、测试报告、文档 | 靠口头描述，边界每天变 |
| 输出 | 分类表、候选 PR、证据清单、风险说明 | 直接改核心逻辑，影响难评估 |
| 验证 | 有测试、lint、链接检查、可复现命令 | 只能靠人读一遍感觉对不对 |
| 权限 | 默认只读，写入走隔离分支 | 直接改主分支或生产配置 |
| 停止 | 有预算、时间、证据不足等停止条件 | 只写"继续优化"，没有退出点 |

> 五项里只要有两项落在右边，先补测试、补状态、补边界，再考虑自动 loop。

### 适合的 6 类试点场景

1. **技术稿事实核验** — 读多、改少、证据可列
2. **CI 失败分流** — 输入结构化，验证路径明确
3. **文档链接和配置漂移检查** — 可自动扫描，可人工确认
4. **重复故障归类** — 需要并行阅读大量日志和 issue
5. **陈旧 feature flag / 实验配置清理** — 天然可拆，结果可先只读输出
6. **依赖升级预检查** — 可以先生成影响范围和测试建议

> **Agent 可以跑得久一点，但最后的结论能被证据压住。**

## 预算：先放进任务卡

Loop 一旦跑起来，每一轮都可能重新读上下文、调用工具、生成计划、做验证。预算不能等到账单出来再看，**它一开始就放进任务卡**。

示例任务卡（每日 CI 分流）：

```
循环名称：每日 CI 分流
触发频率：每天 09:30
输入范围：过去 24 小时失败 job、最近 20 个 commit
最大运行：30 分钟
最大分支：一次最多处理 5 个失败簇
权限：默认只读；修复必须进入独立 worktree
验证：只运行相关测试，不跑全量回归
停止条件：无新失败、证据不足、预算到达、需要人工决策
交付物：失败分类表、可复现命令、候选 PR、剩余风险
```

这张任务卡把几件事说清楚了：loop 能看什么、能改什么、最多跑多久、哪些情况要停、什么算完成、人什么时候接手。

> 这些边界如果不写，loop 会很快变成一个乐观的 while 循环。它会一直尝试，一直解释，一直"看起来有进展"。**但 momentum 不是 progress。**

## 状态记忆

几篇资料里都提到了极简例子 Ralph loop：

```bash
while :; do cat PROMPT.md | claude-code; done
```

这行命令粗糙，但它点出了一个关键事实：**每一轮都重新读取外部文件和当前仓库状态，而不是指望聊天窗口记住一切**。

> **loop 的记忆不能只放在对话里。**

长任务里，麻烦的不是 Agent 不努力，而是每一轮都像新来的同事。它不知道昨天试过什么，不知道哪个方向已经被否掉，不知道哪个测试因为环境问题可以暂时忽略，也不知道哪条边界绝对不能碰。

loop 需要一个对话之外的状态载体。可以是 `plan.md`、issue、看板、数据库，也可以是一份很普通的 Markdown：

```markdown
## 当前目标
修复订单导出最近 24 小时的 CI 失败。

## 已尝试
- 读取 job 1324 日志，失败点在 export timeout
- 检查最近 20 个 commit，疑似与批量查询改动有关
- 尝试增加超时，不通过，已回滚

## 已验证
- 单测 OrderExportTest.test_large_batch 复现失败
- 小批量导出正常

## 禁止事项
- 不改权限模型
- 不改导出文件格式

## 下一步
- 对比批量查询前后 SQL
- 如无法定位，交给人工看慢查询日志
```

> **如果没有状态记忆，loop 就会变成一串断开的 prompt。看起来连续，实际上每轮都在重新开始。**

## 人在场：Loop 越强，判断越要提前

Loop Engineering 常被误读成：人可以从流程里拿掉。若飞的体感刚好相反：**Loop 越强，人的判断越要提前出现**。

以前人是在每一轮 prompt 里临时判断：下一步做什么、哪个结果可信、哪个文件不能碰、哪个测试失败可以接受、什么时候停。现在这些判断会提前变成规则、模板、权限、预算和停止条件。

Graham Neubig 那个例子给的提醒也在这里：他的 loop 没有把所有待办直接丢给 Agent，而是先整理信息、走一遍优先级，再由人决定哪些任务值得委派。这一步看起来慢，其实是在保护系统不被"自动化的热情"带偏。

**保守的 5 条原则**：

1. 先只读，后写入
2. 先低风险，后核心路径
3. 先小频率，后高频率
4. 先人工确认，后自动合并
5. **先写停止条件，再写继续条件**

> 很多自动化出问题，不是因为不会继续，而是因为不知道什么时候停。一个成熟的 loop，最好能诚实地说：
> - "我没有足够证据继续"
> - "这次修改超过了授权范围"
> - "预算已经到达"
> - "验证结果不稳定"
> - "需要人做产品判断"

## 7 天试点

如果团队想试，从一个很小的 7 天试点开始。不用一上来就做"自动开发平台"，先做一个"每日可验证循环"。

- **第 1 天：选场景** — 选一个低风险任务（CI 分流、文档链接检查、技术稿事实核验）
- **第 2 天：写任务卡** — 把输入范围、权限、预算、停止条件、交付物写下来
- **第 3 天：做 Skill** — 把项目规则、常见误区、验证命令、输出格式写成 Skill
- **第 4 天：接状态记忆** — 用 `plan.md`、issue 或看板记录已尝试、已验证、阻塞和下一步
- **第 5 天：跑一次手动 loop** — 人手动触发每一步，看哪些地方需要自动化
- **第 6 天：加自动触发** — 只给一个固定频率，设置时间上限和预算上限，输出先进入人工 inbox
- **第 7 天：复盘** — 看三件事：省了多少重复排查、误报多少、人工复核是否更轻

**复盘验收口径**：

| 指标 | 看什么 | 算可继续 |
|---|---|---|
| 命中率 | loop 找到的问题里有多少确实值得处理 | 人工复核后仍有稳定收益 |
| 误报率 | 有多少输出会浪费人的时间 | 不影响日常节奏 |
| 回滚率 | Agent 写入后有多少需要撤回 | 写入先控制在候选分支 |
| 成本 | token、运行时间、工具调用次数 | 账单和收益能对上 |
| 证据 | 每个结论有没有来源、命令、日志或测试结果 | 人能在 5 分钟内复核一轮 |

> 如果这些数说不清，问题通常不在 Agent，而在工作流本身还没有被定义清楚。这套试点不炫目，但它能回答一个关键问题：**团队到底缺的是更强的 Agent，还是更清楚的工作现场**。很多时候，答案是后者。

## 收束

Loop Engineering 这个词可能会热一阵，也可能很快被下一个词盖过去。但它指向的变化是真实的——当 Agent 可以长时间运行、调用工具、拆任务、开 worktree、接插件、找 sub-agent 复核时，工程师的发力点确实会移动。

以前把大量精力花在"这一轮怎么说清楚"，接下来会有更多精力放在"这类事怎么被系统稳定地做完"。**这不是提示词工程的结束。提示词被放回工程系统里。**

若飞对 loop 的态度：**可以试，值得学，不用神化。**

- 它能放大已经清楚的工作流，但想清楚工作流这件事仍在人这里
- 它能持续处理可验证的小闭环，但模糊的业务判断仍要有人负责
- 它能让工程师少做重复操作，但不能让工程师少理解系统

> **循环越自动，人的判断越要在场。因为 loop 本身不知道你是在加速理解，还是在逃避理解。这道分界线，最后还是要由人负责。**
