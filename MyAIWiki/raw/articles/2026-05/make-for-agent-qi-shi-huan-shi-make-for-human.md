创业者闭门探讨：Make for Agent ，其实还是 Make for Human

来源：Founder Park
原文链接：https://mp.weixin.qq.com/s/CZxwlhWg6PxwDmsIY0YXuw
抓取时间：2026-05-27

---

本文作者「马多灵」，极客公园资深研究员。「Make for Agent」主题闭门 Meetup 的主持人。

上周六，我们组织了一场闭门 Meetup，主题是：Make for Agent 的产品应该怎么做？面向一线的 Agent 创业者和 Builders。

原本规划的议题是以「Agent IM → Agent 协作 / Workspace → Agent Habitat」，但在实际交流中，我们发现，大家关于第一层 Agent IM 的探讨花费了大部分的时间。这也传递出了当下 Agent 创业中一个非常实际的信号：当前业内做 agent 产品的真实卡点并不在一些远大的前景上，反而在离人最近的入口、身份、权限、上下文和控制感。

甚至，连「产品的用户不再是人」这个说法也需要打个问号。核心变化可能不再是当用户从 Human 变成了 Agent，而是 Human-AI 协作中的「责任链」被重新拆开了。

印象最深的，是来自现场的一位创业者感慨：「四五月份其实挺无聊的，因为大家太共识了。但今天交流发现，大家其实没什么共识，只有宏观上共识，在微观上差异很大。」

宏观共识太便宜了，大家都同意工作流会变、个人会被放大。都对，但没什么用。有价值的，其实是这场交流中的一些微观分歧：

- Agent 到底要不要 IM？
- 人类到底管目标、过程还是结果？
- Agent 是否应该拥有持续身份？
- 企业客户买的是效率，还是管控？
- 什么任务应该用 App，什么任务才值得上 Agent？
- Agent 之间通信时，传的是消息、意图、状态，还是责任？
- 当 Agent 出错时，是用户负责、产品负责、模型负责，还是组织负责？

以下为清洗后的核心正文节选：

## 01 Agent 不需要像人一样聊天，但它需要被追责

文章认为，Agent 不一定需要人类式 IM，但一定需要通信基础设施：寻址、可靠送达、事件总线、状态同步、上下文引用。关键不是“能不能聊天”，而是通信是否可追责、是否在关键时刻可见。

## 02 真正的问题不是通信，而是身份

Make for Agent 的第一性问题不是界面，而是身份系统。没有独立身份，就没有细粒度权限；没有细粒度权限，就没有清晰责任。Agent 真正进入组织时，必须回答“它代表谁行动、权限来自谁、出了事谁负责”。

## 03 人类介入不是“什么时候”，而是“管什么”

Human-in-the-loop 的重点不是让人始终卡在流程中间审批，而是在责任链关键断点上介入：目标设定、权限边界、结果验收。产品层需要把控制感设计成可调节能力，而不是统一流程。

## 04 Make for Agent 的生意，仍然是 Make for Human

哪怕产品口号是 Make for Agent，最终买单的仍然是人和组织，因此真正购买的是控制感与可治理性。文章把当前最容易落地、最容易产生收入的 Agent 产品，归因于软件工程天然具备责任基础设施（Git、PR、CI、测试、review、rollback）。

## 05 企业 AI 落地的瓶颈不是模型，是上下文和想象力

文章把企业 Agent 的上下文拆成三层：data context、knowledge context、identity context。企业 Agent 难落地，不只是模型不够强，而是上下文天然嘈杂、边界混乱、责任沉重，需要先完成显式化、结构化、可治理。

## 06 最危险的误判：把 Agent 当新人

Agent 既不是传统工具，也不是组织新人，更像是一种新的执行单元。把它完全当工具，会低估其自主性；把它完全当新人，又会把产品做成拟人化低效剧场。

## 07 产品也要讲究 Agent 友好度

面向人的设计解决控制感，面向 Agent 的设计解决结构化输入、低噪声上下文、明确权限与可执行接口。文章提醒：不是把人类 UI 改成 Agent UI，而是在同一产品内同时满足两套需求。

## 08 Make for Agent，其实是 Make for Responsibility

全文最终收束到一个核心判断：Make for Agent，本质是 Make for Responsibility。Agent 的出现，不只是多了一个“用户类型”，而是逼着产品把目标、权限、上下文、行动和后果重新拆开，并显式设计责任链。
