# Agent 技术演化新范式（2026）Digest

## 一句话总结

这篇文章的核心观点是：Agent 的核心模块没有变，但实现范式已经从早期的 Prompt/Function Call 驱动，演化为更加工程化、文件系统化、运行时化、可积累的体系。

## 四阶段演化

1. **早期 Agent（被动式 ReAct）**
   - 一问一答
   - 依赖明确指令
   - 擅长短链路小任务
   - 缺少长期规划能力

2. **Workflow Agent（结构化可控）**
   - 用固定流程弥补模型不确定性
   - 特别适合 to B 和重复性任务
   - 以稳定性和可解释性见长

3. **自主 Agent（复杂规划与长程任务）**
   - 能拆解模糊目标
   - 能持续执行多步任务
   - 逐步从“辅助者”变为“执行者”

4. **自进化 Agent（持续学习与自我升级）**
   - 能沉淀技能、知识库、记忆
   - 能从历史任务中积累经验
   - 目标是“越用越好用”

## 六个核心维度的变化

### Prompt：深耦合 → 渐进式加载
- 早期把人设、规则、示例都塞进 System Prompt
- 现在强调把动态内容外置到 `SKILL.md`、`USER.md`、`SOUL.md`、`AGENTS.md` 等文件
- 核心是让系统提示词更稳定，把变化内容模块化加载

### Planning：思维链 → 长程规划
- 从简单 CoT 走向复杂任务拆解
- 能生成 Todo List，动态调整计划
- 还能按子任务需求构建或调用子 Agent

### Memory：检索增强 → 文件系统化
- 短期记忆更强调压缩、摘要和重点提取
- 长期记忆更强调 Markdown 文件沉淀
- 趋势是文件系统 + 向量检索混合管理

### Tools：Function Call → CLI / Script
- 早期依赖 API 和函数 Schema
- 现在更强调 CLI 原生能力和脚本封装
- 模型更容易利用命令行和脚本这种“先天熟悉”的工具形式

### Workflow：刚性编排 → 动态混合封装
- 早期依赖固定状态机式流程
- 现在更倾向于 Skill 驱动，Workflow 做主干兜底
- 推荐“Skill 为主，Workflow 为辅”

### Environment：无状态 → 运行时环境
- Agent 不再只是一次性调用
- 需要 workspace、文件读写、状态持久化、隔离和权限边界
- 本地环境灵活，沙箱环境更适合企业级稳定运行

## 最值得记住的结论

> Agent 正在从“魔法调优”走向“系统工程”。

真正的变化不是模块名字变了，而是每个模块背后的工程实现逻辑都升级了。

## 对实际落地的启发

- 不要再把 Agent 理解成“大 Prompt + 几个 Function Call”
- 文件系统、技能文件、运行时工作区、脚本资源、混合工作流，已经成为新一代 Agent 的基础设施
- 企业落地要同时考虑灵活性、稳定性、可控性和长期可积累性
