# 【深度拆解】OpenClaw vs Hermes：多 Agent 架构设计

- 原文链接：https://mp.weixin.qq.com/s/0GvtgYRJBSietf24K-d7ug
- 来源：微信公众号
- 获取时间：2026-05-14

## 核心观点

这篇文章围绕一个很实际的问题展开：**多 Agent 架构到底什么时候有必要？如果真的要做，OpenClaw 和 Hermes 分别是怎么设计的？**

作者没有直接站队，而是先强调一个前提：

> 多 Agent 不是默认更高级，也不是所有产品都应该抄的平台设计。

文章引用了 Devin 团队“Don’t Build Multi-Agents”的思路，先把问题说透：

- 很多场景下，单 Agent + 好的 Context Engineering 就足够了
- 多 Agent 的主要风险是：
  - 上下文割裂
  - 隐性决策丢失
  - 错误复合
  - 协调成本上升

但作者也认为，多 Agent 并不是错，而是在任务复杂度、平台属性和系统目标上升之后，成为一种合理的工程回应。

因此，这篇文章真正想说明的是：

1. 多 Agent 的适用边界是什么
2. 常见多 Agent 模式有哪些
3. OpenClaw 和 Hermes 分别如何在工程层面实现多 Agent
4. 两者的核心差异不在“有没有多 Agent”，而在**多 Agent 被放进了什么运行边界里**

作者最后给出的最关键判断是：

- **OpenClaw：把子 Agent 放进会话边界**
- **Hermes：把子 Agent 放进进程边界**

这个差异会继续影响：

- 结果回传方式
- 状态管理方式
- 权限裁剪方式
- 生命周期管理方式
- 中断与清理方式

## 文章结构梳理

### 1. 多 Agent 不是默认答案

文章开头先纠偏：

很多人看到 OpenClaw、Hermes 这种系统用了多 Agent，就容易默认认为：

- 多 Agent = 更先进
- 别人用了，自己也该用

但作者反复提醒：

> 多数业务并不一定需要多 Agent。

因为单 Agent 在很多任务里依然表现稳定，而多 Agent 会显著增加系统工程复杂度。

真正值得上多 Agent 的场景，通常是：

- 任务复杂度高
- 上下文容易膨胀
- 工具种类很多
- 任务存在天然并行性
- 需要明确权限隔离
- 希望把局部失败限制在局部执行单元里

### 2. 多 Agent 的价值，本质是把复杂任务拆成隔离执行单元

作者把多 Agent 定义成：

- 一个主 Agent
- 若干子 Agent
- 一套编排机制
- 一条结果回传链路

而不是简单理解成“同时跑多个模型”。

它要解决的问题主要有两类：

#### （1）模型层面的瓶颈
- 长上下文污染
- 推理成本上升
- 工具误用概率变高

#### （2）工程层面的耦合
- 状态管理混在一起
- 并发难隔离
- 权限难收口
- 局部异常容易拖垮整个流程

所以多 Agent 的本质不是“让系统更花哨”，而是：

> 把复杂任务拆成多个拥有独立上下文和权限边界的执行单元。

### 3. 常见多 Agent 实现模式

文章总结了几类经典模式：

#### （1）调度-执行模式
- 父 Agent 负责拆分任务与收敛结果
- 子 Agent 负责局部执行

这是最基础、也最常见的模式。

#### （2）分层编排模式
- 子 Agent 不只是执行，还能继续拆分任务
- 适合层次明显的大任务

但这种模式风险更高，容易任务树失控，因此必须限制递归深度、子 Agent 数量和工具权限。

#### （3）专家路由模式
- 把任务按类型交给不同角色的 Agent
- 比如搜索专家、代码专家、安全专家、文案专家

作者明确说，这是他自己实践中用得很深的模式。

#### （4）生成-审查模式
- 一个 Agent 先生成
- 另一个 Agent 负责审查、质疑、校验
- 再由主 Agent 决定是否采纳

这很适合代码审查、事实核对、推理校验。

#### （5）Ensemble / Mixture-of-Agents 模式
- 多个参考模型并行生成候选结果
- 再由一个聚合器综合输出

这里的重点不一定是“多工具 Agent”，也可能只是多条独立推理路径。

### 4. OpenClaw 的多 Agent：子会话模型

文章把 OpenClaw 的设计总结得很清楚：

> 父 Agent 创建的不是临时函数调用，而是一个新的子会话。

这个子会话有：

- 会话标识
- 生命周期状态
- 工具策略
- 父子关系
- 工作目录
- 角色和层级信息

然后运行网关接管它，后续结果通过事件链路回传。

作者重点强调了 OpenClaw 的几个设计点：

#### （1）统一入口负责创建子运行时
父 Agent 只需提交任务和运行参数，系统统一处理：

- 运行时选择
- 会话模式 / 一次性模式
- 上下文隔离 / 继承
- 沙箱策略
- 线程绑定
- 模型覆盖

#### （2）角色与层级写入会话元数据
系统会记录：

- 当前层级
- 当前角色
- 父子关系
- 工作目录
- 是否允许继续派生子 Agent

这些元数据会直接影响工具权限和恢复逻辑。

#### （3）行为约束与工具约束分开
- 提示词层负责告诉子 Agent 应该怎么行动
- 运行时层负责裁剪它实际能用什么工具

#### （4）结果不是同步返回，而是事件回传
OpenClaw 不是“父调用 -> 子返回”，而是：

- 子任务进入注册表
- 等待完成
- 事件投递回目标会话
- 必要时唤醒上层继续收敛

这让它天然更适合：

- 持久会话
- 跨线程
- 跨渠道交付
- 生命周期可观测与可恢复的场景

### 5. Hermes 的多 Agent：进程内委派模型

Hermes 的设计思路和 OpenClaw 很不一样。

它不是围绕“子会话”来展开，而是：

> 父 Agent 发起一次任务委派，在同一进程里创建多个子 Agent 对象并行执行，最后把结构化结果同步返回。

作者总结 Hermes 的关键设计有：

#### （1）多 Agent 能力被封装成一个工具
父 Agent 调用委派工具，就像调其他工具一样。

这让接入成本低，但也意味着：
- 多 Agent 更多是“工具级能力”
- 不是会话系统级能力

#### （2）子 Agent 是新的执行单元，但仍在同一进程
它们有自己的：

- 提示词
- 工具集
- 任务标识
- 进度回调

但不拥有独立会话身份。

#### （3）权限继承 + 强制裁剪
子 Agent 的工具上界不能超过父 Agent，系统还会默认剥离一些能力，比如：

- 继续委派
- 主动向用户追问
- 写长期记忆
- 主动发消息
- 执行脚本代码

#### （4）角色降级机制
Hermes 也有执行角色 / 调度角色区分，但是否允许继续派生，要结合：

- 当前层级
- 最大派生深度
- 是否开启继续委派开关

默认配置下，Hermes 也是单层并行展开。

#### （5）结果以结构化数组同步返回
OpenClaw 走事件回传，Hermes 更像同步调用：

- 父 Agent 发起委派
- 等待结果数组返回
- 再继续后续推理

所以 Hermes 的调用链更短，但也意味着更多状态管理要在当前进程里自己补齐。

### 6. Hermes 对运行期问题的补强

因为 Hermes 是进程内并行，所以它在运行期额外处理了几个问题：

#### （1）运行状态表
记录：
- 子 Agent 标识
- 父 Agent 标识
- 任务目标
- 当前层级
- 状态
- 模型
- 工具调用次数

解决“并行子 Agent 现在跑到哪了”的可观测性问题。

#### （2）中断递归传播
父 Agent 被打断时，子 Agent 也会收到中断信号，避免父停了子还在后台跑。

#### （3）文件状态登记
记录：
- 哪个任务读过什么文件
- 哪个文件最近被谁写过
- 写回前内容是否过期

这套机制本质上是解决多 Agent 并发修改文件时的覆盖问题。

### 7. 文章的最终比较

作者最后没有简单说谁更优，而是把差异收敛成一句很关键的话：

- **OpenClaw 的边界是会话**
- **Hermes 的边界是进程**

因此两者的后续设计也顺理成章地不同：

| 维度 | OpenClaw | Hermes |
|---|---|---|
| 子 Agent 形态 | 子会话 | 进程内子 Agent 对象 |
| 结果回传 | 完成事件投递 | 同步结构化返回 |
| 运行边界 | 会话系统 | 当前进程 |
| 适合场景 | 持久会话、跨线程、跨渠道 | 短链路委派、低接入成本 |
| 状态管理重点 | 生命周期与事件回传 | 运行期可观测与文件协调 |

作者自己的偏好是：

> 目前更喜欢专家模型（专家路由模式）。

## 我的理解

我觉得这篇文章最有价值的，不是单纯比较 OpenClaw 和 Hermes，而是帮我们把“多 Agent”从概念讨论拉回工程判断。

### 1. 多 Agent 的问题从来不是“能不能做”，而是“值不值得做”

很多团队讨论多 Agent，第一反应是：
- 怎么拆
- 怎么编排
- 怎么并发

但更前面的问题应该是：

> 当前任务复杂度，真的已经高到需要多 Agent 了吗？

如果：
- 上下文还不大
- 工具不多
- 任务天然是线性的
- 权限边界也不复杂

那一个 Agent 加好的上下文工程，往往更稳。

### 2. OpenClaw 和 Hermes 的差别，本质上是“系统重心”不同

OpenClaw 更像一个：
- 会话系统
- 编排系统
- 权限与生命周期系统

Hermes 更像一个：
- 进程内委派框架
- 并行执行框架
- 同步结果收敛框架

所以不能只看表面上“都能起多个 Agent”，而要看：

- 你的产品是不是会话型产品
- 你是不是要跨渠道持续交付
- 你是否需要长期状态与事件治理
- 你是否更关心短链路并发执行效率

### 3. 多 Agent 架构的真正代价，是状态治理成本

文章里反复提到：
- 上下文割裂
- 决策丢失
- 协调成本上升

这三个点非常关键。

很多人以为多 Agent 的成本是 token 或延迟，其实更深层的是：

- 谁知道当前任务树进展到哪
- 谁知道哪个结论是最终版
- 谁知道哪个子 Agent 的失败该怎么处理
- 谁来保证多个子 Agent 修改结果不互相污染

也就是说，**多 Agent 真正难的是“系统收口”，不是“把任务分出去”。**

### 4. 专家路由确实是一个很现实的中间解

我也比较认同作者对专家路由模式的偏好。  
因为它相比层级递归委派，有几个现实优势：

- 职责边界更清楚
- 任务描述更容易标准化
- 工具权限更容易收窄
- 输出格式更容易统一
- 系统更容易调试与复盘

换句话说：

> 如果团队还没有很强的多层编排能力，先把“专家路由 + 单层并行 + 明确验证”做好，往往是更稳的路径。

## 对 OpenClaw / Agent 系统设计的启发

### 1. 多 Agent 之前，先把 Context Engineering 做好
如果上下文工程做不好，多 Agent 只会把混乱拆成多个小混乱。

### 2. 多 Agent 设计首先是边界设计
先想清楚：
- 是会话边界还是进程边界
- 是同步返回还是事件回传
- 是单层并行还是允许递归派生
- 是角色路由还是通用执行

### 3. 角色、权限、生命周期必须结构化
这些东西不能只写在 prompt 里，最好进入：
- 会话元数据
- 运行时状态
- 工具权限策略
- 调度控制逻辑

### 4. 子 Agent 完成不是结束，收敛才是结束
真正难的不是跑起来，而是：
- 汇总
- 冲突处理
- 失败隔离
- 结果落地
- 清理与回收
