## 正文

大家好，我是PaperAgent，不是Agent！

当前推理模型（如 OpenAI o1、DeepSeek-R1）虽然准确率惊人，但动辄生成数千甚至上万字的「思考过程」，导致推理成本和延迟急剧上升。

奇元科技、清华、北大等提出的 **TRS（Thinking with Reasoning Skills）** 框架，无需训练、黑盒兼容，通过将历史推理轨迹蒸馏成可复用的技能卡片，在推理时检索注入，实现了更少 Token、更高准确率的反直觉突破：在数学和编程任务上，Token 消耗降低 6%-59%，准确率不降反升。

## 1. 推理模型的 Token 通胀危机

现代推理模型（LRMs）通过显式中间思考（Chain-of-Thought）大幅提升了数学和代码的可靠性，但也带来了一个生产级瓶颈：测试时计算成本与 Token 数量成正比。

以商业 API 计费模式为例，输出 Token 往往比输入 Token 更贵。当模型面对复杂问题时，会产生大量冗余的验证、试错和回溯循环。业界报告也证实，推理密集型负载正在显著放大基础设施压力。

现有提速方案（如 Chain-of-Draft、TALE、NoWait）本质上都在做同一件事：让模型「想得更短」。但强制压缩思考空间往往导致效率-准确率权衡：简单题省了 Token，难题却容易翻车。

核心问题：能不能不让模型每次都「从零推导」，而是像人类专家一样，直接调用已沉淀的解题经验？

## 2. 核心洞察：从「从零推理」到「回忆技能」

人类专家解题时很少从头推导。他们依赖从过去练习中提炼出的可复用技能，例如「寻找不变量」「双指针」「链式法则」。TRS 正是将这一认知模式系统化：

- **离线（Offline）：** 将模型解决历史问题时的长轨迹，包括成功路径和失败教训，蒸馏成结构化的技能卡片（Skill Card）
- **在线（Online）：** 面对新问题时，检索最相关的技能卡片注入 Prompt，引导模型走「直达路径」

标准 CoT 在求解积分时需要经历「分部积分→三角替换→试错」的高 Token 消耗探索；而 TRS 检索到「链式法则 + 换元法」技能后，可以直接三步出解，Token 成本大幅降低。

## 3. 方法详解：TRS 框架

### 3.1 技能卡片的设计（Skill Card Schema）

每个技能卡片是一个高度结构化的紧凑文本，包含五个字段：

| 字段 | 含义 |
|------|------|
| Trigger | 适用场景触发词 |
| Do | 核心操作步骤，最小可执行配方 |
| Avoid | 反模式、常见陷阱 |
| Check | 必须验证的约束或不变量 |
| Risk | 边界情况与失败模式 |

对于正确的解题轨迹，卡片提炼成功模式；对于错误轨迹，卡片提炼「反模式 → 修正策略」。这种「从失败中学习」的设计，是 TRS 能在难题上提升准确率的关键。

### 3.2 离线构建技能库

1. 对源问题运行推理模型，得到轨迹和结果
2. 用更强的蒸馏模型将轨迹压缩为技能卡片和 10-20 个检索关键词
3. 以 Key-Value 形式存入技能库：Key = Concat(问题, 关键词)，Value = 技能卡片

论文使用 DEEPMATH-103K 和 NEMOTRON-COMPETITIVEPROGRAMMING-V1 进行验证。

### 3.3 在线检索与注入

面对新查询时：

- **检索：** 使用 BM25（数学）或 Hybrid（BM25 + Dense Embedding，代码）检索 top-k 技能
- **注入：** 将技能卡片前置到 Prompt 中
- **轻量门控：** Prompt 中附带仲裁指令：「仅使用直接适用的技能；忽略无关或矛盾建议」

虽然注入技能增加了输入长度，但它消除了冗余探索分支、试错循环和重复验证。实验表明，输出 Token 的减少量远大于输入 Token 的增加量，最终端到端成本和延迟均下降。

## 4. 主实验：打破效率-准确率权衡

### 4.1 数学推理（DeepMath-103K）

关键发现：

- Doubao Seed 在几乎不损失准确率（-0.2%）的情况下，Token 数腰斩，成本降低 53.8%
- GPT-4o-mini 在 TRS 加持下，准确率提升 1.8%，成本降低 6.9%
- GPT-OSS-120B 保持准确率不变，成本降低 16.9%

### 4.2 代码竞赛（Competitive Programming）

- GPT-4o-mini：准确率从 22.0% 提升到 24.4%，成本下降 6.3%
- Doubao Seed-2.0：准确率从 63.6% 提升到 64.4%，成本下降 6.0%
- GPT-OSS-120B：准确率从 54.2% 提升到 58.3%，Prompt 增加导致成本微升 4.8%，但准确率提升显著

## 5. 深度分析：为什么 TRS 能赢？

### 5.1 难题上优势更大

现有提速方法（TALE 强制预算、CoD 极简草稿、NoWait 抑制反思词）在难题上普遍出现崩溃。

按基线思考长度切片显示：题目越难，TALE 和 CoD 的准确率下滑越明显；TRS 在 GPT-OSS 上，最难区间准确率从约 45% 提升至约 80%，同时将 Token 从约 15k 压到约 7k。

结论：强制「想短」会削弱深度推理；TRS 通过提供导航地图（技能卡片），让模型在复杂解空间中不迷路，自然不需要那么长的试错轨迹。

### 5.2 控制实验：不是简单的 RAG

消融实验证明，TRS 的收益不能被简单检索解释。只有结构化技能卡片 + 足够覆盖率的组合才能释放 TRS 潜力。这说明模型需要的不是「相关上下文」，而是可执行的过程性指导。

### 5.3 跨模型迁移：强模型蒸馏，弱模型受益

用 Doubao 生成的技能库给 OSS 用，或用 OSS 技能库给 Doubao 用，均能带来正向收益。同模型风格对齐时收益最大。

工程意义：企业可以用强模型离线蒸馏技能库，部署时供轻量模型检索使用，实现「大师经验，学徒执行」的成本结构。

### 5.4 检索策略：数学靠 BM25，代码靠 Hybrid

数学问题表面触发词 lexical overlap 高，BM25 即可；代码问题表面描述差异大但算法模式相似，需要 Dense Embedding 捕捉语义。论文默认：数学用 BM25(k=1)，代码用 Hybrid(k=5)。

### 5.5 外部竞赛数学迁移：AoPS 技能库

作者从 AoPS 竞赛题库蒸馏 7,616 张技能卡片，在 AIME 2024/2025/2026 和 HMMT 2025 上测试。结果显示：25 个模型-基准对中，13 对准确率提升，20 对成本降低。
