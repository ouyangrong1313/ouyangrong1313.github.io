## 核心命题(一句话)

**AI 原生 ≠ 人人都用 AI 工具;AI 原生 = 让 AI 真正参与团队协作流程,把视角从"AI 能不能帮我写代码"切换为"AI 依据什么写代码"。Spec-Kit 拉上限、依赖团队基建;BMAD 拉下限、依赖人审阅。**

## 5 个核心观点

### #01 AI 原生组织的"割裂感" = 个人提效 ≠ 组织提效
- 个人要求:把 AI 能力发挥到极致
- 组织要求:为 AI 准备好稳定的环境
- 一线使用者看"酷",组织建设者看"到处是规则、人工兜底、半自动流水线"
- 真正要做好必须处理清单:权限、监听范围、PR 改动范围、单测兜底、模型冲突、上下文压缩、误判优先级、合并事故责任等十几项
- **核心结论**:AI 原生 = 人人都用 AI 工具,但 人人都用 AI 工具 ≠ AI 原生

### #02 SDD(Spec-Driven Development)= 当 AI 进入研发流程后,如何组织需求/方案/任务/代码
- GitHub 2025 年开源 Spec-Kit 把 SDD 推到更多开发者面前
- 沿这条线社区出现三股不同风格的工具:
  - **Spec-Kit** — 标准化规格流程(强秩序)
  - **OpenSpec** — 轻量级规格变更(改什么)
  - **BMAD** — 完整 AI 研发流程编排(角色化 + 圆桌)
- 共同回答:**"维护 AI 稳定运行需要的上下文环境"** = 需求边界 + 代码边界 + 特殊业务规则

### #03 Spec-Kit:秩序感强,但真实项目复杂度经常不在主流程里
- 4 步斜杠命令:`/specify` → `/plan` → `/tasks` → `/implement`
- 价值:强制把散乱信息标准化,解决"返工内耗"
- **真实项目 3 大硬骨头**:
  1. **多仓适配**:主仓 spec 怎么被前/后端子仓各自消费;改一个字段影响哪些地方
  2. **顺序 vs 并行**:Spec-Kit 是 1234 顺序流程,现实前后端是并行
  3. **来回拉扯**:实际是 123 → 234 → 12341234 反复,需求/方案/任务/代码不断互相推
- **作者三类扩展(补足 Spec-Kit 链路之外的事)**:
  1. **多仓适配** — 封装按端自动切片的实现命令
  2. **多视角评审** — 搞一组角色化评审命令(架构/后端/QA 各看各的)
  3. **门禁卡点** — Feature 启动前/API ID 冲突检查;写完后/合规验证;发布前/全量检查
- **适用边界**:✅ 项目产品沉淀好 + 工程事实源不错 + 单一 Feature;❌ 基建薄弱需要大量人为治理的复杂项目

### #04 BMAD:更像一个 AI 团队在围着项目追问
- **是什么**:AI-driven development framework = 角色化 Agent + 引导式工作流
- 4 大阶段:**分析(可选)→ 规划 → 方案设计 → 实现**
- 内置角色:产品 / 架构师 / 开发者 / QA
- 特色功能:**Party 模式**(多角色同会话互呛)+ **圆桌评审**(多 Agent 审同一问题)
- **作者用 BMAD 的两个被点醒瞬间**:
  - **瞬间 1:从 SaaS 错位成 BPO** — BMAD 引导他发现这是 BPO 系统(把公司内部资源包装做对外业务输出)而不是标准 SaaS,建设方向从源头被改
  - **瞬间 2:圆桌挑战"行为基线"** — 写"切换后研发几乎无感"被架构 Agent 追问"你拿什么证明?老系统行为基线你采了吗?",最终落成硬约束 **"没有基线,不切换"**
- **砍需求要分阶段持续砍**:
  - PRD 阶段一次性砍 13 项
  - 架构砍一轮
  - epics 又砍一轮
  - 逻辑都一样:留它写它测它维护它的代价 > 它能带来的收益
- **BMAD 最大的问题:心智负担**:
  - 工作量翻倍(强迫症 + 怕 AI 脑补 = 细节全审)
  - 流程长、内容多、审阅累
  - 不擅长的领域(前端审后端)要先吃透再做判断
  - AI 是"断断续续"协同,不像人类"拉个会评完各自分头干"
  - **整体时间其实不见得比原始人类协同快**

### #05 一句话的差异

| 维度 | **Spec-Kit** | **BMAD** |
|---|---|---|
| 拉什么 | **拉你的上限** | **拉你的下限** |
| 依赖 | 团队基建 + 规范 + 能力 | Agent 内部封装 + 圆桌挑战 |
| 适合 | 团队能力强、基建完善 | 一人团队 / 角色缺失的小团队 |
| 主要风险 | 真实项目复杂度不在主流程,需大量补链路外机制 | 心智负担重,审阅工作量不亚于人类评审 |

> **最终拍板都要靠人,差别在于 AI 帮你补的是上限还是下限。**

## 7 个分析角度

### 1. 视角的切换
- 从"AI 能不能帮我写代码?"切换为"AI 依据什么写代码?"
- 这是组织建设 vs 个人提效的根本分水岭

### 2. SDD 三剑客的分工
- Spec-Kit = 秩序 + 标准
- OpenSpec = 变更 + 演进
- BMAD = 编排 + 角色化
- 三者可以组合用,不是单选题

### 3. Spec-Kit 在"重构迁移"场景的真实表现
- 多仓:submodule 挂载、规则文件按端分发
- 漂移风险:"先把代码搞通"会变成代码新/文档老
- 实际推进是 1234 反复回拉,不是一次走完

### 4. BMAD 的两个"点醒瞬间"
- 业务定位:从标准 SaaS 错位成 BPO → 建设方向从源头改
- 质量基线:从模糊"对方无感"到硬约束"没有基线不切换"

### 5. 砍需求的"分阶段持续砍"原则
- 不是上线前一次性砍,是每阶段砍一波
- 砍的判定标准:**"代价 > 收益"** 即可砍

### 6. 心智负担是 AI 协作的真实成本
- BMAD 把代码速度提上去了,但审阅工作量没减少
- 跟 AI 协同是"断断续续",跟人类协同是"一次会议"
- **"输入确定时,BMAD 提效;前期方案设计阶段,反而比人类评审更累"**

### 7. AI 原生团队的核心
- **不是"多买几套 AI 工具"**
- **是"重新组织协作流程"**
- AI 是新的协作成员,不是新的 IDE

## 关键术语

- **SDD**(Spec-Driven Development):规格驱动开发
- **Spec-Kit**:GitHub 2025 开源,标准化规格流程(`/specify` / `/plan` / `/tasks` / `/implement`)
- **OpenSpec**:轻量级规格变更
- **BMAD**:AI-driven development framework,角色化 Agent + 引导式工作流 + 圆桌评审
- **Party 模式**:多 Agent 同会话互呛
- **圆桌评审**:多角色 Agent 对同一问题分别分析
- **BPO**:Business Process Outsourcing,作者最终发现这个系统是 BPO 而非标准 SaaS
- **行为基线**:老系统历史演化出来的行为事实,没有基线不能切换
- **心智负担**:跟 AI 协作时大量审阅和决策带来的认知压力
- **硬约束**:落成可验证条款的强约束(如"没有基线不切换")
- **AI 原生**:让 AI 真正参与团队协作流程,而非"人人有 AI 工具"
- **拉上限 vs 拉下限**:Spec-Kit 把团队能力上限拉高,BMAD 把团队能力下限托住
