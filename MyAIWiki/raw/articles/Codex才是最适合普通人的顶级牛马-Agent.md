# 卸载 Claude Code 吧，Codex 才是最适合普通人的顶级牛马 Agent

- 原文链接：https://mp.weixin.qq.com/s/8J7bNljg5AN7M8LhNyCtLw
- 来源：微信公众号 / 智见AI
- 获取时间：2026-05-14

## 核心观点

这篇文章表面上是在吹 Codex，但真正有价值的地方不是“哪个工具更强”，而是它借一个长视频教程，讲清楚了一个更重要的趋势：

> AI 编程工具正在从“写代码助手”变成“组织工作的 Agent 工作台”。

作者并不是单纯把 Codex 说成一个更强的代码生成器，而是把它描述成一个可以同时串起：

- 研究
- 文档
- 网页
- iPhone App
- 启动视频
- 投资人 Deck
- 数据库
- 社媒自动化

的统一工作现场。

所以这篇文章最值得吸收的结论其实是：

> 普通人真正该学的，不是神奇 Prompt，而是如何把 AI 从一个聊天框，升级成一个能持续产出资产的项目工作台。

文章一路展开后，核心思想会逐步落到几个关键词上：

- 项目目录
- 文件沉淀
- 权限边界
- Preview / Steer 反馈回路
- Skill / Plugin 分工
- Computer Use
- 多线程任务包调度
- 主链路优先
- 上下文工程

作者最后想推的不是“Codex 比 Claude Code 更好”，而是：

> 会用 Agent 的人，越来越像一个小团队负责人：给目标、给边界、给反馈、做验收，而不是亲手做每一步。

## 文章结构梳理

### 1. Codex 的重点不是更会写代码，而是更像工作台

文章开场先讲一个 1 小时 43 分钟的 Codex 教程。  
作者说自己最大的感受不是“Codex 会写代码”，而是：

- 它能同时做 iPhone App
- 做 Web 落地页
- 做投资人 Deck
- 做启动视频
- 处理数据库和认证
- 做 X 发帖自动化

也就是说，它更像一个“项目后台”而不是一个问答框。

这里最关键的判断是：

> AI 工具正在从“补代码”转向“组织工作”。

传统 AI 编程的问题是：
- 上下文都挤在一个对话里
- 任务串行
- 产物散落
- 每次像重新开工

Codex 这种 Agent 工作台，则尝试把这些东西都落到结构里：
- 项目目录
- 文件
- 权限
- 模型
- effort
- Skill
- Plugin
- 定时任务

### 2. 起点不是 Prompt，而是项目目录

作者非常看重开头那个看似普通的动作：

- 先创建一个真实项目文件夹

因为这个目录会成为 Agent 后续持续工作的现场：
- 创建文件
- 编辑文件
- 保存表格
- 累积研究结果
- 引用之前的产物

文章里有个细节很重要：

> 新的 Agent 可以 `@` 之前已经生成的文件

也就是说，上下文开始从“聊天历史”转向“项目文件”。

这部分其实就是非常典型的上下文工程思想：

- 不要每次把资料重新塞进对话框
- 而要把上下文变成文件
- 再把文件变成 Agent 可长期操作的工作现场

### 3. 权限决定自动化程度，但不能一把梭

作者提到视频里的做法是倾向直接开 `full access`，逻辑是：

- 让 Agent 少问、多做
- “let it cook”

这确实能显著提升效率，但作者自己也补了一个现实边界：

- 临时实验项目可以放开
- 生产项目、客户项目、重要配置文件一定要有边界

所以真正的重点不是“要不要放权”，而是：

> 你要知道 Agent 在哪个沙盒里干活。

这其实已经从工具使用进入工程治理层了。

### 4. Preview 和 Steer：未来和 Agent 协作的重要反馈回路

作者很喜欢 Codex 的两个能力：

#### （1）Preview
你可以直接在预览结果上评论，把反馈交给 Agent。

这种模式比传统“回编辑器再描述问题”的回路更短。

#### （2）Steer
更关键的是：

> Agent 运行过程中也能插入修正，不必等它全部完成。

作者认为这件事非常重要，因为 Agent 工作时长在变长：
- 以前十几秒返回
- 现在可能跑几分钟甚至十几分钟

如果不能中途纠偏，效率会被锁死。

所以这部分真正要学的，不是按钮，而是新的协作姿势：

- 发完 Prompt 之后，不是失联
- 也不是全程手动接管
- 而是保持低频但高价值的干预

### 5. Skill 和 Plugin：一个是流程资产，一个是能力连接器

文章中段对 Skills 和 Plugins 做了一个很好的区分：

#### Plugin
- 给 Agent 接外部能力
- 比如 Google Calendar

#### Skill
- 教 Agent 一套可复用的做事流程
- 比如 YouTube Researcher Skill

作者强调一个观点：

> Prompt 是消耗品，Skill 是资产。

因为一次性 Prompt 只能解决眼前任务；  
而 Skill 可以：
- 复用
- 迁移
- 持续打磨
- 成为工作流资产

对普通人的启发也很直接：

- 经常写周报 → 做周报 Skill
- 经常拆视频 → 做视频拆解 Skill
- 经常做选题 → 做选题筛选 Skill
- 经常发社媒 → 做发布前检查与改写 Skill

### 6. Computer Use：AI 开始走进软件界面

文章最有冲击力的一段是让 Codex 去操作 Figma。

作者先测试了一个简单动作：
- 在画布上放一个 `hello world`

然后再让它进一步做一个未来感鞋类品牌的 landing page，甚至用语音给需求。

这件事的重要性不是页面做得多精美，而是边界变了：

> AI 开始从文本世界进入软件界面，尝试操作鼠标键盘和可视化工具。

当然作者也没有神化：
- Figma 结果并不完美
- 按钮可能重叠
- 视觉仍需反复调

但价值在于：

- 先让 Agent 干掉大量机械动作
- 人类只负责提目标、看结果、做局部纠偏

这其实很像管理一个小团队，而不是手把手写每个像素。

### 7. 多线程不是瞎忙，而是把任务包交给多个 Agent 并行推进

这是文章中我觉得最重要的一部分之一。

作者开始做一个叫 `Chorus` 的产品，同时推进：

- 移动端设计
- iPhone App
- Web 落地页
- 投资人 Deck
- 启动视频
- X 发帖自动化
- Supabase 数据库和认证
- Tally 表单收集
- Typefully 社媒发布

作者用一个词来描述这个方法：`serial tasking`

我的理解是：

> 人不是同时瞎忙，而是不断把高质量任务包分发给多个 Agent，然后切到下一条线。

也就是说：
- 每个 Prompt 是一个任务包
- 每个任务包有独立上下文
- 每条线能异步推进
- 人扮演的是调度者，不是单线程执行者

这其实就是 Agent 时代最值得练的新工作习惯。

### 8. 做 App 的顺序：先能用，再好看

作者做 Chorus 移动 App 时，先让 Agent 生成基础 Apple 风格 screens，并明确要求：

- 先把所有屏幕在功能上做对
- 后面再加视觉 flair

这是一个很成熟的产品判断。

因为 AI 非常容易在“看起来漂亮”上先给你幻觉，但真正重要的是：
- 信息架构是否顺
- 页面流转是否通
- 核心功能是否可用

后续作者又继续修真实细节：
- 顶部固定
- 底部渐隐
- 内容滚动淡出
- tab bar 遮挡列表等

这说明：

> AI 能快速把你从 0 拉到 0.7，但从 0.7 到 0.9 仍然依赖人的产品感觉。

### 9. 数据库和认证：这是 Demo 和真产品的分界线

文章后面继续往下做了：
- 认证
- 数据库
- 数据保存
- 准备 TestFlight

作者让 Agent 选 Supabase/Postgres，理由是：
- 适合快速搭产品
- AI 易于理解和操作

这里的核心启发是：

> 页面只是壳，数据才让产品开始“活”起来。

用户能注册、登录、保存数据，App 才真正从演示变成可测试产品。

作者为了先打通主链路，甚至暂时关掉了邮箱确认，这个动作非常工程化：

- 先让主流程跑通
- 再补安全和边界

### 10. 视频、Deck、社媒自动化：Agent 工作台的边界不再只是写代码

文章后半部分继续展示：

#### Remotion 启动视频
作者给出非常具体的帧数、坐标、时间点反馈。  
这说明 AI 做视频/设计时，反馈越具体越容易改对。

#### 投资人 Deck
Codex 生成 Deck，再放进 Canva 继续人工编辑。  
后面甚至把 Claude Code 拉进来补设计优势。

#### Typefully 自动化
作者让 Codex 根据 API 创建一个发帖 Skill，还要求测试时用水果 emoji 标记，区分测试内容与正式内容。

这里最值得学的，不是炫，而是这几个非常现实的实践：

- 上下文文件化
- 测试加标记
- 主链路优先
- 反馈具体化
- 多线程调度
- 产物进入可编辑工具链

### 11. 作者最终提炼出的 SOP

文章最后其实是在总结一套通用的 Agent 工作台方法：

1. 开一个真实项目目录
2. 把所有产物放进去
3. 让 Agent 先读目标、读资料、生成计划文件
4. 计划通过后再拆成多条独立线
5. 每条线开独立 chat
6. 每个 chat 只负责一个明确产物
7. 发 Prompt 时带上相关文件和验收标准
8. Agent 跑时不要盯着等，切去下一条线
9. 出结果后做方向纠偏或细节反馈
10. 最后回到计划文件统一记录状态、阻塞点和下一步

作者的结论很到位：

> 不要再只盯着一个对话框等结果。人应该像项目负责人一样，给方向、给边界、给反馈、验收资产。

## 我的理解

我觉得这篇文章最值得保留的，不是标题里“Codex 比 Claude Code 更适合普通人”这个结论，而是它背后的方法论。

### 1. 真正的分水岭不是 Prompt，而是“有没有工作台思维”

大多数人还停留在“问一个问题，等一个答案”的使用方式。  
但 Agent 工具越来越像一个项目现场：

- 有目录
- 有文件
- 有权限
- 有工具
- 有 Skill
- 有多线程任务
- 有验收标准

谁先掌握这种工作台思维，谁就会明显拉开效率差距。

### 2. 上下文文件化，比聊天历史堆积更有复利

这一点我非常认同。  
把上下文变成文件、计划、表格、规范，远比把所有资料塞在聊天历史里可靠。

因为文件：
- 可以引用
- 可以复用
- 可以被不同 Agent 读取
- 可以跨对话延续
- 可以沉淀成资产

### 3. 未来人类越来越像 Agent 团队的负责人

作者其实已经把这个变化描得很清楚了。  
以后真正高价值的能力，是：

- 定目标
- 划边界
- 分任务
- 控方向
- 做验收

而不是自己去执行每个动作。

### 4. Skill 的意义被普通人长期低估了

一句话 Prompt 的上限很低，因为它不会沉淀。  
Skill 才是真正会复利的东西。

如果一个人把高频任务都做成 Skill，他的 AI 工作台会越来越像一支熟悉自己工作方式的小团队。

### 5. 这篇文章其实是在讲“Agent 时代的产品经理能力”

表面上它在讲 Codex，实际上它在讲：

> 当执行成本被 AI 压低之后，真正稀缺的，是判断、组织和验收能力。

所以会用 AI 做产品的人，反而更需要：
- 产品感觉
- 上下文设计能力
- 反馈能力
- 协作与调度能力

## 对 AI Coding / Agent 落地的启发

### 1. 从“聊天式使用”升级到“项目式使用”
先建目录、放资料、写计划、设验收，再启动任务。

### 2. 一条任务线一个上下文
不要把所有事都塞进同一个对话。

### 3. 让文件成为跨 Agent 协作介质
比反复口头解释更稳。

### 4. 用 Skill 固化高频流程
Prompt 可以快用，Skill 才能复利。

### 5. 主链路优先
先能跑，再漂亮；先可测，再完善。

### 6. 反馈尽量具体
UI、视频、内容、自动化任务都一样：越具体越好改。
