# 从零设计生产级 Multi-Agent Harness：架构、评估、记忆、成本与 MCP 工具接入全拆解

- 原文链接：https://mp.weixin.qq.com/s/JPhcyDc4JwRmnMQ-76A-FQ
- 来源：微信公众号 / 腾讯云开发者
- 获取时间：2026-05-14

## 核心观点

这篇文章的主线非常清晰：**生产级 Multi-Agent 系统的核心竞争力，不在于 Prompt 有多花哨，也不在于 Agent 有多少，而在于背后的 Harness（多智能体执行框架）够不够稳。**

作者把 Harness 定义为 Agent 的“操作系统”：

- Prompt 是台词
- Agent 是演员
- 工具是道具
- 模型是大脑
- 而 Harness 是导演、调度、安全规章、票务系统和舞台底座

这篇文章真正想解决的问题是：

> 为什么很多 Agent Demo 看起来很能打，一进生产就开始失控？

作者给出的答案是：问题不在模型本身，而在**运行时底座缺失**。如果没有一个真正可控的 Harness，系统很容易出现：

- 工具重复调用
- Token 爆炸
- 子 Agent 失败拖垮整条链路
- 结果看起来正确但过程完全黑盒
- 接一个新工具要改很多胶水代码
- 业务追问“这个结论怎么来的”时无法解释

因此，文章把生产级 Multi-Agent Harness 拆成五个必须正面设计的模块：

1. 架构编排（Orchestration）
2. 工具治理（Tool Registry）
3. 状态与记忆（State & Memory）
4. 评估体系（Evaluation）
5. 成本控制（Budget / Cost Control）
6. 外加一节单独讨论 MCP 工具接入

文章最后落点非常明确：

> 未来团队之间比的不是“谁的 Agent 更多”，而是“谁的 Harness 更稳”。

## 文章结构梳理

### 1. Harness 是什么：不是 Prompt 拼盘，而是运行时底座

作者先纠正一个常见误解：

很多团队以为做 Agent，就是：
- 一个 System Prompt
- 几个工具
- 一个编排器
- 一个前端输入框

这样只能叫 Demo，不叫生产系统。

Harness 和几个相近概念的区别：

- **不是 Prompt 模板**：Prompt 解决“模型怎么理解任务”，Harness 解决“模型怎么可靠完成任务”
- **不只是 Orchestrator**：Orchestrator 管“顺序”，Harness 还要管资源、记忆、成本、安全、可观测性
- **不只是 Framework**：Framework 是积木，Harness 是把积木拼成生产建筑的工程方案

这部分的核心判断是：

> Multi-Agent 不是“多 Prompt 拼盘”，而是一个完整运行时系统。

### 2. 架构编排：Agent 负责局部智能，Harness 负责全局控制

这部分是全文的结构主轴。

作者非常强调一个原则：

> Agent 负责出主意，Harness 负责拿决定。

不能把所有决策权交给 Planner Agent，因为大模型并不是天然可靠的调度器，它没有：

- 成本意识
- 并发意识
- 权限意识
- 全局一致性意识

因此 Orchestrator 必须独占五类决策权：

1. **任务生命周期**
   - 创建、规划、执行、审查、完成、失败都要有明确状态机

2. **执行计划裁决**
   - 计划可以来自 Planner，但计划生成后必须由 Harness 接管

3. **Agent 路由**
   - 任务应该交给谁，不能完全由模型自由决定

4. **失败处理**
   - 失败后是重试、降级、跳过还是终止，不能让失败 Agent 自己决定

5. **硬终止条件**
   - `max_steps`
   - `max_tokens`
   - `max_duration`
   - `max_tool_calls`

这里还有个很重要的工程细节：

- **Planner 应输出声明式计划**
- 不应直接输出命令式调用

因为声明式计划允许 Harness 介入审查、重排、裁剪和拒绝执行。

### 3. 工具治理：Tool Registry 是权限边界，不是函数集合

作者把工具治理讲得很到位。

观点很明确：

> 工具不是普通函数，而是生产资源的对外授权点。

你给 Agent 一个工具，相当于给了它一把权限钥匙。  
这把钥匙能开哪些门、谁能用、能用多久、要不要留痕、要不要审计，都必须提前设计。

生产级 Tool Registry 至少要维护这些元信息：

- 工具名称
- 工具描述
- 输入参数 JSON Schema
- 允许调用的 Agent 列表（RBAC）
- 调用超时与速率限制
- 风险等级
- 是否需要人工确认
- 输出结果结构
- 审计日志策略

作者强烈建议：

> 即使只有 3 个工具，也从第一天起强制走 Tool Registry。

因为这不是装饰层，而是结构层。如果初期不统一入口，后期工具调用会变成一团“散落的特权代码”。

### 4. 状态与记忆：让 Agent 记住该记的，忘掉该忘的

这是全篇非常工程化的一节。

作者指出，记忆系统最常见的失败方式有四种：

- 记得太少
- 记得太多
- 不分层
- 不遗忘

因此必须把 **状态** 和 **记忆** 分开：

#### 状态（State）
用于当前任务执行，短生命周期，关注一致性。

分三层：
- `Working State`
- `Session State`
- `Execution Log`

#### 记忆（Memory）
用于跨任务复用，长生命周期，关注相关性。

分两类：
- `Episodic Memory`
- `Semantic Memory`

文章特别强调两个被低估的设计点：

#### （1）检索时机
- 前置注入：稳定但费 Token
- 按需检索：省钱但模型可能忘记查

最佳实践是：

> 混合模式：少量高置信前置注入 + memory_search 工具按需调用

#### （2）遗忘机制
作者的观点我很认同：

> 记忆不是仓库，而是花园。要定期修剪。

基于访问频次、时间、重要性等做：
- 删除
- 压缩摘要
- 保留原文

否则记忆系统只会越用越钝。

### 5. 评估体系：不要只看最终答案，要看轨迹

这是 Multi-Agent 系统和单 Agent 最大不同之一。

单 Agent 常常只比最终输出。  
Multi-Agent 不够，因为你会遇到这种情况：

- 最终答案对了，但中间用了未授权数据源
- 最终代码能跑，但工具调用十几次都在浪费
- 最终结论看着完整，但关键事实来自错误检索
- 某次成功只是因为重试碰巧撞上了正确路径

所以作者把 Eval 分成四层：

1. **Component Eval**
   - 单 Agent 工具选择、参数合规、角色职责

2. **Trajectory Eval**
   - 步骤是否必要、顺序是否合理、是否陷入循环

3. **Task Completion Eval**
   - 是否完成用户目标、是否有事实错误、覆盖是否充分

4. **End-to-End Eval**
   - 用户是否采纳、人工返工率、处理时长、单位任务成本

这部分还有一个关键提醒：

> LLM-as-Judge 不是万能药。

事实正确性、权限合规、代码可运行性等，优先要用：

- 单元测试
- Schema 校验
- 规则引擎
- 检索来源校验
- 人工抽检

而不是只靠模型打分。

### 6. 成本控制：Token Budget 是生产系统生命线

这一节非常实用。

作者指出 Multi-Agent 为什么特别烧钱：

- 每个 Agent 都有 System Prompt
- 每个 Agent 都要带上下文
- 工具结果会不断回填模型
- Planner、Worker、Reviewer 多轮协作
- 失败还会重试

所以必须有实时 Budget 控制，而不是事后看账单。

核心思路是：

> 按任务复杂度分配预算，执行过程中持续监控，并在不同阶段触发不同降级策略。

文章总结了三大策略：

#### （1）Model Routing
不是所有步骤都用最强模型：
- 分类、摘要、格式转换 → 小模型
- 复杂推理、最终合成 → 大模型
- 高风险审查 → 强模型 + 规则校验

#### （2）Context Compression
保留最近几轮原文，把更早历史压缩成：
- 关键事实
- 决策
- 未解决问题
- 工具结果引用

但事实型 / 合规型任务中，关键证据不能随便压缩。

#### （3）Budget 分级降级
- 绿区：正常执行
- 黄区：压缩上下文
- 红区：切小模型 + 跳过 CoT
- 熔断区：强制收束，返回 partial result

这一节最重要的落点是：

> 当你能回答“完成一个合格业务任务平均花多少钱”时，Agent 系统才真正进入可运营阶段。

### 7. MCP 工具接入：标准化很好，但不能裸奔

作者把 MCP 的价值总结得很准确：

> MCP 让工具接入变得便宜，Harness 让工具调用变得可信。

MCP 的三大意义：

1. 快速扩展工具能力
2. 复用生态中的 MCP Server
3. 解耦工具实现与模型接入方

但作者也特别强调：

- 不要让 Agent 直接连 MCP Server
- 必须先过 Tool Registry
- MCP 提供“能力”
- Harness 提供“治理”

最佳实践包括：

- MCP Server 单独配额
- 白名单开放工具，而不是黑名单
- 高风险工具必须人工确认
- 所有 MCP 调用都必须带 Trace

这部分其实在提醒：

> 标准化工具接入不是终点，治理能力才是终点。

### 8. 可观测性与落地路线

最后这部分非常务实。

作者指出：

传统后端出问题看日志、指标、链路；  
Agent 系统更需要 Trace，因为很多错误不是“异常”，而是“过程偏移”。

比如：
- 调错工具
- 读错记忆
- 理解错用户目标
- 因预算不足提前收束
- 因上下文压缩丢关键信息

同时作者建议 Agent 系统按三阶段演进：

#### Phase 1：MVP
- 跑通一条端到端闭环
- 不要一开始就上复杂多 Agent

#### Phase 2：Hardening
- 增加权限、预算、评估、审计、回归等治理能力

#### Phase 3：Scale
- 引入分布式队列、多租户、长期记忆治理、统一 MCP 接入平台、A/B 测试、成本看板等

最后还有一个很重要的组织判断：

> Multi-Agent Harness 不是纯算法项目，而是系统工程。

它需要：
- 算法
- 后端
- 平台
- 安全
- 业务专家

一起协作，而不是只靠一个“会写 Prompt 的同学”。

## 我的理解

我觉得这篇文章最有价值的，是它把“Agent 落地”的讨论，从模型能力和 Prompt 技巧，拉回到了真正决定系统能否活下来的工程底座。

### 1. Demo 和生产之间，差的不是模型，而是运行时治理

这是全文最重要的一句话。  
很多系统不是“模型不够强”，而是：

- 没有统一调度
- 没有工具治理
- 没有状态分层
- 没有轨迹评估
- 没有预算控制
- 没有审计与可观测性

这种系统在 Demo 阶段能靠运气跑通，但一到生产就暴露结构性问题。

### 2. Harness 本质上是在给 Agent 补“系统能力”

单看模型，它擅长局部推理；  
但系统真正需要的是：

- 生命周期管理
- 权限控制
- 失败隔离
- 成本调度
- 状态治理
- 工具接入规范化

这些不是模型自然会的，而是 Harness 负责补齐的。

所以 Harness 不是“外围辅助层”，而是 Agent 系统真正的中枢。

### 3. Multi-Agent 最难的地方不是编排，而是收敛与解释

让多个 Agent 跑起来不难。  
真正难的是：

- 为什么这一步该这么跑
- 结果为什么可信
- 出错了该怎么回放
- 成本为什么爆了
- 哪个子 Agent 导致了偏差
- 哪条记忆污染了决策

也就是说：

> 真正的生产挑战，不是生成更多动作，而是让每个动作可控、可解释、可追溯。

### 4. MCP 是好东西，但越标准化越需要治理

这也是很容易被误解的一点。  
MCP 让工具接入变容易，但也意味着：

- 能力暴露更快
- 风险扩散更快
- 权限边界更容易失守

所以 MCP 的普及，不会降低 Harness 的重要性，反而会提高它的重要性。

### 5. 这篇文章其实是在讲“AI 系统工程”

表面上它在讲 Multi-Agent Harness，实际上它讲的是：

> 如何把一个看起来聪明的 AI 演示，变成一个可运行、可运维、可解释、可扩展的生产系统。

这就是 AI 工程化里最稀缺的部分。

## 对 OpenClaw / Agent 落地的启发

### 1. 要先有 Harness 视角，再谈多 Agent
不是先堆 Agent，再补治理。
而是先想清楚：
- 调度怎么做
- 权限怎么收
- 记忆怎么分层
- 工具怎么接入
- 预算怎么监控
- 评估怎么做

### 2. Tool Registry 和 Memory 分层是两个关键支点
很多系统最先失控的，就是：
- 工具调用乱飞
- 记忆污染上下文

这两个点如果前期没立住，后面越长越乱。

### 3. Eval 必须进入常规开发流程
Prompt、模型、工具、记忆策略、Budget 策略的每次调整，都应该像改代码一样走回归。

### 4. 生产级 Agent 团队更像平台团队，不只是算法团队
谁能把治理、平台、审计、评估、成本体系建起来，谁才真的能把 Agent 落地。
