## 一句话总结

**"Agent 面前摆着一张工作台"**——AI 工作台 = **计划 / 上下文 / 执行 / 验证 / 治理** 五层；个人可以 YOLO，团队必须把"任务卡 + 工作区隔离 + 可验证结果 + 合并规则"这四件事说清楚后再多开 Agent；**CLAUDE.md 三段式（What/Domain/Validation）、plan.md 任务协议、Skill = 过程资产、Subagents = 隔离、团队 3 层权限渐进（只读→半自动→受控写入）**构成可落地的最小工作台。

## 关键观点

1. **AI 工作台 5 层结构（核心范式）**
   - 计划（plan.md）/ 上下文（CLAUDE.md）/ 执行（worktree + Skills + Subagents）/ 验证（测试+日志+截图+PR 说明）/ 治理（allowlist + hooks + 权限 + 回滚）
   - "工作台搭得越清楚，Agent 干活越有边界"

2. **多开 Agent 之前，先把"可控并行"说清楚**
   - 三个常见翻车：文件互相覆盖 / 上下文污染 / 产出变多但 review/回滚跟不上
   - 必须先回答 5 问：任务怎么拆 / 工作区怎么隔离 / 结果怎么验 / 冲突怎么收 / 错了怎么退
   - 每开一个 Agent，先写一张"任务卡"（任务 / 工作区 / 边界 / 输入 / 交付 / 合并规则）

3. **plan.md 是轻量任务协议，不是项目管理文档**
   - 核心 8 段：目标 / 不做什么 / 背景和输入 / 可能涉及的文件 / 实施步骤 / 验收标准 / 验证命令 / 当前状态
   - 进阶 2 段：证据要求 / 合并条件
   - **重点是后两条**——Agent 提前收工通常因为"完成条件没写清楚"
   - `/ce-plan` 不写通用计划，先读代码库找模式/约定/历史经验，再写带验收标准的 plan.md

4. **上下文不是越多越好，而是放对位置**
   - **5 层上下文**：当前窗口 / 项目规则（CLAUDE.md）/ 任务状态（plan.md）/ 历史资料 / 结果
   - **3 档载入**：直接放进窗口（这一轮推理） / 放文件让 Agent 读（任务过程中多次用） / 放外部检索（只在需要时查）
   - **CLAUDE.md 三段式**（重 Validation）：What / Domain / Validation
   - 两条极端要避开：窗口塞满资料（模型抓不住重点） vs 只给一句需求（Agent 凭训练记忆猜）

5. **Skill = 过程资产，不是高级 prompt**
   - 区分："注意测试质量"（文档式）vs "先写失败用例 → 跑一次确认失败 → 写最小实现 → 跑测试 → 输出证据"（流程式）
   - 任何重复超过两次的事情都值得变成 Skill
   - 一个可用 Skill 至少含：触发场景 / 先读资料 / 步骤 / 命令 / 失败模式 / 输出证据 / 失败 gotchas
   - 第一版先写"验证类 Skill"（如 api-change-verification）

6. **hooks = "做到某一步时系统自动插手"**
   - 两类入口：危险动作前拦（删除/迁移/发布/改生产配置/敏感目录）/ 工具执行后补信号（整理失败日志、要求输出验证证据）
   - 挂在 `PreToolUse` / `PostToolUse` / `Stop` / `SubagentStart` / `PreCompact` 等事件上
   - Skill vs hooks 分工：Skill 描述流程 / hook 拦截 + 补日志 / Subagent 隔离搜索与评审 / plugin 打包团队能力 / **没有任何机制能代替权限治理**

7. **Subagents 的核心是"隔离"，不是"多角色"**
   - 适合拆出去的：批量检索 / 独立 review / 安全审计 / 大仓库扫点 / 只读资料整理 / 权限边界更窄的任务
   - **不适合拆**：需要频繁协商 / 共享大量中间状态 / 实现和规划强耦合
   - 多 Agent 麻烦在"结果怎么合并、证据怎么保留、失败怎么回到流程里"

8. **Agent 正在从"代码助手"变成"执行层"**
   - 读邮件 / 查日历 / 搜 Slack / 操作网页 / 拉会议记录 / 跑内部工具
   - 团队必须分层渐进：**只读 → 半自动 → 受控写入**
   - 权限表：读代码允许 / 跑测试允许（加超时）/ 写本地文件条件允许（worktree 里）/ 发 PR 人工确认 / 访问 Slack/邮箱先只读 / 改生产配置默认禁止 / 登录态操作网页高风险
   - **接入真实系统前先写 5 行威胁模型**：能读什么敏感信息 / 能否把信息发出去 / 有无 prompt injection / 误操作能否撤回 / 审计能否追到人

9. **PR 模板：让证据跟着代码一起出现**
   - 代码生成几乎免费以后，review 才是瓶颈
   - Agent 会静默失败——错误输出经常"看起来很合理"
   - PR 必带：目标 / 改动范围 / 关键设计取舍 / 跑过的命令 / 通过+失败结果 / 未覆盖风险 / 人工多看一眼的位置
   - 没有证据 = PR 把工作从"写代码"转移到"猜它有没有做对"

10. **5 个最小行动 + 7 天小试点**
    - 5 动作：计划文件 / CLAUDE.md / 验证 Skill / hooks / Subagents
    - 7 天：选任务 → 最小 plan.md → 一小段 CLAUDE.md → 隔离环境实现 → 认真 review → gotchas 写回 → 看 4 个问题（重复劳动 / 验证 / review 成本 / 复用）
    - 4 个朴素指标：返工次数 / 验证遗漏 / 上下文污染 / 合并耗时 / 经验沉淀

## 7 角度 × 3 钩子（写作复用）

### 角度 1：范式（AI 工作台 5 层）
- 钩子 1：计划 / 上下文 / 执行 / 验证 / 治理五层最小交付物表
- 钩子 2：工作台搭得越清楚，Agent 干活越有边界
- 钩子 3：Matt 个人 YOLO / 团队分层治理 = 同一范式，两种节奏

### 角度 2：并行治理（可控并行）
- 钩子 1：多开 Agent 之前必须先回答 5 问
- 钩子 2：每开一个 Agent，先写"任务卡"
- 钩子 3：worktree 解决物理隔离，验收/评审/合并还得另外设计

### 角度 3：计划协议（plan.md）
- 钩子 1：8 段核心 + 2 段进阶（证据要求 / 合并条件）
- 钩子 2：Agent 提前收工通常因为"完成条件没写清楚"
- 钩子 3：plan.md 同时给 Agent、reviewer、下一个会话、未来的自己看

### 角度 4：上下文工程（5 层 × 3 档）
- 钩子 1：上下文不是越多越好，是放对位置
- 钩子 2：CLAUDE.md 三段式（重 Validation）
- 钩子 3：避免两个极端（窗口塞满资料 / 只给一句需求）

### 角度 5：过程资产（Skill / hooks / Subagents 分工）
- 钩子 1：Skill 是流程，不是最佳实践文档
- 钩子 2：hooks 是"做到某一步时系统自动插手"
- 钩子 3：Subagents 的核心是隔离，不是"多几个角色"

### 角度 6：团队权限（3 层渐进 + 5 行威胁模型）
- 钩子 1：只读 → 半自动 → 受控写入
- 钩子 2：登录态操作网页 = 高风险（隔离账号 + allowlist）
- 钩子 3：5 行威胁模型（能读什么 / 能否发出去 / prompt injection / 误操作可撤回 / 审计可追人）

### 角度 7：评审与沉淀（PR 模板 / 7 天小试点）
- 钩子 1：代码生成几乎免费后，review 才是瓶颈
- 钩子 2：Agent 静默失败 = 错误输出"看起来很合理"
- 钩子 3：7 天选中等复杂度 bug 修复，看 4 问 + 5 指标
