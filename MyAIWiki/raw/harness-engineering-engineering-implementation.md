# 万字干货！Harness Engineering如何工程化落地？

作者：白家杰
公众号：腾讯云开发者
日期：2026-04-22

---

Harness Engineering 的概念已经火了有一阵了，全网很多文章基本都是在讲理念，讲为什么今天做 AI 开发，不能只靠一段提示词，也不能把模型当一个"更聪明的代码补全"。但与此同时，一个问题也在持续地发酵：

**理念我理解了，可是真正落到工程里，我第一步到底该做什么？**

这个问题特别重要。因为 Harness 这个词听上去很大，很像一个抽象的方法论，但它如果不能落到工程、目录、文档、脚本、工作流里，那它最后就只是一句漂亮的话。

事实上，不同项目落地出来的 Harness，表面上可能完全不一样。有的项目是 CI 很强，有的项目是规范很强，有的项目是多 Agent 很强，有的项目甚至只是把几个脚本和模板整理得非常扎实。但如果你往下挖，会发现它们解决的其实都是同一个问题：

**如何让 AI 在你的项目里，持续、稳定、规范、顺畅地做出你真正想要的结果。**

这一篇拿 JK Launcher 这个真实工程做例子，把如何一步一步把 Harness 搭出来的，原原本本讲清楚。

---

## 核心概念速览

| 概念 | 它主要回答什么问题 | 在工程里的角色 |
|------|------------------|---------------|
| **Rule** | 什么事绝对不能乱来 | 基础规矩、红线、底线 |
| **Skill** | 这件事具体应该怎么做 | 固定动作的标准操作手册 |
| **Sub Agent** | 复杂任务由谁分工处理 | 不同阶段的专业角色 |
| **Workflow** | 这些角色按什么顺序接力 | 前进、暂停、打回、重跑规则 |
| **Scripts** | 最后谁来判断到底做没做好 | 统一门禁和事后验证 |
| **MCP** | AI 怎么安全接上外部工程系统 | 外接能力与工具链 |

---

## 第一章：几个核心概念

### Rule

Rule 可以理解成给 AI 写的一套"工程规矩"。不是需求文档，不是设计方案，更不是脚本。它更像是带一个新人开发时，先告诉他的那些基础原则：
- 什么能做，什么不能做
- 什么做完必须验证
- 什么约定绝对不能绕过去

Rule 的核心作用，不是帮助 AI "变聪明"，而是帮助 AI 少犯一些本来不该反复犯的低级错误。Rule 更像一个团队的"研发制度"——制度不是拿来创造价值的，制度是拿来减少混乱的。

**但 Rule 只是软约束，不是硬门禁。** AI 理论上应该遵守它，但并不一定每次都真的遵守。

### MCP

MCP（Model Context Protocol）本质上是一种标准方式：把「仓库之外的能力」接进 AI 的工作链路里。

可以把它理解成一层可编排的接口：AI 不再只依赖当前对话和本地代码，还能对接 Wiki、知识库、表格、各类平台 API。

当想把 Harness 从「开发闭环」推到「工程交付闭环」时，MCP 往往会变得关键：交付链路里大量是 CI、签名、制品、发布、回写状态 等系统能力。

### Skill

Skill 解决的是"这件事具体应该怎么做"的问题——固定动作的标准操作手册。

比如把编译、测试、验证做成 Skill：
- Skill 负责把"做这件事的标准动作"固化下来
- 让 AI 在执行时不是临时想怎么做，而是调用已经定义好的标准流程

### Sub Agent

Sub Agent 解决的是"复杂任务由谁分工处理"——不同阶段的专业角色。

多 Agent 不是为了"分任务"，而是为了"分角色"：让不同阶段由不同的专业角色来处理。

### Workflow

Workflow 解决的是"这些角色按什么顺序接力"——前进、暂停、打回、重跑规则。

Workflow 不只是定义流程步骤，还要定义：
- 什么情况下可以前进
- 什么情况下需要暂停
- 什么情况下需要打回重做
- 什么情况下可以重跑

### Scripts

Scripts 解决的是"最后谁来判断到底做没做好"——统一门禁和事后验证。

Scripts 不是可选的，而是必须的。没有 Scripts，就没有真正的"判卷人"。

## 第二章：真正开始搭 Harness 之前，先做了什么

在真正讲"第一步怎么搭 Harness"之前，先把 JK Launcher 的规格设计文档（SPD）做好。

**SPD 的核心作用：**
- 把"做什么"和"怎么做"分开
- 让 AI 知道这次开发的规格是什么
- 给后续的 Rule、Skill、Agent 提供判断依据

---

## 第三章：Rule 很重要，但不要迷信 Rule

Rule 是基础，但 Rule 不能解决所有问题。

当 Rule 越来越多、需求越来越复杂的时候，模型会出现三种典型情况：
1. 它忘了某条 Rule
2. 它觉得某条 Rule 和这次需求"无关"
3. 它知道这条 Rule，但偷懒没做，还给自己找理由

所以 Rule 非常重要，但 Rule 不能单独解决问题。

---

## 第四章：为什么要把编译、测试、验证做成 Skill

"改完代码以后必须完成编译、测试和事后验证"——这类动作如果做成 Skill：

- 定义清晰：什么是"编译成功"、什么是"测试通过"
- 执行标准：每一步都有明确的标准动作
- 可复用：同类任务都可以调用这个 Skill

---

## 第五章：为什么最后一定会走到结构化的多 Agent

当任务越来越复杂，单个 Agent 无法高效处理所有情况时，结构化的多 Agent 变得必要：

- **分角色**：不同阶段由不同专业角色处理
- **分关注点**：每个 Agent 只需要关注自己负责的部分
- **可协作**：通过 Workflow 定义角色之间的协作规则

---

## 第六章：七个 Agent 不是拍脑袋拆出来的，而是被问题一步一步逼出来的

多 Agent 的拆分不是设计出来的，而是被问题逼出来的：

遇到什么问题 → 拆分出新的 Agent 来处理 → 继续遇到新问题 → 继续拆分

最终形成的 Agent 数量和职责，是根据实际情况逐步演化的结果。

---

## 第七章：多 Agent 落地后，如何一轮一轮补稳

多 Agent 落地后，不是终点，而是起点。需要不断补稳：

**三层补稳机制：**

1. **第一层：把流程结构单独抽成流程定义文件**
2. **第二层：把每个 Agent 的输入输出抽成角色契约**
3. **第三层：再补一个流程校验脚本**

---

## 第八章：为什么最后一定会走到 Scripts，尤其是总验证脚本

**当 Rule 开始管不住、Agent 开始出现随机性时，需要 Scripts 来兜底。**

Scripts 要解决的三类问题：
- **A 类：静态规范问题** —— 代码风格、命名规范等
- **B 类：基础交付门槛** —— 编译通过、测试通过
- **C 类：工程一致性** —— 构建产物、版本号、交付记录

总验证脚本的作用：把"是否完成"这件事，从 Agent 的责任变成系统的责任。

---

## 第九章：如何让 AI 带上「整个项目的记性」

AI 不能只依赖当前对话，需要带上"整个项目的记性"。

落到工程里，是 **dev-map 和 任务看板** 两块：
- dev-map：项目的技术债务、决策记录、历史上下文
- 任务看板：当前任务的进度、阻塞点、下一步

别把「项目级上下文」理解成堆字，而是理解成"项目级记忆"——让 AI 知道这个项目是怎么走到今天的。

---

## 第十章：团队级 Harness 里，Memory 为什么可以靠边站

在团队级 Harness 里，Memory 的优先级可以降低，因为：

- 团队级关注的是"流程"和"规范"
- 个人记忆应该沉淀到文档和工具里
- 团队成员可以流动，但项目记忆要持久

---

## 第十一章：人和 AI 的关系，以及后面还能怎么走

Harness 不是为了取代人，而是为了让 AI 和人更好地协作。

核心关系：人定义规则和目标，AI 执行和验证，人检查和修正。

---

## 第十二章：Harness 的全貌——四块拼图如何一起转

约束与规范 → 执行与协作 → 验证与判断 → 记忆与上下文

**四块拼图：**
1. Rule + Skill（约束与规范）
2. Sub Agent + Workflow（执行与协作）
3. Scripts（验证与判断）
4. Memory/Context（记忆与上下文）
