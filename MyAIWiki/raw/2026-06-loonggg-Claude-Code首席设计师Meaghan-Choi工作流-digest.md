## 一句话总结

**Anthropic Claude Code 首席设计师 Meaghan Choi 的日常 AI 工作流 6 条核心:①worktree 并行(同时开 4-5 个 Claude)②让 AI 自己做初判并给理由 ③不只让 AI 写代码,所有重复性工作全扔出去(commit / PR / Slack)④用定时任务让 AI 自动巡逻代码与产品质量(检查设计师参与度)⑤AI 现阶段做不好设计,人把控方向 ⑥始终开 Opus + 1M 上下文 + auto + loop,能自动化的全自动化,人只在判断环节介入。**

## 核心观点(7 个)

1. **并行工作靠 worktree** — 同时开 4-5 个 Claude 窗口,worktree 给仓库创建隔离副本,互相不打架
2. **让 AI 自己做决策,然后告诉你为什么** — `/prototype` 不再自己挑,而是让 Claude 选一个并解释;初判也交给 AI,人只做最终确认
3. **没有人再手写自定义技能** — 全部让 Claude 帮忙生成
4. **不只是写代码,所有重复性工作都丢给 AI** — 自动 commit / 发起 PR / 截图记录 / Slack 通知同事审核;小修复(按钮间距 / 颜色不统一)随手发个云端任务,自动变成代码提交,改动小的甚至自动通过审核上线
5. **定时任务自动巡逻产品质量** — Claude 每天扫描前端改动,检查设计师是否参与过(Slack / 会议纪要 / 文档协作记录),没参与就自动生成替代设计方案 + 打包成 PR + 私信工程师提醒;目前因 AI 设计能力不够,关掉了自动发消息但保留整流程,等模型升级
6. **AI 做不好设计,人仍要把控方向** — AI 降低"能不能做"的门槛,但"该不该上"的判断还得人;门槛降低反而要求"质量把控体系"更强
7. **始终开 auto 模式 + loop + Opus 1M 上下文** — auto 模式分类器自动判断风险,没风险直接执行;loop 让 Claude 自己循环执行到完成;核心理念:**能自动化的全自动化,人只在真正需要判断的环节介入**

## 7 个分析角度 × 3 个开头钩子

### 角度 1:现象 / 反常识

- 钩子 1:"Anthropic 工程师日常同时开 4-5 个 Claude 窗口"——我们用 AI 还停留在"一问一答"阶段
- 钩子 2:Claude Code 首席设计师已经不写代码,只搭"自动巡逻 + 自动决策"的系统
- 钩子 3:她把"小修复"交给 AI 直接上线,自己连工作会都不开

### 角度 2:痛点 / 矛盾

- 钩子 1:门槛降低 vs 质量把控——工具越强,标准越要严,反而更费心力
- 钩子 2:并行 Claude 互相打架怎么办?worktree 给答案
- 钩子 3:AI 能力还"不够好"时,要不要等?她的答案是先把流程搭好等模型升级

### 角度 3:原因 / 为什么

- 钩子 1:为什么 auto 模式能省下大量"确认疲劳"?
- 钩子 2:为什么"AI 自己选 + 给理由"比"AI 给 5 个方案 + 人挑"效率高?
- 钩子 3:为什么"先把流程搭起来,等模型升级"是更高 ROI 的策略?

### 角度 4:方法 / 怎么做

- 钩子 1:worktree + 同时 4-5 个 Claude 窗口 = 个人并行的标准操作
- 钩子 2:`/prototype` 自定义技能让 AI 选 + 解释
- 钩子 3:把"写完代码到上线"全链路交给 AI(commit / PR / 截图 / Slack 通知)

### 角度 5:流程 / 迭代

- 钩子 1:从"一问一答"→ "并行 worktree"→ "auto + loop 自动循环"→ "定时巡逻"
- 钩子 2:AI 巡逻产品(检查设计师参与度)→ 自动生成替代设计 → 自动打包 PR
- 钩子 3:即便 AI 设计能力不够,流程先搭好,等下一代模型直接接上

### 角度 6:类比 / 跨域

- 钩子 1:Meaghan 的"先把流程搭好等模型升级"= 提前写好插件,新模型来了直接接
- 钩子 2:auto 模式 = "有判别力的无人机操作员",不需要驾驶员时刻确认
- 钩子 3:全链路自动化 = 从"开一个会讨论按钮间距"到"AI 直接修"

### 角度 7:行动 / 启示

- 钩子 1:你的 AI 还停在"一问一答"吗?试试 worktree 并行 3 个
- 钩子 2:你团队里哪些"小修复 / 重复工作"还占着工程师时间?列 5 件扔给 AI
- 钩子 3:你做 AI 工作流时,有没有"先把流程搭好等模型升级"?

## 我的理解

- **"先把流程搭好,等模型升级"是 2026 年最高 ROI 的 AI 战略** —— Meaghan 关掉"自动发消息"但保留整套流程 = **永远不浪费自己搭好的工作流**;这条主线与 [[Claude-Code作者Boris-我已经不写prompt了我写loop]] [[Addy-Osmani-Loop-Engineering]] [[Agentic-Engineering-AI-Workbench]] 是同一信号的不同切面
- **"auto + loop + Opus 1M 上下文"是 Claude Code 推荐的"高交互"配置组合** —— 但这套组合需要配合 **清晰的护栏(分类器判断风险 / loop 终止条件)**,否则容易失控;与 [[Claude-Code架构深度解读-Agent系统的真正护城河不在模型-而在-Harness]] 中"模型再强也需要 Harness 约束"互相印证
- **"用定时任务自动巡逻产品质量"是 Anthropic 内部的"AI 自我监控"实践** —— AI 不只帮工程师写代码,还帮设计师/PM 检查"有没有被绕过";这条与 [[Claude-Code团队5条工作原则-Fiona-Fung分享]] 的"Trust but verify"是同一条规则的产品化形式
- **"AI 做不好设计,人把控方向"是重要的"AI 边界论"** —— 与 [[Claude-Code之父品味不是人类护城河]] 中 Boris 的"品味也会被模型侵蚀"形成互补:Boris 说品味被侵蚀,Meaghan 说当下设计上 AI 仍不够好;两者合起来 = **AI 在"判断 + 审美"上边界还在,但边界会持续移动**;人要做的是 **持续把"今天模型做不到"的事结构化地留给未来模型**
- **"让 AI 自己做决策 + 给理由"是"判断力外包"的具体实现** —— 跟上 [[买了一样的AI为什么别家的比你的强]] 的"模型是商品,判断是资产"主线 —— Meaghan 把"选择哪个方案"这种初级判断也包进 Claude = **判断的颗粒度从"拍板"下沉到"初判 + 解释"**
- **对 Seetong 团队的 3 个可借鉴动作**:
  1. **worktree 并行改造**:选一个迭代试"同时 3 个 Codex 跑不同子任务"(参考 [[多Agent使用边界与并行判定]])
  2. **"AI 自动巡逻"小试点**:写一个 cron 任务,每天扫 Seetong Bug / 需求,自动标红"测试已 3 天未响应"等异常
  3. **Opus + 1M 上下文 + auto 模式**:在重决策(架构 / 重大 Bug 排查 / 跨模块重构)任务上用这套配置,先小范围验证
