## 核心观点

1. **Prompt Engineering 是入口**：LLM旅程从Prompt开始，因为Prompt是最便宜的杠杆。很多问题先通过更好的Prompt解决，不要一上来就学RAG、Agent、微调。

2. **RAG 看起来简单，做好很难**：真实RAG涉及文档解析→数据清洗→分块→向量化→建索引→查询改写→混合召回→RRF融合→Rerank→TopK过滤→上下文拼接→生成→低置信处理→全链路记录。

3. **Context Engineering 是核心**：上下文工程比召回更重要。高阶系统关注"当前任务下，模型应该看到什么"，而不是简单的"用户问什么召回什么"。

4. **微调优先级最低**：绝大多数企业AI应用前期不需要微调，先把业务定义清楚、Prompt写好、知识库搭好、Workflow跑通。微调最重要的是数据，不是代码。

5. **Agent 工程是 Harness**：Agent背后全是工程叠加，解决的是"如何让模型在真实环境里稳定执行任务"。让模型调用工具不难，让模型稳定调用工具很难。

6. **生产级思维是分水岭**：AI应用不是demo跑通就结束了，稳定性、延迟、成本、权限、监控、降级都是工程问题，增加至少20%成本。

7. **普通人机会在业务落地**：算法岗门槛高、岗位少，AI机会在业务落地和AI应用工程。碎片化学习是进不去AI行业的真正原因。

## 7个分析角度

### 1. 学习路径视角
Avi的8层路线图：Prompt → RAG → Context Engineering → Fine-tuning → Agent → Deployment → Optimization → Evals/Observability/Safety

作者压缩为6层更适合普通人：
- L1: AI项目认知框架
- L2: Prompt/Workflow
- L3: RAG/Context Engineering
- L4: AI Coding
- L5: Agent
- L6: Evals/Observability/Data Flywheel

**钩子**：你现在的学习，是在哪一层？

### 2. 工程复杂度视角
越底层越复杂，越上层越接近业务：
- Prompt: 简单，但企业不愿付费
- RAG: 看起来简单，真正做好很难
- Context Engineering: 承上启下，最关键
- Agent: 编排复杂，涉及状态管理、失败恢复、安全兜底
- Deployment: 生产问题才出现，并发、延迟、流式、GPU利用率
- Optimization: 账单来了才知道重要性

**钩子**：你是在学工具，还是在学工程？

### 3. 成本视角
- Demo阶段：效果不错，体验顺滑
- 生产阶段：token成本、响应速度、调用频繁、上下文过长、步数太多
- 优化方向：prompt caching、上下文压缩、模型分级调用、大小模型级联、RERANK、量化、蒸馏、批处理

**钩子**：你的AI项目，第一张账单是多少？

### 4. 组织视角
DevOps ≠ MLOps ≠ LLMOps，解决的问题不同：
- DevOps: 代码有没有bug、服务稳不稳
- LLM应用: 模型为什么今天答明天答、Prompt改了为什么退化、知识库召回为什么没用、成本为什么飙升

**钩子**：你的团队在用传统软件思维做AI项目吗？

### 5. 职业转型视角
三类人群：
- AI转型者：缺项目全局视角，需要AI应用工程框架
- AI项目负责人：缺完整判断框架，需要系统性串连
- AI创业者：必须知道成本结构、难点、落地路径

**钩子**：你是哪类？你的最大短板是什么？

### 6. 竞争格局视角
- Coze/Dify/FastGPT/n8n 横向对比
- AI Coding 不是选择题，是必选题
- Manus/OpenClaw/Hermes 本质是老问题的新解法

**钩子**：你还在追热点吗？

### 7. 信息组织视角
知识库只是Context Engineering的一部分：
- 低阶关注召回
- 高阶关注上下文组织
- 真正难的是：怎么把信息组织成模型最应该看的上下文

**钩子**：你的AI系统，是在做召回还是在做上下文组织？
