# Agent 原理架构与工程实践 — 拆解

**来源**：你不知道的 Agent：原理、架构与工程实践
**标签**：#主题/AI-Agent #场景/技术博客

---

## 核心观点（5个）

1. **Harness 比模型更关键** — 更贵的模型带来的提升没有想象中那么大，验收基线、执行边界、反馈信号、回退手段才决定系统能否收敛
2. **调试 Agent 优先检查工具定义** — 多数工具选择错误都出在描述不准确，不在模型能力
3. **评测系统本身的问题比 Agent 出问题更难发现** — 如果一直在 Agent 代码上反复调，效果未必明显
4. **上下文工程决定稳定性** — Context Rot 是最常见的失效模式，分层管理是解法
5. **工程细节比复杂循环更重要** — 消息解耦、状态外化、分层提示、记忆整合、安全边界

---

## 7个分析角度

### 角度1：Agent Loop 核心逻辑

不到20行代码的 while 循环：感知→决策→行动→反馈。核心循环稳定，新能力通过三种方式接入：扩展工具集、调整系统提示、状态外化到文件。

### 角度2：Workflow vs Agent

| 控制权 | 代码预定义 | LLM 动态决策 |
| 维护成本 | 改流程需重新部署 | 调系统提示即可 |
| 适用场景 | 流程固定 | 需要灵活判断 |

### 角度3：Harness 四部分

- **验收基线**：目标明确，对错有机器可执行的判断标准
- **执行边界**：工作空间隔离、权限控制
- **反馈信号**：压缩、召回、验证
- **回退手段**：崩溃可恢复，任务图持久化

### 角度4：上下文分层设计

| 层级 | 策略 |
|------|------|
| 常驻层 | 短、硬、可执行，保护缓存命中 |
| 按需加载 | Skills 描述符常驻，完整内容触发时注入 |
| 运行时注入 | 动态信息每轮按需拼入 |
| 记忆层 | MEMORY.md 不直接进提示，需要时读取 |
| 系统层 | Hooks/代码规则处理确定性逻辑，不进上下文 |

### 角度5：ACI 工具设计原则

好工具 vs 差工具：
- **粒度**：对应 Agent 目标 vs 对应 API 操作
- **示例**：update_yuque_post（一次完成目标）vs get_post + update_content + update_title（粒度过细）
- **返回**：与下一步决策直接相关 vs 完整原始数据
- **错误**：结构化含修正建议 vs 通用字符串 "Error"

### 角度6：评测三要点

1. **Pass@k** 验证能力上限，**Pass^k** 保证上线质量
2. **transcript** 看执行记录 + **outcome** 看最终结果，两类都要覆盖
3. **先修评测再改 Agent**，失真信号比错误更危险

### 角度7：多 Agent 协作协议

- **协议先于协作**：消息结构 {request_id, from_agent, to_agent, content, status, timestamp}
- **隔离先于并行**：Worktree 隔离文件修改，任务图管理依赖
- **子 Agent 只回摘要**：搜索和调试细节留在自己的上下文里

## 21个开头钩子

### 钩子1（反常识类）

1. "更贵的模型带来的提升很多时候没有想象中那么大，反而 Harness 和验证测试质量对成功率的影响更大。"
2. "调试 Agent 行为时，应优先检查工具定义，因为多数工具选择错误都出在描述不准确，不在模型能力。"
3. "评测系统本身的问题很多时候比 Agent 出问题更难发现。"

### 钩子2（Agent Loop 类）

4. "Agent Loop 的核心实现逻辑抽象后其实不到 20 行代码。"
5. "感知 -> 决策 -> 行动 -> 反馈四个阶段不断循环，直到模型返回纯文本。"
6. "新能力基本只通过三种方式接入：扩展工具集、调整系统提示、把状态外化到文件。"

### 钩子3（Harness 类）

7. "Harness 是指围绕 Agent 构建的测试、验证与约束基础设施，至少包括四个部分：验收基线、执行边界、反馈信号、回退手段。"
8. " Harness 要做的就是把任务推进右上角，让对错有机器可以执行的判断标准，而不是靠人盯。"
9. "OpenAI 3个工程师5个月写了百万行代码，将近1500个PR，传统开发速度的10倍。"

### 钩子4（上下文工程类）

10. "Transformer 的注意力复杂度是 O(n²)，上下文越长，关键信号越容易被噪声稀释。"
11. "上下文分层：常驻层、按需加载、运行时注入、记忆层、系统层。确定性逻辑放进上下文是浪费。"
12. "Skills 描述要足够短、足够像路由条件。反例不是可选项，是 Skill 描述能不能起作用的关键。"

### 钩子5（工具设计类）

13. "工具设计按 ACI 原则：工具应对应 Agent 的目标，而不是底层 API 操作。"
14. "好工具：边界清楚、结构化错误给出修正建议。差工具：参数模糊、错误不可修正。"
15. "Programmatic Tool Calling：让模型用代码编排多个工具调用，token 可从约150,000降到约2,000。"

### 钩子6（记忆系统类）

16. "记忆分四种：工作记忆、情景记忆、程序性记忆、语义记忆。Agent 不具备原生时间连续性。"
17. "记忆整合触发阈值：tokenUsage / maxTokens >= 0.5。流程必须可回退，系统只移动指针，不删除原始消息。"
18. "MEMORY.md、按需检索和可回退整合，是跨会话保持一致性的关键。"

### 钩子7（工程实践类）

19. "真正让 Agent 跑稳，靠的不是更复杂的循环，而是消息解耦、状态外化、分层提示、记忆整合和安全边界。"
20. "协议先定，隔离先做，再谈协作和并行。多 Agent 下幻觉会互相放大，交叉验证能打断错误传播链。"
21. "安全边界三件事：谁能用、能在哪用、做了什么可以追踪。Prompt Injection 防护：最小权限 + 敏感操作显式确认 + 标注外部内容边界。"
