## 正文

### 什么是文件管理？

说是文件管理，其实更准确来说是**信息管理**。约翰·惠勒提出过一个著名的观点：It From Bit，万物皆比特，或者更通俗一点儿来说，万物皆信息。

这个观点对于当下正处于信息社会的我们来说应该不难理解，一切实体和概念都是信息，一切交互本质都是信息的交互。我们写下一篇笔记，实际是记录下了一些信息，储存一些音频、视频、文字乃至其他各种格式的文件，本质上也是储存了这些信息。具体写了些什么是次要的，传递了什么样的信息才是最重要的。

每个文件中都藏着或多或少的信息，把这些信息分门别类按照一定的逻辑整理归纳，就是文件管理。

### 为什么要文件管理？

我们日常生活中哪怕意识不到这一点也会为了便利性自发的进行文件管理。

**一类是工具类信息（文件）**，比如我们工作时高复用的图表、音视频、文字素材等等，不仅要储存，还要通过适当的方法和工具进行整理，确保有需要的时候能迅速找到这个信息（文件），而不是望硬盘兴叹。

有些仓鼠党喜欢收集各种资源，没有统一合理的整理，经常出现：这个东西我是不是有？有的话我放在哪儿了？为什么找了一圈没找到？算了还是再下一遍吧。

仅仅储存信息是不够的，我们需要的是在对的时间对的地点需要时找到对的信息。你储存了这个信息，但需要的时候却找不到它，那对你来说只不过是占用磁盘的垃圾信息。

**第二类信息是任务类信息（文件）**，相信大家工作时都遇到过多多少少大大小小的任务，有些是即刻就能完成的有些是需要长期跟进的；有些需要你分配给下属有些又需要你汇总给领导；有些要求你在几天几周后提交什么有些又在大半年后突然找你要立项时的某某文件，不胜枚举。

不同任务之间有些是孤立的有些又存在各种各样的联系。无论用什么方法，最重要的还是根据轻重缓急、分门别类让任务们排排坐，有了目标，这事儿就完成了一半。

**第三类是观点知识类信息（文件）**，像各类文娱影视作品、收藏的 KOL 动态视频文章、购买的课程等等，一般也是最为庞杂的。

对于这类信息首先是要分门别类进行规整，不同领域的信息分开放，再根据大领域内部的体系进行细分。对于这类信息做到这一步还是不够的，更进一步需要把冗长的文章、复杂的视频等提炼出内核，即变成自己的东西，用自己的语言复述出来，就是老话说的「把书读薄」。至此分别完成了信息的收集、过滤、整理、内化，最后一步还要进行输出，针对一个信息、观点、知识展开自己的论述，也就是老话说的「把书读厚」。

像著名的「费曼学习法」，就是通过去教一个完全不了解这个知识点的人，能把他教明白，那你自己自然也就懂了。

虽然对信息进行了分类，但实际工作生活中会发现这三者并不是一个非此即彼的关系，很可能它既是工具类信息又是任务类信息，既是观点知识类信息又是任务类信息，抑或是三者兼而有之。

纯粹的信息意义不大，海量的信息需要进行过滤、整理、归纳，把无效低质的内容过滤掉，根据信息之间的同一性进行归类整理，继而通过这些信息归纳出一些一般性规律，这才能被称为知识。到了这里一个个或大或小的知识点就成了输出时的军火库，通过不断学习和总结归纳把孤立的知识点串联成知识体系，在输出时随时调用。

### 如何进行文件管理？

### 信息输入，文件接收

邮箱里有个 Inbox 的概念，即「收件箱」，现在也被广泛应用在各个领域。所有接收到的文件信息都被统一放在一个固定的 Inbox，可以理解为你文件管理的入户门，各类文件信息需要在这里完成第一道粗检，然后被分配到各个岗位。

**减少 Inbox 的入口**

互联网时代，大家每天创作和接收到的信息都是海量的。面对海量的零碎信息，尤其是各个平台之间越筑越高的壁垒，首当其冲的是 Inbox 过多。

对此我的应对手段是减少 Inbox 入口。国内的泛创作平台个人使用较多的是 B 站、知乎、抖音，就会针对性进行关注去重，在 A 平台关注过某位博主就不会在 B 平台关注，确保信息来源的唯一性。当然现在还有很多切片和非本人的分享等等，人为筛查信息是不可避免的。

然后通过 Chrome 扩展 Save to Notion，把所有平台的内容汇总到 Notion。个人习惯是 All In Notion，但用什么工具无所谓，甚至你就想用一个 Word 都可以，重点在于整合统一 Inbox 的入口。当然也未必是要整合到一个，根据实际情况几个也是可以的。

**通过 Inbox 缓解焦虑和防止信息过载**

人焦虑的原因之一就是不清楚自己到底还有多少事情要做，这种不确定性会极大打击生活、创作、工作的积极性，陷入一个虚无状态。

对于任务类信息同样可以使用 Inbox 的思路，把所有工作统一丢到 Inbox 里，这样至少会清楚我的任务总量是多少，才好进行下一步的任务分配和管理。

面对多输入入口的信息和文件，海量内容最终会信息过载，把人压垮，这种崩溃同样源于不确定性。我不知道我到底收藏了多少东西、我不清楚到底还有多少任务、我说不准某个信息文件到底在那里。未知，会带来恐惧。

**一些 Inbox 的个人实际应用分享**

对我个人而言，会使用 **Eagle** 作为主要的文件类信息 Inbox。D 盘根目录上有一个名为「01 Eagle Data」的文件夹，储存着 Eagle 的数据库。我把 Chrome 的下载路径、QQ 微信等 IM 软件的下载路径、PixPin 的截图自动保存、工作软件的导出路径都映射到这里，通过自动导入，确保我接收和产出的工具类、任务类文件信息都会统一进到 Eagle。

Eagle 内通过「智能文件夹」功能也建立了内部的 Inbox，一旦文件被导入，如果它满足「没有被分类文件夹」也「没有被打上标签」，那就会被规则自动分类到内部的「Inbox」文件夹。一旦我对文件进行了处理，既分配了文件夹也打了标签，就意味着它已经从一个无主无组织的文件纳入到了我的管理体系里，会自动移出 Inbox。

对于知识类、观点类的文件信息，我更倾向使用 **Notion**。在 Notion 中分别有 Quick Note 和 Web Clipper 两个页面作为我个人信息和外部信息的 Inbox。

对于互联网上的各类信息，我会用 Save to Notion 保存到 Web Clipper。收藏时会打上来源、作者、分类标签和自己的短评、收藏原因，Notion 也会自动保存源链接方便回溯。之后闲暇时会进行一遍粗筛，认为还有复读精读价值的就会打上 Check，它会自动进入 Read Again 的页面，留到我明确有比较充裕时间和高专注力时进行二次阅读。

Read Again 的内容则会反复精读最后提炼出他的核心进行内化。如果还有引用的价值就保留到 Final Clipper，没有就删掉保持整个 Inbox 的简洁。

### 统一命名，善用工具

**没有合理的命名，就是没有意义的文件**

姓名不仅对人来说很重要，对于文件来说同样重要，它是文件最基础的名片。大名鼎鼎的 Everything 相信已经不用再多做介绍了，它可以在 Windows 平台迅速进行文件搜索。面对海量文件，如果记不清到底放在哪儿了，通过 Everything 可以快速找到。但前提是你记得文件的名字，因为 Everything 是基于文件名进行搜索的。

**命名需要考虑的问题**
- 概括性强吗？能否描述清楚这个文件的内容？
- 可读性高吗？如果太长会不会影响阅读？
- 适用性广吗？是否会对内外部交接产生影响？
- 方便快速检索吗？是否会影响查找效率？
- 影响软件兼容性吗？是否会导致某些软件 BUG？

**个人的命名习惯**

- **领域**：用缩写如 DIR（导演）、DUB（配音）、REC（录音）、MIX（混音）、SD（声音设计）来简化命名
- **项目**：在一个领域内的具体项目，是这个文件的主标题和本质
- **版本**：对于需要多次调整迭代的文件至关重要
- **执行人**：对于团队合作对接比较有用
- **标签**：通过命名进行另类标签
- **备注**：针对文件更细节的描述或者评论等
- **地点**：用来标注如拍摄地点、录制地点等
- **时间**：完整的八位年月日比如 20260101

**命名的注意事项**
- NTFS 系统上每个文件最多 255 个字符，一个汉字会占用两个字节
- 不同内容可以根据权重调整顺序，但要统一便于文件排序管理
- 如有必要可以团队内部对字母和符号进行含义映射
- 考虑兼容性尽量避免使用：! @ # ￥ % & < > ? , { } ' 「`以及空格等字符
- 考虑某些软件问题如无必要尽量避免使用中文

**善假于物，合理利用工具提升效率**

通过合理使用工具可以大幅度提高命名效率：

- **Basehead**：管理音频文件，几十项参数排序、筛选、批量操作
- **Eagle**：管理图像类素材
- **Calibre**：管理电子书
- **ReNamer**：批量重命名工具，支持正则表达式

### 大道至简，文件夹永不过时

相对于各类效率化工具，文件夹管理系统至今还是最具有普适性、门槛最低、操作最便利的方案。

**杜威十进制图书管理法**

杜威十进制图书分类法依据学科或学问领域来组织图书馆的馆藏。主要的领域包含哲学、社会科学、自然科学、科技与历史。这些领域组成该分类中的十大分类，每个分类下设 10 个次分类。采用阿拉伯数字，由 3 个整数组成，分别代表主分类、次分类及含小数点的细项分类。

用我个人的习惯来假设下，一个人的文件主要分为生活、工作两大领域，我们现在在一个空的磁盘内建立「01 生活」「02 工作」两个文件夹。用 01 02 而不是 12，是因为在数字排序中，优先对首位进行排序。

通过编号的方式建立一套成体系的文件夹，有新的文件丢到对应的文件夹就好了。建议文件夹层级不要超过六个。

**PARA System**

PARA 系统是由著名的生产力专家 Tiago Forte 提出的，四个字母的缩写：Project（项目）、Area（领域）、Resource（资源）、Archive（归档）。

- **Project**：是有明确的目标和 Deadline 的事件
- **Area**：是需要长期关注、学习、维护的领域
- **Resource**：是外部或内部产生的知识点
- **Archive**：是已经完成不需要再进行关注但还不能删除的内容

### 消化复用文件，输出倒逼输入

**储存管理不是目的，不储存不管理才是**

说了这么久的文件管理，储存文件不是目的，管理文件也不是最终目的，只是手段。终极的目的还是在合适的时空场景准确快速找到合适的文件。文件是拿来用的，不是存起来供起来展示的。

随着不停的复用，有些文件会被我们消化合并掉。我们不会认真记笔记写床前明月光，也不会认真建一个文档叫 26 英文字母表，因为这些文件（信息）随着一次次复用和输出被刻进了底层记忆里，成为了自己的一部分。

想要形成这种本能，就是反复念叨的，复用复用再复用。在一次次输出中、写下的一篇篇文章说出的一句句话做出的一个个视频里反反复复用，不同场景用，慢慢你就会发现这些东西不需要储存不需要记忆，张口即来。

**输出是核心，探索输出的方式**

上面主要论述的是复用消化，最后一个环节是输出。

有个很有趣的事情是哪怕使用相同的素材、工具、文件、信息，两个不同的人也可能产出不同的东西。这说明在文件管理背后，还有个很值得讨论的话题：文件使用。同一个信息文件可以应用在两个不同的场景里，面对同一个需求也可以使用不同的文件。

食材相同，但未必做出一样的菜。

想要掌握输出的方法，除了少数天才的直觉和灵光一现，多数普通人只能靠提高输出、创作、表达的频率，继续倒推就需要有充足的「军火」供给，并且一个杂乱的「军火库」是不够的，倒逼你整理消化自己的「军火库」，继而为了扩充「军火库」需要扩大输入。所谓输出倒逼输入。

### 文件归档，合理备份

**及时归档，珍惜精力**

前面的 PARA 系统有说过 Archive，就是归档。已经完成的项目、不再需要的文件，都需要及时归档。不仅仅是腾出来它占据的那点儿储存，更重要的是从精力上解放使用者，不再把精力放在上面被消耗。

列出当日的任务表，完成一项划掉一项，每次划掉都会带来一次正反馈，会给人一种解放了的舒爽感。明确的归档清理掉过时的文件，把宝贵的精力收回来投入到更重要的事情上。

对于个人来说归档不用做得特别复杂，按照自己正常使用的管理体系分门别类把文件打包就行。

**合理冗余备份，关键时刻能救命**

这里介绍一下比较通行的 **321 备份原则**：
- **3 份副本**：除了原始数据外，应当至少保证还有另外两份数据的副本
- **2 种存储介质**：除了内部存储（例如电脑自带的硬盘），还应当至少将一份数据存放于外部的存储介质中，例如 U 盘、移动硬盘、磁带、光盘等
- **1 份异地存储**：至少将一份副本存放于有一定物理距离的异地，以防止因洪水、火灾、地震等原因造成的数据团灭

作为个人用户当然没必要严格遵循 321 原则。首先对文件进行筛选，一些易得性高的文件没必要备份比如一些公开的文档、软件的安装包等等。

我个人的习惯会通过 Ashampoo Backup Pro 每分钟自动加密压缩备份一类文件到 OneDrive 和 NAS，确保我正在工作，突然因不可抗力导致设备损坏，我也能通过网盘和 NAS 找回文件。每半个月收到 Backup All 的任务提醒会根据 List 把需要备份的数据文件导出到一个专门的备份文件夹。

除此之外每次任务提醒还会拿一块移动硬盘接入电脑，软件也会自动把上述需备份文件备份到这个移动硬盘中进行冷备份。
