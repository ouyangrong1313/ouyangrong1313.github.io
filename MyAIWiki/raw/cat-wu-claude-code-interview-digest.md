# Cat Wu 访谈拆解：AI时代PM的唯一护城河

## 核心观点

**AI时代，PM的核心价值不是规划，而是搭建让团队高速运转的框架。**

当功能周期从6个月压缩到1天，PM的首要任务从"规划协调"变成"帮助团队更快发布"。但这不是PM的终结，而是PM的进化——从需求管道变成发布基础设施。

Anthropic能从一个几乎没有胜算的后来者变成行业领头羊，靠的是两件事：**使命对齐**和**极度专注**。

---

## 角度一：发布速度革命

### 核心洞察

Anthropic几乎每一天都有重大功能上线，但Cat说这背后不是新模型，而是**流程和文化问题**。三件事让发布周期从6个月压缩到1天：

1. **目标定清楚**：LLM太通用，反而带来模糊性。PM必须说清楚核心用户是谁、要解决什么问题、什么叫成功
2. **Research Preview机制**：不用做到完美才发布，先出去再迭代
3. **极度精简的跨团队协作**：工程师说"好了"，第二天就能完成文档+PMM+公告

### 开头钩子

1. "你还在用6个月的发布周期做产品？Anthropic已经把它压缩到1天了。"

2. "大部分公司的发布流程，工程师说一句'好了'之后，还要等三周才能让用户用上——这中间到底发生了什么？"

3. "当发布门槛低到'不需要完美'，反而是进步最快的时候。Anthropic靠这一招超过了所有对手。"

## 角度二：PM角色重构

### 核心洞察

Cat采访了几百个想进Anthropic做PM的人，发现大多数人的思维方式还停留在旧时代：做路线图、对齐需求、协调跨团队。

AI时代这套失效了。功能周期从6个月到1天，PM的首要任务从规划变成"帮助团队更快发布"。

但PM没有消亡，而是**角色融合**：工程师在做PM工作，PM在写代码，设计师在写代码。

唯一不可替代的是**产品品味**——面对GitHub上数万条Feature Request，能判断哪些值得做、用什么方式做。

### 开头钩子

1. "PM这个职位还有没有前途？我采访了Anthropic产品负责人，她的答案让所有人都意外。"

2. "当工程师开始做PM的工作，当PM开始写代码，这个行业正在发生什么？"

3. "GitHub上有数万条Feature Request，能判断哪些值得做的能力，来自任何背景，但极其稀缺——这就是AI时代PM唯一的护城河。"

## 角度三：使命驱动型组织

### 核心洞察

Anthropic能从一个落后者变成领头羊，靠的是两件事：

**使命对齐**：招募真正相信"把安全AGI带给全人类"的人。这个使命直接指导决策——即使Claude Code失败了，只要Anthropic整体成功，就很开心。

**极度专注**：使命是一把筛子，过滤掉所有分散注意力的方向。不做社交网络、不做资讯Feed，只做主线任务。

### 开头钩子

1. "为什么Anthropic没有做社交网络、没有做资讯Feed？因为使命不允许。"

2. "一个团队愿意接受'我的项目失败了但公司成功了我很开心'——这种组织是怎么形成的？"

3. "在同行都在四处扩张的行业里，专注反而成了最大的竞争优势。"

## 角度四：让模型反思错误

### 核心洞察

Cat说这是她**最被低估的AI使用技巧**：

当模型做了一个出乎意料的操作，不要直接改系统提示词，而是先问它：**你为什么这么做？**

模型的回答往往揭示真相：可能是提示词里有一段让它困惑的内容，可能是它没意识到某个步骤是任务的一部分，可能是它把任务交给了子智能体但没有检查执行结果。

这种对话让你知道该修改什么，而不是盲目调整。

### 开头钩子

1. "你还在遇到bug就改提示词？试试先问模型：你为什么这么做？"

2. "大部分人用AI的方式都是错的——不是问它怎么做，而是问它为什么这么做。"

3. "当模型出错时，大多数人的反应是惩罚它。Cat Wu的建议相反：先问它为什么。"

## 角度五：95%自动化等于零

### 核心洞察

Cat的观点：**95%的自动化几乎等于没有自动化。**

如果你把一件事自动化到95%，然后停下来监督它，那这95%反而消耗了你的时间——因为你不能真正依赖它。

她的建议：找到一件你真正想自动化的事，花时间把它做到100%。教模型你的偏好，给它反馈，让它把反馈固化进技能里。这个过程慢，但一旦做成了，你才能真正解放。

### 开头钩子

1. "你自认为已经实现了自动化，但为什么每天还要花两小时监督它？"

2. "95%的自动化不是胜利，是陷阱——因为你永远不能真正放下它。"

3. "Cat Wu说：找到一个你真正想自动化的事，然后做到100%——这个建议值200美元。"

## 角度六：Claude Code vs Desktop vs Cowork

### 核心洞察

Cat给出了清晰的区分：

- **Claude Code CLI**：需要所有最新功能、同时跑多个编程任务的人
- **Claude Code Desktop**：做前端工作，可以打开实时Preview；或不熟悉终端的人
- **Cowork**：产出不是代码的任务——整理邮件、处理Slack、写文档、做PPT

**一个简洁的使用习惯**：产出代码用Claude Code，产出其他东西用Cowork。

### 开头钩子

1. "Claude Code、Desktop、Cowork——你用对了吗？大部分人三个都用错了。"

2. "想要一个AI助手帮你写代码，另一个帮你写PPT？这个思维本身就是错的。"

3. "Cat Wu的一天：早上醒来，桌上已经有一份20页的PPT——她是怎么做到的？"

## 角度七：Evals是被忽视的核心工作

### 核心洞察

大多数人不愿写评估，但Cat认为这是严重误解。

没有Evals，你不知道新功能是否真的有效，不知道新模型是否让产品变好，也不知道哪里在退步。

Claude Code的代码评审之所以敢交给Claude，是因为有一套严格的评估体系确认Claude达到了标准。

**写好Evals本身是一门手艺**，需要真正理解产品的目标和失败模式。

### 开头钩子

1. "为什么没有人愿意写评估？为什么这是AI时代最大的误解？"

2. "GitHub Copilot能帮你写代码，但谁来评估它的代码是不是真的更好？"

3. "当你把代码评审交给AI，你凭什么相信它？Anthropic的回答是：先建好Evals。"
