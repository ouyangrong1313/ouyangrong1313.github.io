# AI 个人知识库问题深度拆解 — 消化

**来源**：吹了几年的 AI 个人知识库，为什么还是那么难用？
**标签**：#主题/AI-Agent #主题/效率 #场景/技术博客

---

## 核心观点（5个）

1. **真正难的是「让知识状态稳定运行」** — 不是存资料和搜，而是维护一份可验证的长期知识状态
2. **模型可以提议，系统负责权威** — 模型说"我觉得X"，但系统要问"凭什么、证据是哪段、什么时候有效"
3. **维基只是投影，不是事实源** — 底层应该是主张+证据+版本+快照，不是维基页面
4. **知识层和权限层必须分开** — 知识负责事实，权限负责动作，模型不能自己批准自己
5. **运行时+知识层+权限层三层缺一不可** — 现有方案（LLM Wiki/Hermes/OpenClaw/RAG）都只摸到一部分

---

## 7个分析角度

### 角度1：四层框架的认知价值

把"记忆"分成四层：
- **文档知识库**：存资料的问题
- **会话上下文**：塞窗口的问题
- **记忆运行时**：长期agent稳定运行的问题
- **知识状态系统**：多年积累准确可审计的问题

大多数讨论混乱，是因为把四件事都叫"记忆"。

### 角度2：各方案的优缺点

| 方案 | 解决什么问题 | 没解决什么问题 |
|------|------------|---------------|
| RAG/Obsidian | 检索资料 | 不能积累、不知道过期/矛盾 |
| OpenClaw自沉淀 | 让AI自己整理 | 模型既提议又决定真相，危险 |
| Hermes | 运行时稳定性 | 知识模型轻，没有evidence/时间属性 |
| Karpathy LLM Wiki | 持续积累 | 证据容易变软，维基成最终事实源 |

### 角度3：邮件事故的故障链

```
用户约束：不要执行动作
→ 会话上下文：约束只是对话里的一句话
→ 自动压缩：模型决定什么重要
→ 约束丢失
→ 后台任务继续
→ 工具真的执行
→ 用户中止失败
```

核心：**关键约束不能放在"模型自己总结、自己读回去"的链路上**。

### 角度4：主张（Claim）作为知识基本单位

主张 = 主体 + 谓词 + 对象 + 限定条件 + 证据 + 置信度 + 状态

为什么主张比段落好？
- 段落很难被精确反驳
- 可以说"这条主张的证据不成立了"，系统能处理

### 角度5：Hermes做对了什么

1. **冻结快照** — session启动时记忆状态做成冻结视图
2. **异步管线** — 记忆操作后台sync，prefetch准备下一轮
3. **Review不在主链路** — fork quiet agent判断是否值得沉淀
4. **Provider抽象** — 内置记忆是主线，外部可选接入

### 角度6：为什么Wiki会悄悄变成事实源

- LLM综合有损：丢细节、加解释、过度归纳
- 维基越完整，看起来越像"答案"
- 越像答案，越懒得回去看原文
- 问题越来越严重但不易察觉

### 角度7：对Seetong/APP研发的启示

如果要搭AI辅助的监控系统：
1. **运行时边界** — 统一管理记忆读写，不能到处都能操作
2. **冻结快照** — 对话开始时固定上下文，避免说着说着变掉
3. **权限分离** — 控制云台/查看录像这种危险动作，不能模型自己批准
4. **证据链** — 每条判断要能追到原始日志/录像片段

## 21个开头钩子

### 钩子1（核心问题类）

1. "真正难的不是'把资料存好，让AI搜'。真正难的是：让一份长期知识状态，在AI系统里稳定运行。"
2. "模型可以提议，但谁来负责事实？这句话是我现在理解个人知识库问题的核心。"
3. "过去两三年，'AI+个人知识库'这个故事至少讲了四轮，但每次用到一定规模都会遇到相似的问题。"

### 钩子2（四层框架类）

4. "很多讨论之所以混乱，是因为我们把四件事都叫成了'记忆'——文档知识库、会话上下文、记忆运行时、个人知识状态，其实是完全不同的东西。"
5. "现在很多'AI个人知识库'的问题在于：它们试图用第一层的工具，去解决第四层的问题。"
6. "中间两层基本被跳过了——用一个文档库，期待它变成长期知识状态。"

### 钩子3（RAG问题类）

7. "RAG本质上是检索系统，不是知识维护系统。今天得出的判断，明天不会自动继承；这次发现的关系，下次也不会自然沉淀。"
8. "它不知道哪份资料过期了，不知道两份资料相互矛盾，不知道一个判断三个月前成立现在已不成立。"
9. "用一个为'检索'优化的系统，期待它做'维护'的事情——这就是为什么RAG用到后面越来越难用。"

### 钩子4（模型独大危险类）

10. "模型既负责提议，也负责决定什么是真相——这就容易出事。"
11. "关键约束不能放在'模型自己总结、自己读回去'的链路上。"
12. "模型可以提议，模型可以总结，模型可以帮忙发现候选记忆。但最终什么能进入长期状态，不能由模型自己说了算。"

### 钩子5（Hermes设计类）

13. "Hermes做对了三件事：冻结快照、异步管线、review不在主链路——工业级的运行时设计。"
14. "但Hermes的知识模型很轻：MEMORY.md和USER.md，没有claim，没有evidence字段，没有时间属性。"
15. "Hermes解决了'记忆什么时候进入模型'的问题，却没有完全解决'什么东西有资格成为事实'的问题。"

### 钩子6（Wiki证据变软类）

16. "问题是：维基上的判断从哪来？很多时候它只会告诉我'综合自多份文档'，但点回去看，找不到清晰的证据链。"
17. "维基不能是底层事实源。它只能是某种投影。真正的底层应该是更细的东西：主张、证据、版本、快照。"
18. "LLM一次次综合的过程不是无损的——它会丢细节、加自己的解释、过度归纳。这些有损综合在维基里堆叠几次之后，你已经追不回原始事实了。"

### 钩子7（实践启示类）

19. "知识负责事实，权限负责动作。危险动作必须由独立权限层负责，模型不能自己批准自己。"
20. "如果要搭一个真正能长期使用的个人知识系统，先不要问用什么工具，先问六个问题：运行时/证据/审核/快照/投影/权限边界。"
21. "真正难的不是写代码，而是定义：什么算一条好的主张？什么样的证据算支持？哪些判断可以自动接受，哪些必须人工确认？"
