## 一句话总结

**React 比 HTML 更适合作为 AI 设计稿格式**——因为 React 的组件化、树形结构、数据驱动、文本化交付和 Git 版本管理，能让 Coding Agent 在设计稿基础上低成本理解、修改和扩展；而巨大 HTML 扁平、非结构化、错误不可见，已不适合 Agent 时代。

## 核心观点（5 个）

1. **设计稿 ≠ 静态展示**：Claude Design 是 UX 交互设计工具，要求设计稿具备真实交互和数据能力，HTML 做不到
2. **组件化是复杂度必然选择**：UI 复杂到一定程度，巨大 HTML 不可行；React 天然支持小组件化，修改只需加载一个
3. **树形结构对 Coding Agent 友好**：层级清晰 vs HTML 的扁平化，开发和 AI 都能一眼看清组件关系
4. **Agent 时代错误可被快速修正**：React/JS 有明确报错和组件定位，HTML 样式错误反而难 debug
5. **Claude Design 四件套交付**：HTML（骨架）+ CSS（设计系统）+ React（组件结构）+ data.jsx（数据），文本化天然支持 Git

## 我的理解

- **"React 比 HTML 更适合 AI 设计稿" 是 2026 年 AI Coding 范式在设计侧的延伸**——和 [[54万行代码的顿悟-Markdown才是新编程方式]] 同一主线：可读、可 diff、可版本管理的"结构化文本"是 AI 时代一等公民
- **与 [[Claude-Code动态工作流-让AI自己写Harness-这事靠谱吗]] 的呼应**：都强调"结构比提示词更重要"——React 的树形结构就是设计稿侧的"harness"
- **设计稿 → 组件 → 代码** 的链路打通，是 Coding Agent 从"改文件"到"改设计"的关键一步
- **"data.jsx" 这个提法很新颖**——传统设计工具只关心"长什么样"，Claude Design 关心"长什么样 + 数据怎么流"；这是给开发者真正可用的设计资产
- **"Git 友好 + diff 协作"** 模式可推广到所有 AI 协作资产：文档、配置、设计稿、测试用例——只要是文本，都应该进 Git
