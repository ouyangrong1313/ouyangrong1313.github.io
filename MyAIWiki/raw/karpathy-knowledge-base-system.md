你不需要复杂的 RAG 系统。你只需要一个聪明的文件夹结构和一个 LLM。

Andrej Karpathy：

OpenAI 创始成员之一

2026 年 4 月初，他在 Twitter 上分享了自己如何用 Obsidian + Claude Code 构建个人知识库的方法。这条推文 3 天内获得 5000+ star，因为它解决了一个所有知识工作者都面临的问题：

如何让收集的信息真正变成可用的知识？

核心洞察：你可能根本不需要 RAG

Karpathy 的核心发现是：

我原本以为要用复杂的 RAG 系统（向量数据库 + 嵌入模型 + 检索流程），但 LLM 已经足够擅长自动维护索引文件和简要摘要。

在他最新的研究 wiki 中：

~100 篇文章
~40 万字
LLM 几乎可以轻松处理所有查询

传统 RAG vs Obsidian 方案对比

| 维度 | 传统 RAG | Obsidian 方案 |
|------|----------|---------------|
| 技术复杂度 | 向量数据库 + 嵌入模型 + 检索流程 | 只有 Markdown 文件 |
| 成本 | 需要服务器、API 调用 | 几乎免费 |
| 可解释性 | 黑盒，难以查看内部 | 完全透明，人眼可读 |
| 维护成本 | 需要技术背景 | 任何人都能用 |
| 适用规模 | 百万级文档 | 百~千级文档 |
| 设置时间 | 数小时~数天 | 10 分钟 |

结论：对于个人或小团队（<10 人），Obsidian 方案已经足够。

Raw 和 Wiki 的本质区别

很多人用 Obsidian 做笔记，最后都变成了一个巨大的文件夹——什么都往里丢，丢完就再也找不到。

Karpathy 的方案把知识库分成两层：raw/ 和 wiki/。

一句话区分

raw/ = 你收集来的原始素材，原文照搬，不用整理
wiki/ = LLM 帮你消化后的结构化知识，可以检索、可以提问、可以复用

具体区别

| | raw/ | wiki/ |
|--|------|-------|
| 里面是什么 | 别人写的原文（文章、论文、笔记） | LLM 提炼重写的概念文章 |
| 谁写的 | 作者原文，你只是搬运 | LLM 根据原文重新组织 |
| 格式 | 长短不一、格式混乱、没有统一结构 | 统一格式、有标题、有摘要、有索引 |
| 有没有关联 | 文件之间互不相连 | 用 [[链接]] 互相连接，形成知识网络 |
| 能不能提问 | 不能。你得自己翻 | 能。直接问 LLM，它会检索相关文章回答你 |
| 谁在维护 | 你（把看到的好文章丢进去） | LLM（编译、更新索引、补充关联） |
| 你需要做什么 | 看到好内容，丢进去就行 | 什么都不用做，说一句"编译 wiki" |

一个类比

如果把知识库比作做饭：

raw/ = 冰箱里的生食材。你从市场买回来，直接塞进去，不用洗不用切
wiki/ = 做好的菜。食材被处理过、组合过、摆好盘，随时可以上桌

你不需要自己切菜炒菜（手动整理笔记），LLM 就是你的厨师。你只管往冰箱里塞食材，说一句"帮我做菜"，菜就出来了。

完整工作流程

数据摄入 (Ingest) → Wiki 生成 → Q&A 查询 (Query) → 健康检查 (Lint)

三个核心动作

Ingest（摄入）：喂资料给 AI，自动更新 10-15 个相关页面
Query（查询）：提问得答案，好答案直接回写成 Wiki
Lint（体检）：定期检查，找矛盾、过时内容、孤立页面

数据摄入的两种方式

方式一：快速收集（适合批量摄入）
使用 Obsidian Web Clipper 一键保存：
浏览网页时点击 Clipper 扩展
自动转为 Markdown 并保存到 raw/
图片自动本地化（需要 Local Images Plus 插件）

方式二：深度拆解（适合精品内容）
结合 Dan Koe 的逆向工程（Reverse Engineering）方法

什么样的文章值得深度拆解？
你读完后有冲动转发
里面有至少一句话让你想截图
你隐约觉得"这种写法我学得会，但不知道怎么描述"

完整工作流：
看到好文章 → 用 Claude Code 进行逆向拆解 → 生成拆解文档（包含 7 个新角度 + 每个角度配 2-3 个开头钩子）→ 自动保存到 raw/ 文件夹 → 让 AI 转译成 Wiki 形式 → 自动打标签，与其他内容关联

具体操作：
1. 拆解文章（在 Claude Code 中）
2. 保存拆解结果到 raw/
3. 转译成 Wiki
4. 自动打标签（#主题/xxx #手法/xxx #场景/xxx）

最终效果：
找灵感？打开 INDEX，看看之前拆过的 7 个角度
学手法？点 #手法/好奇心循环 ，看 3 篇不同文章是怎么用同一个手法的
找素材？按主题浏览，一键调出所有相关拆解

核心原则总结

| 原则 | 说明 |
|------|------|
| LLM 写，你读 | wiki 是 LLM 的领域，你很少手动编辑 |
| 累积式增长 | 每次查询都可能增强 wiki |
| 索引即检索 | 聪明的文件结构替代复杂的检索算法 |
| 透明可解释 | 所有数据都是人眼可读的 Markdown |
| 从简单开始 | 不需要一开始就构建复杂系统 |

何时需要"真正的" RAG？

Obsidian 方案适合：
- 个人研究者
- 小团队（<10 人）
- 文档数量 < 1000
- 需要可视化、可编辑的知识库

需要 RAG 的场景：
- 数万~百万级文档
- 需要毫秒级响应
- 多用户并发查询
- 企业级部署

Karpathy 的建议：

为什么不先从 Obsidian 开始？如果发现规模超出了它能处理的范围，再迁移到 RAG。大多数人不需要真正的 RAG 系统。
