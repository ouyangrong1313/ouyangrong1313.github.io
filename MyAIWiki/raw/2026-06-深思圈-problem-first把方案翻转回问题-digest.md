## 一句话总结

**每个"需要做 XX"都是一个没说清楚的问题的"压缩版本"；PM 用 /problem-first 技能，90 秒、8 个部分（方案跳跃诊断 / 底层问题 / 假设挑战+验证 / 问题陈述 / 三个替代框架 / 想法筛选漏斗 / 草稿信 / 强制全部输出）把方案解压回问题；同样适用于点子太多的 PM 筛选 backlog（50 个想法 90% 死于"证据状态：无"）。**

## 核心观点（7 个）

1. **每个方案都是问题压缩版** — 团队感知到痛点，但跳过定义问题直接给答案；PM 工作 = 把答案"解压"回问题
2. **不否定方案，把方案当研究材料** — 政治上更安全（"挖进路线图底下找问题"而非"挡在路线图前面"）；保持判断力
3. **Munger 翻转** — Invert, always invert；"这个方案能解决什么"不如"这个方案在响应什么问题"；先翻转，再前进
4. **8 个部分输出** — 方案跳跃诊断 / 底层问题 / 假设挑战（带"如果错了风险"和"验证方案"）/ 问题陈述 / 三个替代框架 / 想法筛选 / 草稿信 / 90 秒
5. **假设挑战必须带验证** — 没验证方法的假设是"更精致的猜测"；/problem-first 强制在开始构建前就知道用什么数据证伪
6. **三个替代框架的威力** — 团队需求"通知系统"折叠了 3 个完全不同的产品（UI 状态模式 / 信任与状态问题 / Agent 告警）；没对比几乎无法提前发现错误
7. **逆向用法：想法筛选漏斗** — 50 个想法 → 90% 死于"证据状态：无" → 3-5 存活 → 1 个下周带证据包提案；瓶颈不是生成是筛选

## 7 个分析角度 × 3 个开头钩子

### 角度 1：现象 / 反常识

- 钩子 1：90% 的"好点子"在证据状态这一步就该死
- 钩子 2：通知系统做对了 UI 状态但解决错了问题
- 钩子 3：跳过草稿信 = 制造沟通真空

### 角度 2：痛点 / 矛盾

- 钩子 1：被点子淹没 vs 想法不够
- 钩子 2：路线图写好了但 PM 不信 — 怎么切入
- 钩子 3：技术工作和沟通工作的成本不对等

### 角度 3：原因 / 为什么

- 钩子 1：为什么 PM 总是跳过"我的假设如果是错的"
- 钩子 2：为什么否定方案是政治自杀
- 钩子 3：为什么"更快"不是 AI 的主要价值

### 角度 4：方法 / 怎么做

- 钩子 1：8 个部分一次 AI 调用，90 秒
- 钩子 2：每条假设挑战 = 风险 + 验证方案
- 钩子 3：草稿信 = 让团队拉到同一对话起点

### 角度 5：流程 / 迭代

- 钩子 1：入职第 30 天不要打"先研究问题"这张牌
- 钩子 2：把"解压方案"当成工作习惯，不只 PM
- 钩子 3：PM OS 加载公司上下文，让建议针对真实约束

### 角度 6：类比 / 跨域

- 钩子 1：Munger 翻转 = "我们怎么会失败" > "我们怎么成功"
- 钩子 2：客户说"我需要更便宜" = 路线图说"建通知系统"，是同一类错误
- 钩子 3：协议 vs 提示词 = 同样的 8 个部分，不管输入什么

### 角度 7：行动 / 启示

- 钩子 1：先想"这个方案在响应什么问题"
- 钩子 2：每条假设必须带验证方法
- 钩子 3：工程师、设计师、业务同学都能用 /problem-first
