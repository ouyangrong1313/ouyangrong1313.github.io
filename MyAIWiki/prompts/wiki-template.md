> 参考 LLM-Wiki Skill 核心方法论：每篇文章 = 原文 + 知识节点 + 关联图谱

---

## 使用方式

**1. 写新文章时**：直接复制下面"## 模板"区，填入实际内容
**2. 编译脚本时**：调用 `prompts/wiki-template.md` 作为模板源
**3. 验证时**：用底部"## 验证清单"逐条检查

## 知识节点提取规范

### 合格节点 ✅
- 独立成段可被理解（不依赖本文其他段落）
- 对应一个可 grep 的关键词（建议中文 2-6 字 / 英文 1-3 词）
- 不与同篇其他节点语义重叠
- 至少被 1 篇文章 inbound 引用（否则是孤立节点）

### 反例 ❌
- "Codex 是 OpenAI 的编程工具" — 太宽泛，是定义不是节点
- "Harness 层" — 没具体含义，需要更细的拆解
- 长句子（超过 30 字）— 应该拆成多个节点

### 节点提取步骤

1. **通读文章**，把每个"独立概念 / 判断 / 方法"标出来
2. **去重合并**：把同义/近义的概念合并成一个节点
3. **可 grep 化**：把节点名改成 2-6 字关键词（方便 `rg` 查询）
4. **数量控制**：5-10 个最核心的，其余进"正文要点"区
5. **关联检验**：每个节点问自己"我能在知识库里找到 inbound 链接吗？"

## 关联图谱填写规范

### 三个维度

| 维度 | 含义 | 典型问题 |
|------|------|---------|
| **上游** | 本文哪些节点基于已有文章 | "X 概念来自 [[Y]]，我在 Y 的基础上重新阐述" |
| **下游** | 本文哪些节点被后续文章应用 | "X 概念在 [[Z]] 中被验证 / 落地" |
| **同级** | 本文哪些节点与已有文章是同主题不同视角 | "X 概念在 [[W]] 中有另一种解释" |

### 链接数量建议
- **上游**：1-3 个（不是 0）
- **下游**：0-2 个（如果还没有，就写"暂无，待后续应用"）
- **同级**：1-3 个（同主题其他视角）

## Frontmatter 字段说明

| 字段 | 必填 | 格式 | 说明 |
|------|------|------|------|
| `title` | ✅ | 字符串 | 文章标题 |
| `category` | ✅ | 枚举 | 01-ai-agents / 02-ai-coding / 03-productivity / 04-app-dev / 05-content-creation / 06-ai-tech / 07-rag-systems |
| `tags` | ✅ | 数组 | 标签数组，使用 `#主题/xxx #场景/xxx #节点/xxx` 格式 |
| `nodes` | ✅ | 数组 | 知识节点数组（5-10 个） |
| `links` | ⚠️ | 数组 | 至少 1 个内链（推荐 3+） |
| `date` | ✅ | YYYY-MM-DD | 写入日期 |
| `source` | ⚠️ | 字符串 | 原始来源 |

## 验证清单（每篇新文章完成时检查）

### 必填项
- [ ] `title` / `category` / `tags` / `nodes` / `date` 全部存在
- [ ] `nodes` 数量在 5-10 之间
- [ ] 至少 1 个 `links` 指向已有文章
- [ ] 分类目录与 `category` 一致

### 知识节点质量
- [ ] 每条节点独立可理解（不依赖本文其他段落）
- [ ] 每条节点对应一个可 grep 的关键词
- [ ] 节点之间不重叠
- [ ] 至少 50% 节点能在 grep 后被命中（即不是冷僻词）

### 关联图谱
- [ ] 上游 / 下游 / 同级 至少 2 个维度有内容
- [ ] 链接都是真实存在的（不是凭印象写）
- [ ] 没有孤立节点（每个节点至少被 1 篇文章引用）

### 一致性
- [ ] 标题与原文标题一致
- [ ] 核心结论能用一句话讲清
- [ ] 分类目录正确
